Static HTML-документ

Коротко о блоке

Статический HTML-документ — это основной HTML-документ страницы, который браузер может разобрать, превратить в DOM и показать без запуска JavaScript-приложения. 

Static HTML scenario
→ Full HTML document
→ HTML Parser читает HTML
→ DOM создаётся по мере чтения HTML
→ Обнаружение CSS / images / fonts
→ Parser дошёл до конца HTML
→ Можно сделать первый render?
→ HTML виден пользователю
→ Native browser behavior: links / forms / controls
→ Static HTML интерактивен

Браузер не превращает HTML в страницу одним действием. HTML проходит через parser: текстовый поток становится tokens, tokens обрабатываются стадией tree construction, а результатом становится DOM Document, который затем может быть использован для отображения страницы.

1. Границы блока

Блок начинается с HTML-документа, полученного как основной документ навигации: 

<!doctype html>
<html lang="ru">
  <head>
    <title>Page</title>
    <link rel="stylesheet" href="/style.css" />
  </head>
  <body>
    <h1>Title</h1>
    <p>Text</p>
    <a href="/about">About</a>
  </body>
</html>

Блок заканчивается состоянием, когда HTML уже виден, а страница обладает базовой нативной интерактивностью: ссылки переходят по href, формы отправляются по action и method, поля ввода принимают фокус и текст, стандартные controls работают встроенными механизмами браузера.

Внутри блока находятся:

HTML parsing
→ tokenization
→ tree construction
→ DOM construction
→ resource discovery
→ parser complete
→ first render possibility
→ native interactivity

DOMContentLoaded стоит сразу после завершения parsing-а в общей схеме загрузки документа. В этой статье он упоминается как следующая контрольная точка, но сам механизм события не разбирается.

2. HTML parser

HTML parser — часть браузера, которая принимает входной поток HTML и строит DOM Document.

В HTML Standard модель parsing-а описана так:

stream of code points
→ tokenization
→ tree construction
→ Document

Stream of code points — поток символов, полученный после декодирования байтов HTML-документа. Для parser-а это ещё не дерево и не набор элементов, а последовательность символов: 

< h 1 > T i t l e < / h 1 >

Parser обрабатывает этот поток в два основных этапа.

Tokenization — разбор текста на tokens. Token — промежуточная единица parser-а: уже не сырой текст, но ещё не DOM node.

Tree construction — построение DOM tree из tokens. На этом этапе parser не просто “читает теги”, а изменяет DOM Document: создаёт элементы, вставляет их в подходящее место дерева, закрывает открытые элементы и обрабатывает текстовые узлы.

3. Tokenization

Tokenization превращает HTML-текст в поток tokens.

Пример:

<p class="lead">Hello</p>

Упрощённый поток tokens:

start tag token: p, class="lead"
character tokens: H e l l o
end tag token: p

Основные tokens для этой ветки:

DOCTYPE token создаётся из:

<!doctype html>

Он влияет на режим документа. Современный <!doctype html> включает стандартный режим обработки.

Start tag token создаётся из открывающего тега:

<section id="main"></section>

Он содержит имя тега и атрибуты. Именно такой token обычно приводит к созданию element node.

End tag token создаётся из закрывающего тега:

</section>

Он сообщает tree construction, что соответствующий элемент должен быть закрыт.

Character token представляет текстовые символы между тегами. Последовательность character tokens обычно превращается в Text node. 

character tokens: H e l l o
→ Text node: "Hello"

Comment token создаётся из HTML-комментария:

<!-- note -->

Комментарий может попасть в DOM как comment node, но не отображается как обычный текст страницы.

EOF token (End Of File) означает конец входного HTML-потока. В диаграмме этому соответствует момент Parser дошёл до конца HTML.

4. Tree construction

Tree construction — стадия HTML parser-а, которая получает tokens от tokenizer-а и изменяет DOM Document. 

tokens
→ tree construction
→ DOM tree внутри Document

Она начинается не “после всего tokenization”, а работает по мере появления tokens. Tokenizer создаёт следующий token, затем tree construction обрабатывает его и обновляет DOM tree. Поэтому DOM строится постепенно.

Пример:

<body>
  <h1>Hello</h1>
  <p>Text</p>
</body>

Упрощённо:

start tag token: body
→ создать HTMLBodyElement и вставить в документ

start tag token: h1
→ создать h1 внутри body

character tokens: Hello
→ создать Text node внутри h1

end tag token: h1
→ закрыть h1

start tag token: p
→ создать p внутри body

character tokens: Text
→ создать Text node внутри p

Итоговая структура:

Document
└─ html
   ├─ head
   └─ body
      ├─ h1
      │  └─ "Hello"
      └─ p
         └─ "Text"

Tree construction выполняется браузером как часть HTML parser-а. Внутри этой стадии браузер учитывает текущее место вставки, открытые элементы и правила HTML. Поэтому DOM может отличаться от буквального текста HTML.

Пример:

<ul>
  <li>One</li>
  <li>Two</li>
</ul>

В HTML закрывающий </li> может быть опущен, но parser всё равно строит корректные элементы списка по правилам HTML.

Главный вывод:

HTML source ≠ DOM tree
token ≠ DOM node

HTML source — исходный текст. Token — промежуточная инструкция parser-а. DOM node — объект внутри построенного дерева документа.

5. DOM создаётся по мере чтения HTML

DOM (Document Object Model) — объектная модель документа. Это дерево узлов, где узлами могут быть Document, elements, text nodes, comments и другие части документа.

DOM construction является incremental: дерево расширяется по мере чтения HTML, а не только после получения всего документа. 

получена часть HTML
→ tokenizer создаёт tokens
→ tree construction вставляет nodes
→ приходит следующая часть HTML
→ parser продолжает
→ DOM дополняется

Это важно для понимания видимости страницы. Браузер может начать parsing после первого полученного chunk-а HTML и пытаться подготовить первый render на основе уже доступных данных. Это не гарантирует мгновенный показ каждого узла, но означает, что parsing, построение DOM и подготовка отображения не обязаны ждать полного HTML-файла.

6. Обнаружение ресурсов

Во время parsing-а браузер встречает элементы, которые указывают на внешние ресурсы. 

<link rel="stylesheet" href="/style.css" />
<img src="/hero.png" alt="Hero" />
<link rel="preload" href="/font.woff2" as="font" crossorigin />

Это resource discovery: браузер обнаруживает ссылку на ресурс и может начать его загрузку.

Важно различать:

resource discovered ≠ resource loaded
resource loaded ≠ resource applied

CSS обычно обнаруживается через <link rel="stylesheet"> или <style>. Изображения часто обнаруживаются через <img src>. Шрифты чаще становятся известны после обработки CSS с @font-face; напрямую из HTML они могут быть обнаружены через явный preload: 

<link rel="preload" href="/font.woff2" as="font" crossorigin />

Также у браузеров есть preload scanner: пока основной HTML parser строит DOM, браузер может заранее искать важные ресурсы в доступной разметке и начинать их загрузку. Это не меняет смысл основного блока: ресурс сначала должен быть обнаружен, а загрузка и применение ресурса — следующий этап. Подробнее про получение CSS и построение CSSOM.

7. Parser дошёл до конца HTML

Момент Parser дошёл до конца HTML означает, что основной HTML input обработан до конца. 

HTML input
→ tokenization
→ tree construction
→ EOF token
→ parsing основного HTML завершён

К этому моменту DOM построен из HTML-документа. Но внешние ресурсы могут ещё загружаться. 

parser complete ≠ all images loaded
parser complete ≠ all fonts loaded
parser complete ≠ load event

Следующая важная контрольная точка — DOMContentLoaded. Согласно MDN, это событие срабатывает после того, как HTML-документ полностью разобран, а deferred scripts и module scripts загружены и выполнены; оно не ждёт изображения, subframes и async scripts. В чистом статическом HTML без app scripts эта точка близка к завершению parsing-а, но всё равно является отдельным событием жизненного цикла документа. Подробнее про DOMContentLoaded.

8. Первый render и видимость страницы

Блок Можно сделать первый render? означает, что браузер проверяет, достаточно ли данных для первого визуального результата.

Для статического HTML это можно представить так:

часть DOM уже построена
+ есть нужные стили для показа
+ нет критичного блокера отображения
→ можно показать первый результат

Первый render не обязательно строго ждёт конца всего HTML. Браузер может начать parsing после первого chunk-а данных и пытаться рендерить опыт на основе уже полученной информации. Но перед появлением пикселей HTML, CSS и JavaScript, если он есть, должны быть обработаны в той мере, которая нужна для безопасного показа.

Полная цепочка rendering pipeline остаётся отдельной темой. Для этой ветки достаточно зафиксировать: DOM и стили используются браузером для построения видимого результата. Подробнее про Style → Layout → Paint → Compositing.

9. Static HTML интерактивен

Статическая HTML-страница интерактивна без JavaScript-приложения, если используются элементы с нативным поведением.

Ссылка:

<a href="/about">About</a>

href задаёт цель перехода. При активации ссылки браузер выполняет навигацию.

Форма:

<form action="/search" method="get">
  <input name="q" />
  <button>Search</button>
</form>

action задаёт URL обработки формы, а method — способ отправки данных.

Controls — элементы управления вроде input, button, select, textarea, details. Они могут получать фокус, принимать ввод, переключаться, раскрываться и участвовать в отправке формы без React, Vue или client-side router. 

Static HTML interactive
= нативная интерактивность HTML-элементов
≠ SPA-интерактивность

10. Что важно не перепутать

HTML source ≠ DOM
token ≠ DOM node
tree construction ≠ rendering
parser complete ≠ DOMContentLoaded
DOMContentLoaded ≠ load
resource discovered ≠ resource loaded
font discovered in HTML ≠ font discovered through CSS
static interactivity ≠ JavaScript app interactivity
