React update: commit phase, DOM mutations, useLayoutEffect, browser render и useEffect

1. Где начинается эта часть

Эта статья продолжает ветку клиентского обновления React после setState, batching, scheduling, render phase, reconciliation и Fiber. На этом месте React уже не решает, каким должен быть следующий UI. Это было сделано раньше, во время render/reconciliation. Теперь React должен применить подготовленный результат к уже существующему DOM.

В статье React initial render: первый render компонента, commit, useLayoutEffect и useEffect уже разобран первый запуск React UI: root.render(<App />), первый commit, useLayoutEffect, первый paint и useEffect. Текущий материал не повторяет этот путь. Здесь рассматривается update уже интерактивного интерфейса: например, пользователь нажал кнопку, React пересчитал часть дерева и теперь должен применить изменения к DOM.

2. Что commit получает от render/reconciliation

Commit phase не вызывает компоненты заново. На этом этапе React уже не выполняет reconciliation и не сравнивает деревья с нуля.

Во время render/reconciliation React уже получил новый результат компонентов, сопоставил его с текущим Fiber-деревом и подготовил рабочую версию дерева. Если какой-то DOM-узел должен измениться, соответствующий Fiber-узел уже помечен флагами. Если в поддереве есть работа, это также отражено во внутренних признаках поддерева. Если ветку можно было переиспользовать, React мог сделать bailout и не спускаться глубже без необходимости.

Поэтому commit можно понимать как этап применения. React берёт уже подготовленную работу и переносит её в host environment. В React DOM такой средой является DOM браузера.

Пример:


function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

Когда count меняется с 0 на 1, render/reconciliation уже выяснил, что тип button остаётся тем же, а текстовый потомок должен измениться. Commit не пересоздаёт всю кнопку и не пересчитывает всё дерево. Он применяет минимальную DOM-операцию, которая нужна для соответствия последнему результату render.

3. Что делает commit phase

Фаза commit применяет результат React render к DOM. Внутри React эта фаза устроена сложнее, но для этой статьи важна основная последовательность:

  1. React применяет DOM mutations: вставляет, удаляет или обновляет DOM-узлы.
  2. React обновляет refs, чтобы они указывали на актуальные DOM-узлы.
  3. React выполняет layout effects, то есть useLayoutEffect.
  4. Браузер получает возможность обновить изображение на экране.
  5. React выполняет passive effects, то есть useEffect, с важными оговорками по времени выполнения.

4. DOM mutations

DOM mutations — это реальные изменения DOM, которые React DOM выполняет во время commit.

React может обновить текстовый узел, изменить class, style, DOM property вроде disabled, checked или value, вставить новый DOM-узел, удалить старое поддерево, обновить ref или применить другие изменения, которые были подготовлены на render/reconciliation.

Например:


function SaveButton({ pending }: { pending: boolean }) {
  return <button disabled={pending}>{pending ? "Saving..." : "Save"}</button>;
}

Если pending меняется с false на true, React не должен пересоздавать всё приложение. Для этой ветки достаточно обновить disabled у кнопки и её текст. Если остальная часть дерева не изменилась, она не требует DOM-мутаций.

Это важное отличие от render phase. Компонент может быть вызван заново, но DOM изменится только там, где результат действительно отличается. Официальная модель React прямо разделяет эти этапы: во время render React вычисляет, что должно быть показано, а во время commit применяет минимально необходимые изменения к DOM.

React управляет DOM внутри root container, который был передан в createRoot или hydrateRoot. DOM вне этого root не становится автоматически частью React-дерева. Если сторонний код меняет DOM внутри React root, он может конфликтовать с React, потому что React считает эту область своей управляемой средой.

5. Refs во время commit

Refs связаны с commit, потому что DOM-ссылку нельзя надёжно получить во время render. Во время render React только вычисляет следующий UI. Реальный DOM ещё не обновлён под этот результат.

После commit ref.current может указывать на DOM-узел:


function SearchInput() {
  const inputRef = useRef<HTMLInputElement | null>(null);

  useLayoutEffect(() => {
    inputRef.current?.focus();
  }, []);

  return <input ref={inputRef} />;
}

Здесь inputRef.current читается в useLayoutEffect, то есть после того, как React применил DOM-изменения. Это подходящий момент для синхронной работы с DOM до repaint.

Refs обычно читают в обработчиках событий, useLayoutEffect или useEffect. В render их используют не для чтения текущего DOM, а как стабильный контейнер значения.

6. useLayoutEffect

useLayoutEffect выполняется после DOM mutations, но до того, как браузер перерисует экран. Поэтому код внутри useLayoutEffect видит DOM уже в состоянии текущего commit.

Это делает useLayoutEffect полезным для задач, где нельзя показать пользователю промежуточный кадр. Классический пример — измерить DOM и сразу скорректировать положение элемента.


function Tooltip() {
  const ref = useRef<HTMLDivElement | null>(null);
  const [height, setHeight] = useState(0);

  useLayoutEffect(() => {
    const rect = ref.current!.getBoundingClientRect();
    setHeight(rect.height);
  }, []);

  return (
    <div ref={ref} style={{ top: -height }}>
      Tooltip
    </div>
  );
}

На первом commit React вставляет DOM. Затем useLayoutEffect измеряет высоту и вызывает setHeight. React может выполнить дополнительный render и commit до того, как браузер покажет кадр пользователю. В результате пользователь видит уже исправленное положение, а не промежуточную версию.

Цена такого поведения — блокировка repaint. Код внутри useLayoutEffect, а также state updates, запущенные из него, не дают браузеру показать кадр, пока React не закончит эту работу. Поэтому useLayoutEffect не должен быть универсальным “кодом после render”. Если логика не должна блокировать визуальное обновление, лучше использовать useEffect.

Базовая последовательность commit → useLayoutEffect → paint → useEffect уже разбиралась в статье React initial render: первый render компонента, commit, useLayoutEffect и useEffect. Здесь важно добавить именно update-смысл: при повторном обновлении useLayoutEffect видит DOM уже после текущих DOM mutations, но до следующего кадра браузера.

7. Browser render после commit

После того как React применил DOM-изменения и выполнил layout effects, браузер может обновить изображение на экране. Это уже не React render, а браузерная работа.

Подробно визуальный конвейер разобран в статье Style → Layout → Paint → Composite. В этой статье достаточно связать его с commit.

Если React изменил текст, className, inline styles или DOM-свойства, браузер должен понять, как это влияет на итоговый кадр. Иногда достаточно композитинга, иногда нужен paint, иногда требуется layout. Например, изменение transform или opacity при подходящих условиях может не требовать layout. А изменение размеров, положения или содержимого часто приводит к пересчёту layout.

Важно не смешивать уровни. React DOM mutations меняют DOM. Браузерный render превращает DOM и CSS в пиксели. React не выполняет CSS layout engine и не рисует пиксели напрямую.

Если commit меняет className, атрибуты или inline styles, браузер может пересчитать стили. Подробности CSS loading и CSSOM уже вынесены в статью CSS loading и CSSOM. В текущем материале CSSOM нужен только как часть браузерного ответа на DOM/CSS-изменения после commit.

8. useEffect

useEffect нужен для синхронизации компонента с внешними системами: сетевыми соединениями, подписками, таймерами, browser APIs, аналитикой, сторонними библиотеками.


useEffect(() => {
  const connection = createConnection(serverUrl, roomId);
  connection.connect();

  return () => {
    connection.disconnect();
  };
}, [serverUrl, roomId]);

Если зависимости изменились, React выполняет cleanup старого effect, а затем setup нового effect. Например, если roomId изменился, старое подключение нужно закрыть, а новое открыть уже с новым значением.

В учебной диаграмме useEffect удобно ставить после browser render: DOM уже изменён, визуальный кадр может быть показан, а passive effects выполняют синхронизацию с внешним миром. Но важно помнить официальную оговорку React: если effect вызван взаимодействием пользователя, React может выполнить его до paint, чтобы результат был заметен системе событий. Поэтому на useEffect нельзя опираться для точных DOM-измерений перед кадром. Для визуальной работы, которая должна произойти до repaint, нужен useLayoutEffect.

useEffect не нужен для вычисления данных, которые можно получить из props или state во время render. Если effect сразу вызывает setState, чтобы записать derived state, React делает лишний цикл: render, commit, effect, снова update, снова render и commit. В таких случаях лучше вычислять значение во время render или использовать useMemo, если вычисление дорогое.

Также useEffect не должен заменять обработчики событий. Если действие должно произойти из-за клика, отправки формы или ввода, оно обычно должно быть в соответствующем обработчике. Effect нужен тогда, когда причина — сам факт, что компонент сейчас отображается с такими props/state и должен быть синхронизирован с внешней системой.

9. useEffect может запустить новый update

Если useEffect вызывает setState, React возвращается к началу update-цикла.


function UserName({ userId }: { userId: string }) {
  const [name, setName] = useState<string | null>(null);

  useEffect(() => {
    let ignore = false;

    fetch(`/api/users/${userId}`)
      .then((response) => response.json())
      .then((data) => {
        if (!ignore) {
          setName(data.name);
        }
      });

    return () => {
      ignore = true;
    };
  }, [userId]);

  return <p>{name ?? "Loading..."}</p>;
}

Первый commit может показать Loading.... После commit запускается effect, начинается запрос. Когда данные приходят, setName создаёт новый update. Дальше React снова проходит queue update, batching, scheduling, render/reconciliation и commit.

10. Производительность commit-пути

Commit находится близко к визуальному обновлению, поэтому тяжёлая работа здесь особенно заметна. Сам commit должен быть коротким, а код в useLayoutEffect — минимальным.

Опасный паттерн — смешивать чтение layout и запись стилей в цикле:


useLayoutEffect(() => {
  items.forEach((item) => {
    const width = item.el.offsetWidth;
    item.el.style.height = `${width}px`;
  });
}, [items]);

Такой код может заставлять браузер синхронно пересчитывать layout снова и снова. Лучше разделять чтение и запись:


useLayoutEffect(() => {
  const widths = items.map((item) => item.el.offsetWidth);

  items.forEach((item, index) => {
    item.el.style.height = `${widths[index]}px`;
  });
}, [items]);

Но даже улучшенный вариант всё равно блокирует repaint, если выполняется в useLayoutEffect. Поэтому главный вопрос не “можно ли это написать в useLayoutEffect”, а “должно ли это обязательно произойти до следующего кадра”. Если нет — лучше использовать useEffect или перенести работу в другой момент.

Подробности forced layout, layout thrashing и стоимости визуальных этапов относятся к браузерной статье Style → Layout → Paint → Composite.

11. Что важно не перепутать

React render и browser render — разные вещи. React render вызывает компоненты и вычисляет UI. Browser render превращает DOM и CSS в визуальный кадр.

Reconciliation и commit — тоже разные вещи. Reconciliation вычисляет, что должно измениться. Commit применяет подготовленный результат к DOM.

Re-render не означает DOM mutation. Компонент может быть вызван заново, но если результат совпал с прошлым, React может не менять DOM.

useLayoutEffect — не обычный effect после render. Он выполняется после DOM mutations и до repaint, поэтому может блокировать показ кадра.

useEffect — не место для derived state и не замена обработчиков событий. Он нужен для синхронизации с внешними системами.

Refs на DOM-узлы становятся надёжно доступны после commit. Читать DOM ref во время render — слишком рано.

DOMContentLoaded и load не означают, что React effects завершены. Эти события относятся к HTML parsing, scripts и ресурсам страницы. Их место в браузерном пути разобрано в статье DOMContentLoaded → subresources → load.
