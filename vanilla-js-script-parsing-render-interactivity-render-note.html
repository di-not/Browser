Vanilla JS: как подключён app script и как он влияет на parsing, DOM и render

Коротко о блоке

Этот блок описывает, что происходит, когда HTML parser встречает JavaScript во время чтения HTML-документа.

Обнаружение CSS / scripts / images / fonts
→ Как подключён app script?
   ├─ blocking classic
   │  → Parser-blocking classic script
   │  → Загрузить и выполнить script сейчас
   │  → DOM создаётся по мере чтения HTML
   │  → Можно сделать render?
   │
   ├─ defer / module
   │  → загружается параллельно
   │  → выполняется после parser complete
   │  → DOM мог измениться / listeners навешены
   │  → Можно сделать render?
   │
   ├─ async
   │  → выполняется когда загрузился, порядок не гарантирован
   │  → может изменить DOM / навесить listeners
   │  → Можно сделать render?
   │
   └─ нет script
      → DOM создаётся по мере чтения HTML
      → Можно сделать render?

Способ подключения script определяет, будет ли HTML parser ждать загрузку и выполнение JavaScript, когда именно код получит доступ к DOM, может ли он изменить уже построенную часть документа и будет ли он задерживать DOMContentLoaded.

1. Границы блока

Блок начинается, когда браузер при чтении HTML обнаруживает script.

<script src="/app.js"></script>
<script defer src="/app.js"></script>
<script async src="/app.js"></script>
<script type="module" src="/app.js"></script>

Блок заканчивается на моменте:

Vanilla JS интерактивен

2. Classic script без async/defer

Classic script — обычный script без type="module".

<script src="/app.js"></script>

Если такой script встречен parser-ом и у него нет async, defer и type="module", он является parser-blocking script. Это значит: HTML parser останавливает чтение HTML, пока script не будет загружен и выполнен.

HTML parser читает HTML
→ встретил <script src="/app.js">
→ остановил parsing
→ загрузил script
→ выполнил script
→ продолжил parsing HTML

Если script inline, загрузки нет, но parsing всё равно останавливается на время выполнения:

<script>
  document.body.classList.add("loaded")
</script>

Код выполняется в момент, когда parser дошёл до этого места документа. Поэтому script видит только ту часть DOM, которая уже построена выше по HTML.

<body>
  <h1 id="title">Title</h1>

  <script>
    document.querySelector("#title") // элемент уже есть
    document.querySelector("#footer") // элемента ещё может не быть
  </script>

  <footer id="footer">Footer</footer>
</body>

Parser-blocking script может:

прочитать уже построенный DOM
изменить уже построенный DOM
добавить listeners на существующие элементы
вызвать document.write()

После выполнения script parser продолжает чтение HTML, и DOM снова создаётся по мере чтения документа.

Загрузить и выполнить script сейчас
→ DOM создаётся по мере чтения HTML

document.write() важен отдельно: если он вызывается во время parsing, он может вставить текст в поток HTML, который parser затем обработает. Поэтому в старых страницах parser-blocking scripts могли менять сам поток документа.

3. Влияние CSS на classic script

Classic script без async/defer блокирует HTML parsing. Но на выполнение такого script могут влиять уже обнаруженные stylesheets.

Причина: JavaScript может читать данные, зависящие от CSS, например через getComputedStyle() или размеры элементов. Поэтому браузер может ждать загрузки и обработки уже обнаруженных stylesheet-ов перед выполнением script.

Подробнее про получение CSS и построение CSSOM.

Пример:

<link rel="stylesheet" href="/styles.css">
<script src="/app.js"></script>

Упрощённо:

stylesheet обнаружен
→ script обнаружен
→ browser может ждать stylesheet
→ script выполняется
→ parser продолжает HTML

Это не значит, что любой CSS всегда останавливает HTML parser. CSS loading обычно не блокирует parsing напрямую. Но CSS может задерживать выполнение scripts, а scripts уже могут задерживать parsing или DOMContentLoaded.

4. defer

defer используется для внешнего classic script.

<script defer src="/app.js"></script>

Такой script загружается параллельно с HTML parsing, но не выполняется сразу при обнаружении.

HTML parser встретил defer script
→ script загружается параллельно
→ parser продолжает читать HTML
→ parser дошёл до конца HTML
→ defer scripts выполняются
→ DOMContentLoaded может быть поставлен в очередь

defer scripts выполняются после завершения parsing HTML и до DOMContentLoaded.

Для нескольких defer scripts сохраняется порядок в документе:

<script defer src="/vendor.js"></script>
<script defer src="/app.js"></script>

vendor.js должен выполниться раньше app.js.

К моменту выполнения defer script DOM основного HTML уже построен. Поэтому код может обращаться к элементам ниже по документу:

<button id="submit">Send</button>

<script defer src="/app.js"></script>
document.querySelector("#submit").addEventListener("click", () => {
  console.log("click")
})

Важно: defer имеет смысл для external script с src. Для inline classic script он не работает как механизм отложенной загрузки.

5. Module script

Module script подключается через type="module".

<script type="module" src="/app.js"></script>

Module scripts по умолчанию ведут себя отложенно: они не блокируют HTML parser и выполняются после того, как HTML document разобран. Перед выполнением браузер загружает module graph — сам модуль и его зависимости.

import { init } from "./init.js"

init()

Упрощённо:

HTML parser встретил <script type="module">
→ модуль и зависимости загружаются
→ parser продолжает HTML
→ после parser complete модуль выполняется
→ DOMContentLoaded ждёт выполнение module scripts

Поэтому в общей диаграмме defer / module можно вести одной веткой:

defer / module
→ загружается параллельно
→ выполняется после parser complete

Различие в том, что module script работает как JavaScript module: имеет import/export, отдельную module scope и загрузку зависимостей.

6. async

async script загружается параллельно и выполняется, когда стал доступен.

<script async src="/analytics.js"></script>

Он не ждёт конца HTML parsing и не сохраняет порядок относительно других async scripts.

HTML parser встретил async script
→ script загружается параллельно
→ когда загрузился, выполняется
→ порядок выполнения не гарантирован

Если async script загрузился до завершения HTML parsing, его выполнение может прервать parsing на время выполнения кода.

parser читает HTML
→ async script загрузился
→ parser временно уступает выполнение script
→ script выполнен
→ parser продолжает HTML

Если async script загрузился позже, он выполнится уже после parsing.

parser дошёл до конца HTML
→ DOMContentLoaded мог уже произойти
→ async script загрузился и выполнился

DOMContentLoaded не ждёт async scripts. Поэтому async подходит для кода, который не должен быть условием готовности DOM: аналитика, виджеты, независимые скрипты.

Async script может изменить DOM или навесить listeners, но момент выполнения зависит от загрузки. Поэтому код внутри async script не должен предполагать, что весь DOM уже построен, если он может выполниться до конца parsing.

7. Нет script

Если script не найден, HTML parser продолжает строить DOM без JavaScript-пауз.

HTML parser читает HTML
→ DOM создаётся по мере чтения HTML
→ parser дошёл до конца HTML

Но отсутствие app script не означает отсутствие интерактивности. Ссылки, формы и стандартные элементы управления работают за счёт нативного поведения браузера.

<a href="/about">About</a>

<form action="/search" method="get">
  <input name="q">
  <button>Search</button>
</form>

JavaScript нужен для дополнительной логики: кастомного состояния, динамического изменения DOM, SPA-навигации, сложных listeners и других сценариев.

8. Как scripts влияют на DOM

JavaScript может работать только с тем DOM, который уже существует на момент выполнения.

Parser-blocking script выполняется прямо в месте встречи:

<h1 id="title">Title</h1>

<script>
  document.querySelector("#title") // есть
  document.querySelector("#later") // ещё нет
</script>

<p id="later">Later</p>

Defer и module scripts выполняются после завершения parsing, поэтому обычно видят весь DOM основного HTML-документа.

parser complete
→ defer/module script executed
→ DOM уже построен

Async script может выполниться до или после parser complete, поэтому его код должен учитывать оба варианта.

Практический приём — проверять document.readyState или слушать DOMContentLoaded, если script зависит от полного DOM.

function init() {
  const button = document.querySelector("button")
  if (button) {
    button.addEventListener("click", () => console.log("click"))
  }
}

if (document.readyState === "loading") {
  document.addEventListener("DOMContentLoaded", init)
} else {
  init()
}

9. Как scripts влияют на DOMContentLoaded

DOMContentLoaded срабатывает после того, как HTML полностью разобран, а deferred scripts и module scripts загружены и выполнены.

Подробнее про DOMContentLoaded.

parser complete
→ выполнить defer/module scripts
→ DOMContentLoaded

Parser-blocking classic scripts выполняются ещё во время parsing, поэтому они задерживают достижение parser complete.

parser встретил blocking script
→ parsing остановлен
→ script загружен и выполнен
→ parsing продолжен
→ parser complete позже

Async scripts не являются условием для DOMContentLoaded.

async script загрузился до DCL → может выполниться до DCL
async script загрузился после DCL → выполнится после DCL
DOMContentLoaded его не ждёт

Stylesheets связаны с этим косвенно: deferred и module scripts ждут stylesheets, а DOMContentLoaded ждёт deferred/module scripts. Поэтому CSS может косвенно задержать DOMContentLoaded.

10. Как scripts связаны с render

Script сам по себе не является стадией render pipeline. Он может изменить входные данные для render:

изменить DOM
изменить class
изменить style
добавить или удалить элементы
изменить текст
навесить listeners

Подробнее про Style → Layout → Paint → Compositing.

Если DOM или стили изменились, браузер может снова выполнить нужные стадии:

Style
→ Layout
→ Paint
→ Composite

Render не обязан ждать app script всегда.

defer/module script ещё не выполнен
→ браузер всё равно может показать HTML,
   если DOM и критичные стили готовы

С другой стороны, parser-blocking script может задержать появление DOM ниже себя, а значит может задержать первый render той части страницы, которая находится ниже script.

Поэтому в диаграмме стрелки к Можно сделать render? идут от нескольких входов:

DOM создаётся по мере чтения HTML
CSSOM / нет внешнего CSS
async effect / listeners

Это означает, что render зависит от доступной DOM-структуры, применимых стилей и изменений, которые внёс script.

11. Страница видна и интерактивность

После render страница может быть видна пользователю, но это ещё не всегда означает готовность всего поведения.

Native browser behavior — встроенное поведение браузера, которое не требует пользовательского JavaScript-кода. Например, ссылка с href может запускать навигацию, форма с action и method может отправлять данные, поле ввода может получать фокус и принимать текст.

<a href="/about">About</a>

<form action="/search" method="get">
  <input name="q">
  <button>Search</button>
</form>

Если для поведения достаточно встроенных возможностей HTML, страница может быть интерактивной сразу после того, как соответствующие элементы доступны пользователю.

Страница видна
→ для поведения JS не нужен
→ работает native browser behavior
→ Vanilla scenario интерактивен

Если поведение зависит от JavaScript, видимости страницы недостаточно. Код приложения должен выполниться и зарегистрировать обработчики событий.

Event listener — функция, которую браузер вызовет, когда на выбранный объект придёт определённое событие. Например, addEventListener("click", handler) регистрирует обработчик клика.

const button = document.querySelector("button")

button.addEventListener("click", () => {
  console.log("clicked")
})

В этом примере выполнение app script делает страницу интерактивной для custom JS-поведения, но не меняет внешний вид страницы. Регистрация обработчика события сама по себе не добавляет DOM-элементы, не меняет текст, не меняет class, не меняет inline styles и не меняет CSS-правила. Поэтому одно только навешивание listeners обычно не требует нового прохода через rendering pipeline.

Страница видна
→ ждать выполнения app script
→ JS навесил обработчики
→ Vanilla scenario интерактивен

В этой цепочке нет обязательного шага:

Style
→ Layout
→ Paint
→ Composite

Новый render нужен только тогда, когда при выполнении app script изменилось что-то, влияющее на отображение.

Пример без визуального изменения:

button.addEventListener("click", handleClick)
JS зарегистрировал listener
→ pixels не изменились
→ render pipeline не нужен
→ Vanilla scenario интерактивен

Пример с визуальным изменением:

button.addEventListener("click", handleClick)

button.classList.add("ready")
button.textContent = "Ready"
button.style.display = "block"
JS зарегистрировал listener
+ изменил class / text / style
→ браузер должен обновить визуальный результат
→ Style / Layout / Paint / Composite по необходимости

Не каждое визуальное изменение запускает весь pipeline. Если изменился class, браузер сначала пересчитывает стили. Если новое значение влияет на размеры или положение, понадобится Layout. Если меняется только цвет или фон, Layout может быть не нужен, но может понадобиться Paint. Если меняется transform или opacity в подходящих условиях, браузер может обойтись Composite.

Момент готовности custom JS-поведения зависит от способа подключения script:

blocking classic
→ может навесить handlers во время parsing,
   но только на уже существующие DOM-элементы

defer / module
→ обычно навешивает handlers после parser complete,
   до DOMContentLoaded

async
→ может навесить handlers раньше или позже,
   в зависимости от момента загрузки и выполнения

Важное различие:

видимость страницы
≠
готовность custom JS-поведения

Элемент может быть уже виден и работать нативно, но custom-логика появится только после выполнения script. Например, обычная кнопка может быть видимой, но открытие кастомного меню начнёт работать только после того, как script найдёт эту кнопку и зарегистрирует обработчик события.

Ещё одно важное различие:

готовность custom JS-поведения
≠
обязательная перерисовка

Если script только зарегистрировал обработчики, страница стала интерактивнее с точки зрения поведения, но её пиксели могли не измениться.

12. Что важно не перепутать

parser-blocking ≠ render-blocking

Обычный classic script без async/defer блокирует HTML parsing. Он не обязательно является render-blocking в том же смысле, что критичный CSS, но может задержать render, потому что задерживает построение DOM.

defer ≠ async

defer ждёт конца parsing и сохраняет порядок. async выполняется, когда загрузился, порядок не гарантирован.

module ≈ defer по времени выполнения, но это не classic script

Module script имеет module graph, import/export и отдельную область видимости.

async ≠ DOM готов

Async script может выполниться до того, как parser дошёл до конца HTML.

DOMContentLoaded ≠ async scripts loaded

DOMContentLoaded не ждёт async scripts.

script executed ≠ UI уже отрисован

После выполнения script браузеру ещё может понадобиться Style, Layout, Paint и Composite.

listeners навешены ≠ элемент был виден до этого момента

Элемент мог быть виден раньше, но стать интерактивным для custom JS только после навешивания listener.

страница видна ≠ custom JS-поведение готово

Первый render может показать кнопку, меню или форму до того, как app script зарегистрировал обработчики.

native browser behavior ≠ app behavior

Ссылки, формы и поля ввода имеют встроенное поведение браузера. Кастомная логика, например открытие собственного dropdown, требует выполнения JavaScript.

готовность custom JS-поведения ≠ обязательная перерисовка

Если app script только зарегистрировал обработчики событий, pixels не меняются. Rendering pipeline нужен только при изменениях DOM, текста, class, inline styles, CSS-правил или других данных, влияющих на отображение.
