Next.js App Router: гидратация Client Components

Коротко

Гидратация — это этап, на котором React связывает Client Components с уже существующим HTML, полученным от сервера, и подключает обработчики событий. До гидратации пользователь уже может видеть серверный HTML, но интерактивная React-логика Client Components ещё не обязана работать.

В App Router начальная загрузка устроена слоями:

Server HTML
→ быстрый неинтерактивный предварительный результат

RSC payload
→ описание результата Server Components и мест Client Components

Client JavaScript
→ код Client Components и клиентской среды

Hydration Client Components
→ React связывает Client Components с существующим DOM
→ обработчики событий подключены
→ Client Components interactive

Server Components могут быть видимы как HTML, но сами не гидрируются. Гидратация относится к Client Components: именно они имеют клиентский JavaScript, состояние, эффекты, обработчики событий и доступ к API браузера.

1. Место гидратации в ветке Next.js App Router

Гидратация начинается после того, как у браузера уже есть несколько вещей:

серверный HTML
+ RSC payload
+ загруженный и запущенный клиентский JavaScript
+ понятные границы Client Components

До этого момента браузер уже мог разобрать HTML, построить DOM, показать серверный HTML, загрузить клиентские JavaScript-фрагменты, запустить клиентский bootstrap Next.js / React и использовать RSC payload для согласования Server и Client деревьев. Но всё это ещё не равно гидратации.

Гидратация — более узкий этап: React берёт существующий DOM и связывает его с Client Components.

2. Что такое гидратация

Гидратация — процесс React, в котором серверный статический HTML становится интерактивным. Next.js определяет её как прикрепление обработчиков событий к DOM. React hydrateRoot описывает это как “attach” React к существующему HTML, который уже был создан серверным React-rendering.

Проще:

HTML уже есть в DOM
→ React не создаёт этот DOM с нуля
→ React сопоставляет его с Client Components
→ React подключает обработчики событий
→ UI становится интерактивным

Пример:

"use client";

export function LikeButton({ likes }: { likes: number }) {
  return <button onClick={() => console.log("liked")}>{likes} likes</button>;
}

На сервере Next.js может подготовить HTML-предпросмотр:

<button>10 likes</button>

Пользователь может увидеть кнопку до того, как React подключил к ней onClick. После гидратации React связывает Client Component LikeButton с существующей кнопкой в DOM, и React-обработчик события становится рабочим.

3. Почему гидратация вообще нужна

Без гидратации серверный HTML остаётся только HTML. Он может быть полезным: его можно показать пользователю, браузер может применить CSS, поисковый робот может прочитать текст, пользователь может видеть структуру страницы. Но React-логика Client Components ещё не подключена.

Client Component нужен для того, что должно работать в браузере:

  • состояние через useState;
  • обработчики событий вроде onClick, onChange, onSubmit;
  • эффекты через useEffect;
  • API браузера вроде window, localStorage, navigator;
  • пользовательские хуки, которые зависят от клиентской среды.

Гидратация нужна, чтобы эта логика стала связана с HTML, который уже пришёл с сервера.

Важно: нативное поведение браузера и React-интерактивность — не одно и то же. Например, обычная ссылка <a href="/about"> может работать как стандартная навигация браузера ещё до React-гидратации. Но React-обработчик onClick, состояние компонента, клиентский роутинг Next.js и логика Client Component требуют клиентского JavaScript и гидратации соответствующей части.

4. Что именно гидрируется

В App Router гидрируются Client Components, а не Server Components.

Server Component:

// app/page.tsx
import LikeButton from "./like-button";

export default async function Page() {
  const post = await getPost();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.text}</p>
      <LikeButton likes={post.likes} />
    </article>
  );
}

Client Component:

// app/like-button.tsx
"use client";

import { useState } from "react";

export default function LikeButton({ likes }: { likes: number }) {
  const [count, setCount] = useState(likes);

  return <button onClick={() => setCount(count + 1)}>{count} likes</button>;
}

В этом примере:

<h1> и <p>
→ результат Server Component
→ видимы как HTML
→ сами не гидрируются как Client Components

<LikeButton />
→ граница Client Component
→ требует client JavaScript
→ проходит гидратацию
→ получает рабочий onClick и state

Server Component уже выполнился на сервере. Его задача — получить данные, сформировать результат, попасть в HTML и RSC payload. Он не становится браузерным компонентом с состоянием и обработчиками событий.

Client Component может иметь серверный HTML-предпросмотр при начальной загрузке, но его интерактивность появляется после гидратации.

5. Что происходит во время гидратации

Во время гидратации React делает несколько связанных вещей.

Сначала React берёт уже существующий DOM. Этот DOM появился из серверного HTML, который Next.js отдал браузеру.

Затем React строит клиентское представление для Client Components. Для этого уже должен быть доступен клиентский JavaScript: модули Client Components, их зависимости и клиентская среда Next.js / React.

После этого React сопоставляет ожидаемый результат Client Component с тем HTML, который уже существует в DOM. Если серверный HTML и первый клиентский результат совпадают, React может “прикрепиться” к существующим узлам вместо того, чтобы пересоздавать DOM с нуля.

Затем React подключает обработчики событий и начинает управлять соответствующей частью DOM.

Упрощённо:

DOM уже содержит <button>10 likes</button>

React знает:
- здесь Client Component LikeButton
- props: { likes: 10 }
- модуль LikeButton уже доступен

Hydration:
- сопоставить Client Component с DOM
- сохранить существующий DOM
- подключить обработчики событий
- разрешить дальнейшие state updates

После этого пользовательский клик по кнопке обрабатывается React-логикой.

6. Client Components подключили event handlers

После гидратации Client Components получают рабочие обработчики событий.

В JSX обработчик события выглядит как prop:

<button onClick={() => setCount(count + 1)}>{count}</button>

В серверном HTML нет “React-обработчика” в таком виде. Сервер может отдать HTML кнопки, но функция () => setCount(count + 1) не является частью HTML-документа. Эта функция находится в клиентском JavaScript.

Во время гидратации React связывает эту функцию с DOM-событием. Поэтому после гидратации:

пользователь кликает по кнопке
→ браузер создаёт DOM event
→ React получает событие
→ React вызывает onClick
→ setState запускает обновление Client Component
→ React обновляет UI

До гидратации пользователь мог видеть кнопку, но React-обработчик ещё не был подключён.

7. Client Components interactive

Client Components interactive означает, что клиентские части приложения готовы реагировать на пользовательские действия через React.

Интерактивность может включать:

  • клики;
  • ввод в поля формы;
  • отправку формы;
  • раскрытие модального окна;
  • изменение состояния компонента;
  • чтение window, localStorage, navigator;
  • запуск эффектов;
  • клиентскую навигацию.

Важно: интерактивность не обязательно появляется сразу у всей страницы как единый глобальный момент. В современных React/Next приложениях разные части могут становиться интерактивными независимо, особенно когда используются потоковая передача HTML и границы Suspense.

Поэтому точнее говорить не “страница гидратирована вся сразу”, а:

конкретная Client Component boundary прошла гидратацию
→ её обработчики событий подключены
→ эта часть UI интерактивна

8. Selective hydration

Selective hydration — избирательная гидратация. Это механизм, при котором React может гидратировать части страницы независимо, а не обязательно одним большим блокирующим проходом по всей странице.

Next.js Streaming guide связывает selective hydration с Suspense boundaries: каждая граница Suspense может быть единицей гидратации. Без таких границ React гидратирует страницу одним блокирующим проходом; с ними работа разбивается на меньшие задачи, которые могут уступать управление браузеру.

Это важно для производительности. Если страница большая, монолитная гидратация может надолго занять главный поток браузера. Если работа разбита по границам, браузер может быстрее реагировать на действия пользователя, а React может приоритизировать ту часть, с которой пользователь взаимодействует.

Упрощённо:

Страница без полезных границ
→ большая гидратация одним проходом

Страница с Suspense boundaries
→ несколько меньших единиц гидратации
→ React может приоритизировать важные части

Selective hydration не меняет определение гидратации. Это всё ещё прикрепление React-логики к существующему DOM. Меняется способ организации работы: не всё обязательно должно гидратироваться как один монолит.

9. Гидратация и Suspense

Suspense позволяет показывать временный UI, пока дочерний контент ещё не готов. В Next.js App Router он важен не только для визуального состояния загрузки, но и для разбиения работы: потоковая передача HTML и selective hydration интегрированы с границами Suspense.

В рамках гидратации нужно понимать две роли Suspense.

Первая роль — потоковый HTML. Граница Suspense помогает серверу отправить готовые части раньше и позже дослать содержимое, которое ожидало данные.

Вторая роль — единица гидратации. Граница Suspense помогает React не воспринимать всю страницу как один неделимый кусок работы.

Практический вывод: если интерактивная часть находится внутри отдельной границы, React получает больше возможностей гидратировать её независимо и приоритизировать работу, связанную с пользовательским взаимодействием.

10. Гидратация и useEffect

useEffect — это не механизм гидратации. Это хук React для синхронизации компонента с внешней системой: браузерным API, подпиской, таймером, сетью и другими side effects.

В теме гидратации useEffect часто появляется как способ избежать mismatch. Если часть UI должна отличаться только на клиенте, её не стоит менять прямо в render logic через typeof window !== 'undefined', потому что первый клиентский render должен совпасть с серверным HTML. Вместо этого можно сначала отрендерить одинаковый результат, а уже на клиенте через useEffect переключить состояние.

Пример:

"use client";

import { useEffect, useState } from "react";

export function ClientOnlyLabel() {
  const [mounted, setMounted] = useState(false);

  useEffect(() => {
    setMounted(true);
  }, []);

  return <p>{mounted ? "Client only value" : "Server-safe value"}</p>;
}

Здесь первый результат должен быть безопасным для серверного HTML и первого клиентского render. После этого effect может изменить состояние уже на клиенте.

Важно: такой подход может сделать UI “двухфазным”: сначала пользователь видит серверно-безопасную версию, затем клиентскую. Это нормально для некоторых задач, но не нужно использовать этот приём без необходимости.

11. Hydration mismatch

Hydration mismatch — ситуация, когда HTML, созданный сервером, не совпадает с тем, что React получает при первом клиентском render во время гидратации.

React ожидает, что первый клиентский результат совпадёт с серверным результатом. Это важно: пользователь уже смотрит на серверный HTML до загрузки JavaScript, и внезапная замена содержимого ломает ощущение быстрой загрузки. Кроме того, mismatch может привести к неправильному поведению, включая риск привязки обработчиков событий не к тем элементам.

Пример проблемы:

"use client";

export function TimeLabel() {
  return <p>{new Date().toLocaleTimeString()}</p>;
}

Сервер отрендерил одно время, клиент во время гидратации получил другое. Результат может не совпасть.

Другой пример:

"use client";

export function WidthLabel() {
  if (typeof window !== "undefined") {
    return <p>Client width: {window.innerWidth}</p>;
  }

  return <p>Server render</p>;
}

Серверный HTML и первый клиентский render могут отличаться. Такую логику лучше переносить в useEffect или проектировать так, чтобы первый серверный и первый клиентский результат совпадали.

12. Частые причины hydration mismatch

Частые причины:

  • неправильная вложенность HTML-тегов;
  • интерактивный контент внутри интерактивного контента, например <button> внутри <button>;
  • проверки typeof window !== 'undefined' прямо в render logic;
  • использование window, localStorage, matchMedia и других browser-only API во время render;
  • значения, зависящие от времени: Date, таймзона, локаль, текущее время;
  • случайные значения вроде Math.random() во время render;
  • различие данных на сервере и клиенте;
  • расширения браузера, которые меняют HTML;
  • CDN или Edge-настройки, которые модифицируют HTML-ответ;
  • некорректная настройка CSS-in-JS.

Главное правило:

server render output
должен совпасть с initial client render output

Если первое клиентское отображение отличается от серверного HTML, React получает mismatch.

13. Как исправлять mismatch

Базовая стратегия — сделать первый серверный render и первый клиентский render одинаковыми.

Если значение должно появиться только на клиенте, используйте состояние и useEffect:

"use client";

import { useEffect, useState } from "react";

export function BrowserValue() {
  const [value, setValue] = useState<string | null>(null);

  useEffect(() => {
    setValue(window.localStorage.getItem("key"));
  }, []);

  return <p>{value ?? "Loading value..."}</p>;
}

Если компонент вообще не должен рендериться на сервере, в Next.js можно использовать dynamic с ssr: false:

import dynamic from "next/dynamic";

const NoSSR = dynamic(() => import("./client-only-widget"), {
  ssr: false,
});

export default function Page() {
  return <NoSSR />;
}

Если различие неизбежно и ограничено текстом, можно использовать suppressHydrationWarning, но это escape hatch. Его не нужно превращать в обычный способ “чинить” несоответствия.

<time suppressHydrationWarning>{new Date().toISOString()}</time>

Лучший вариант почти всегда — спроектировать компонент так, чтобы начальный HTML и первый клиентский render совпадали.

14. Гидратация и повторный render

Гидратация не равна обычному первому render в React SPA.

В обычной React SPA без server-rendered HTML часто используется createRoot(root).render(<App />). React сам создаёт DOM внутри root container.

При гидратации используется уже существующий DOM. React не должен создавать всё с нуля. Он должен привязаться к HTML, который уже был создан на сервере.

Сравнение:

React SPA initial render
→ root пустой или почти пустой
→ React создаёт DOM

Hydration
→ DOM уже содержит server HTML
→ React сопоставляет DOM с компонентами
→ React подключает логику

Поэтому для server-rendered HTML используется не createRoot, а hydrateRoot на уровне React. В Next.js этот вызов скрыт внутри фреймворка.

15. Гидратация и обновления после неё

После гидратации React начинает управлять соответствующей частью DOM. Дальше Client Components работают как обычные клиентские React-компоненты.

Например:

"use client";

import { useState } from "react";

export function Counter({ initialCount }: { initialCount: number }) {
  const [count, setCount] = useState(initialCount);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

До гидратации пользователь может видеть кнопку с initialCount. После гидратации к ней подключается обработчик события. После клика setCount запускает обычное React-обновление: render phase, commit phase, DOM update и browser paint.

То есть:

гидратация
→ подключила существующий DOM к React

последующие updates
→ React уже обновляет UI как клиентское приложение

16. Гидратация и последующие навигации

Гидратация особенно важна для initial load: когда пользователь открывает страницу напрямую, обновляет страницу или получает server-rendered HTML.

На последующих клиентских навигациях в App Router модель меняется. Next.js может использовать RSC payload для нового маршрута, а Client Components рендерятся на клиенте без server-rendered HTML для этой новой навигации. Это уже не та же самая гидратация исходного server HTML, который пришёл при первом открытии страницы.

Упрощённо:

Первичная загрузка / refresh
→ серверный HTML уже есть
→ нужна гидратация Client Components

Клиентская навигация внутри приложения
→ Next.js получает RSC payload
→ Client Components работают на клиенте
→ новая часть UI не обязана проходить ту же гидратацию server HTML

Поэтому термин “гидратация” чаще всего относится к первому превращению server-rendered HTML в интерактивное приложение.

17. Как понять, что часть UI ещё не гидратирована

Пользователь может видеть элемент, но React-логика ещё не работает.

Признаки:

  • элемент виден, но onClick не реагирует как ожидается;
  • кнопка выглядит правильно, но состояние не меняется;
  • интерактивный виджет не отвечает на ввод;
  • клиентская навигация ещё не перехватывает действие;
  • DevTools показывает, что JavaScript ещё грузится или main thread занят.

Но важно не путать это с ошибками приложения. Если обработчик не работает после полной загрузки, это может быть уже не “ещё не гидратировано”, а bug в коде, ошибка JavaScript, mismatch или неправильная граница Client Component.

18. Что важно не перепутать

Hydration ≠ загрузка JavaScript

Загрузка JavaScript означает, что файл получен браузером. Гидратация означает, что React связал Client Component с существующим DOM.

Hydration ≠ выполнение JavaScript-файла

Модуль может быть выполнен, но React ещё должен сопоставить компонент с DOM и подключить обработчики событий.

Hydration ≠ RSC payload

RSC payload описывает Server Components tree и места Client Components. Гидратация применяет клиентскую React-логику к существующему DOM.

Hydration ≠ Server Component render

Server Components выполняются на сервере. Они могут быть видимы как HTML, но сами не гидрируются.

Hydration ≠ createRoot render

createRoot(...).render(...) создаёт/обновляет DOM в клиентском React-приложении. hydrateRoot(...) прикрепляет React к уже существующему server HTML.

Hydration mismatch ≠ обычная ошибка CSS

Mismatch связан с несовпадением server-rendered HTML и первого client render.

Client Components interactive ≠ вся страница полностью готова

Интерактивной может стать конкретная клиентская граница, пока другие части ещё загружаются или ожидают работу main thread.
