React client update: user action → event handler → queue update → batching

Коротко

Эта статья разбирает первую часть клиентского обновления React:


Пользователь совершает действие
→ Browser DOM event
→ React event system / Synthetic Event
→ Event handler
→ setState / dispatch
→ Queue update
→ Batching
→ дальше React будет планировать работу

Главная идея:

Обработчик события не обновляет DOM напрямую. Он выполняется как JavaScript-функция, вызывает setState или dispatch, а React ставит обновления в очередь и группирует их. DOM будет изменён позже, после scheduling, render/reconciliation и commit.

Эта статья относится к уже интерактивной клиентской части приложения: например, к Client Components в Next.js App Router после гидратации. Server Components здесь не участвуют как интерактивные компоненты: пользовательское событие обрабатывается клиентским JavaScript.

1. Общая модель

Когда пользователь кликает по кнопке, вводит текст или отправляет форму, React не “магически” меняет DOM в момент клика.

Сначала событие возникает на уровне браузера. Затем React пропускает его через свою event system и вызывает обработчик, который был указан в JSX. Если внутри обработчика вызывается setState или dispatch, React получает запрос на обновление состояния. Этот запрос попадает в очередь. Если таких запросов несколько, React может сгруппировать их, чтобы не запускать лишние render-проходы.

Смысловая цепочка:


click / input / submit
→ браузер создаёт DOM event
→ React получает событие через свою event system
→ React вызывает event handler из JSX
→ handler вызывает setState / dispatch
→ React ставит update в очередь
→ React группирует updates
→ следующий этап: scheduling

Ни один из блоков до batching не означает, что DOM уже обновлён. DOM будет обновляться позже, если после render/reconciliation React дойдёт до commit phase и найдёт изменения.

2. User action → Browser DOM event

Пользовательское действие начинается не в React, а в браузере.

Примеры действий:


click
input
change
submit
keydown
pointermove
focus
blur

Когда такое действие происходит, браузер создаёт объект события и доставляет его в event system DOM. У события есть цель (target), текущий обработчик (currentTarget), фазы распространения, возможность отменить default behavior и возможность остановить распространение.

Пример браузерной модели без React:


button.addEventListener('click', event => {
  console.log(event.target);
  console.log(event.currentTarget);
});

В React разработчик обычно не вызывает addEventListener для обычных UI-событий напрямую. Вместо этого он передаёт обработчик как prop в JSX:


<button onClick={handleClick}>
  Save
</button>

Но под React-синтаксисом всё равно лежит браузерная реальность: пользователь совершает действие, браузер создаёт DOM event, дальше React обрабатывает его своим слоем.

3. React event system / Synthetic Event

В React обработчик получает React event object. Его также называют SyntheticEvent.

Пример:


function SaveButton() {
  function handleClick(event: React.MouseEvent<HTMLButtonElement>) {
    console.log(event);
  }

  return <button onClick={handleClick}>Save</button>;
}

React event object похож на стандартный DOM event и поддерживает знакомые методы:


event.preventDefault()
event.stopPropagation()
event.target
event.currentTarget
event.nativeEvent

Но это не обязательно тот же самый объект, который браузер создал изначально. React event object нормализует поведение между браузерами и отражает то, что ожидает React-код. Если нужен исходный браузерный event, он доступен через event.nativeEvent.

Важные детали:


React event object ≠ всегда прямой native DOM event

event.nativeEvent
→ исходный DOM event

event.currentTarget
→ элемент, к которому привязан текущий React handler

event.target
→ элемент, на котором событие произошло

React docs отдельно предупреждает, что некоторые React events не соответствуют напрямую нативным браузерным событиям. Например, для onMouseLeave nativeEvent может указывать на другой тип браузерного события. Такое сопоставление не является публичным API, на который нужно опираться.

Ещё одна важная деталь: React под капотом прикрепляет event handlers на root, но React event object скрывает эту реализацию. Поэтому event.currentTarget в React отражает ожидаемую для React-дерева точку обработки, а не обязательно currentTarget исходного nativeEvent.

4. Event handler

Event handler — это функция, которую React вызывает в ответ на событие.

Пример:


function CounterButton() {
  function handleClick() {
    console.log('clicked');
  }

  return <button onClick={handleClick}>Click</button>;
}

Обработчик нужно передать, а не вызвать во время render:


// правильно
<button onClick={handleClick}>Click</button>

// неправильно
<button onClick={handleClick()}>Click</button>

Во втором варианте handleClick() выполнится сразу во время render, а не при клике.

Event handler обычно объявляют внутри компонента, поэтому он может читать props, state и локальные переменные, которые были доступны во время того render-а, где этот handler был создан.

Пример:


function Counter({ step }: { step: number }) {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + step);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Здесь handleClick видит count и step из конкретного render-а. Это важно для понимания state snapshot.

5. Event handlers могут иметь side effects

React-компоненты и Hooks должны быть чистыми во время render. Event handlers — другое место. Они выполняются не во время render, а в ответ на действие пользователя, поэтому именно в них обычно размещают побочные действия, связанные с интеракцией.

Примеры допустимой логики в handler:


function Form() {
  const [name, setName] = useState('');

  function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
    setName(event.target.value);
  }

  function handleSubmit(event: React.FormEvent) {
    event.preventDefault();
    console.log('submit', name);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={name} onChange={handleChange} />
      <button>Send</button>
    </form>
  );
}

Здесь обработчики:


читают event
вызывают preventDefault
обновляют state
выполняют прикладную логику

Это нормально. Но важно: даже если handler вызывает setState, он не меняет DOM сразу. Он только сообщает React, что нужно подготовить обновление.

6. Propagation и default behavior

React events поддерживают распространение события вверх по дереву.

Пример:


function Toolbar() {
  return (
    <div onClick={() => console.log('toolbar')}>
      <button onClick={() => console.log('button')}>
        Play
      </button>
    </div>
  );
}

При клике по кнопке сначала выполнится обработчик кнопки, затем событие всплывёт к родительскому div.

Чтобы остановить распространение:


<button
  onClick={event => {
    event.stopPropagation();
    console.log('button');
  }}
>
  Play
</button>

stopPropagation() не отменяет default behavior браузера. Он останавливает прохождение события дальше по дереву обработчиков.

Для отмены default behavior используется preventDefault().

Типичный пример — форма:


<form
  onSubmit={event => {
    event.preventDefault();
    sendForm();
  }}
>
  <button>Send</button>
</form>

Без preventDefault() обычная HTML-форма может выполнить стандартную отправку и перезагрузить страницу. В React-приложении часто нужно отменить это поведение и обработать отправку в JavaScript.

Нужно различать:


event.stopPropagation()
→ не даёт событию пройти к родительским React handlers

event.preventDefault()
→ отменяет стандартное браузерное действие, если оно есть и если событие cancelable

7. Event handler и state snapshot

State в React не ведёт себя как обычная изменяемая переменная внутри уже выполняющегося кода.

Когда React вызывает компонент, компонент возвращает JSX snapshot: UI, props, state, локальные переменные и event handlers для конкретного render-а. Обработчик события, созданный в этом render-е, видит именно эти значения.

Пример:


function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    console.log(count);
    setCount(count + 1);
    console.log(count);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Если на экране 0, оба console.log(count) внутри одного handler-а выведут 0.

Причина:


count внутри handler-а
→ значение из текущего render snapshot

setCount(count + 1)
→ запросить следующий render с новым значением

текущая переменная count
→ не меняется внутри уже выполняющегося handler-а

Вызов setState не переписывает переменную, которую текущий handler уже получил из render-а. Он ставит React задачу подготовить следующий render.

8. `setState` не обновляет DOM сразу

setState — это запрос на обновление состояния.

Пример:


function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Когда handler вызывает setCount(count + 1), React получает новое желаемое значение для следующего render-а. Но это ещё не DOM mutation.

Смысл:


setCount(...)
→ React запоминает update
→ React позже вызовет компонент снова
→ компонент вернёт новый JSX snapshot
→ React сравнит результат
→ если есть изменения, commit phase обновит DOM

Текущая статья заканчивается раньше render/reconciliation и commit phase. Поэтому в этой статье важно удержать границу:


setState
≠
render уже произошёл

render
≠
DOM уже изменён

DOM mutation
→ произойдёт только на commit phase

9. `dispatch` из `useReducer`

useReducer — другой способ описать state update. Он особенно удобен, когда логика обновления сложная или когда несколько обработчиков создают разные действия.

Пример:


type State = {
  count: number;
};

type Action =
  | { type: 'increment' }
  | { type: 'reset' };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 };
    case 'reset':
      return { count: 0 };
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });

  function handleClick() {
    dispatch({ type: 'increment' });
  }

  return <button onClick={handleClick}>{state.count}</button>;
}

На уровне flow dispatch похож на setState:


event handler
→ dispatch(action)
→ React получает action
→ action попадёт в обработку state queue
→ во время следующего render-а React применит reducer
→ reducer вернёт next state

Reducer должен быть чистой функцией: получить state и action, вернуть следующий state. Он не должен мутировать старый state, выполнять запросы, менять DOM или зависеть от случайных внешних эффектов.

В handler-е желательно описывать действие пользователя:


dispatch({ type: 'submitted_form' });
dispatch({ type: 'changed_name', nextName: event.target.value });
dispatch({ type: 'reset_clicked' });

Action не обязан описывать конкретную мутацию поля. Он часто описывает то, что произошло в UI.

10. Queue update

Queue update означает, что React не обязан обработать update немедленно в момент вызова setState.

Когда handler вызывает:


setCount(count + 1);

React получает update вида:


заменить count на значение count + 1,
вычисленное из текущего render snapshot

Когда handler вызывает:


setCount(c => c + 1);

React получает updater function:


когда будешь обрабатывать очередь,
возьми pending state
и верни pending state + 1

Эти два варианта выглядят похоже, но ведут себя по-разному, если обновлений несколько.

11. Передача значения: `setCount(count + 1)`

Рассмотрим пример:


function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Если на экране 0, внутри текущего handler-а count тоже 0. Поэтому все три вызова превращаются в одно и то же:


setCount(0 + 1)
setCount(0 + 1)
setCount(0 + 1)

Для React это три запроса “заменить state на 1”. В результате следующий state будет 1, а не 3.

Это не баг batching. Это следствие state snapshot: handler видит фиксированное значение count из render-а, в котором handler был создан.

12. Updater function: `setCount(c => c + 1)`

Если новое значение зависит от предыдущего значения той же state-переменной, можно передать updater function.


function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(c => c + 1);
    setCount(c => c + 1);
    setCount(c => c + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Теперь React получает не три одинаковых значения, а три функции, которые нужно применить последовательно к pending state.

Если pending state был 0, очередь можно представить так:


c => c + 1
c => c + 1
c => c + 1

Обработка:


0 → 1
1 → 2
2 → 3

Следующий state будет 3.

Updater function особенно важна, когда:


новое значение зависит от предыдущего значения той же state-переменной
несколько updates вызываются в одном handler-е
update может быть отложен
нужно избежать чтения устаревшего snapshot

13. Смешивание replace update и updater function

React обрабатывает очередь updates по порядку. В очереди могут быть как “заменить state на значение”, так и updater functions.

Пример:


function handleClick() {
  setNumber(number + 5);
  setNumber(n => n + 1);
}

Если number в текущем snapshot равен 0, первый update означает:


replace with 5

Второй update означает:


взять pending state и прибавить 1

Итог:


0
→ replace with 5
→ 5 + 1
→ 6

Другой пример:


function handleClick() {
  setNumber(number + 5);
  setNumber(n => n + 1);
  setNumber(42);
}

Если number был 0, итог будет 42:


0
→ replace with 5
→ 6
→ replace with 42
→ 42

Здесь важно понять не “последний setState всегда побеждает”, а более точное правило:


React обрабатывает queued updates по порядку.
Значение replace update заменяет pending state.
Updater function получает pending state и возвращает следующий.

Если все updates — простые replace values, последний replace действительно может определить итог. Если в очереди есть updater functions, итог зависит от порядка и типа каждого update.

14. Batching

Batching — это группировка нескольких state updates в один render-проход.

Классическая модель для React events:


handler начал выполняться
→ внутри handler-а вызываются setState / dispatch
→ React ставит updates в очередь
→ handler завершился
→ React обрабатывает updates вместе
→ один render вместо нескольких

React docs прямо объясняет, что React ждёт окончания выполнения всего кода в event handler перед обработкой state updates. Поэтому UI не обновляется посередине handler-а.

Пример:


function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
  console.log('handler still running');
}

React не обязан обновлять экран после setCount, потом ещё раз после setFlag. Он может дождаться завершения handler-а и обработать updates вместе.

Это даёт два эффекта:


меньше render-проходов
нет промежуточного UI, где обновилась только часть state

15. React не batch-ит разные intentional events как один общий update

Batching не означает, что React бесконечно собирает все updates со всей страницы.

Отдельные намеренные действия пользователя, например разные клики, обрабатываются отдельно. Это важно для корректности UI. Например, если первый клик отключает форму, второй клик не должен быть объединён с первым так, чтобы форма успела отправиться повторно.

Модель:


click #1
→ свой handler
→ свой batch updates
→ React может обновить UI

click #2
→ отдельный handler
→ отдельный batch updates

React применяет batching там, где это безопасно для пользовательского взаимодействия.

16. Automatic batching в React 18+

В React 18 с createRoot batching стал автоматическим для большего числа источников updates.

До React 18 React по умолчанию batch-ил updates внутри React event handlers, но updates из setTimeout, Promise callbacks, native event handlers и других источников могли вести себя иначе.

В React 18 с createRoot updates автоматически batch-ятся независимо от того, откуда они пришли: из React event, timeout, Promise, native event handler или другого события.

Пример:


function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
}

И пример с timeout:


setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
}, 1000);

В React 18 с createRoot оба варианта могут быть обработаны с batching: React сделает один render в конце batch, а не отдельный render для каждого setState.

Для современной Next.js/React-модели это важная базовая предпосылка: batching уже не стоит понимать только как “внутри onClick”. Но статья всё равно начинается с пользовательского действия, потому что это самый понятный вход в flow.

17. `flushSync` как исключение

Иногда нужно принудительно заставить React синхронно применить updates и обновить DOM раньше обычного.

Для этого есть flushSync из react-dom:


import { flushSync } from 'react-dom';

function handleClick() {
  flushSync(() => {
    setCount(c => c + 1);
  });

  // К этому месту React уже попытался обновить DOM для этого update.
}

Это исключение, а не обычный путь. React docs предупреждает, что flushSync используется редко и может ухудшить производительность. В обычном коде лучше позволять React batch-ить updates и планировать работу самому.

Для текущей ветки это важно как контраст:


обычный setState
→ update queue
→ batching
→ scheduling позже

flushSync
→ принудительно сбросить update синхронно
→ использовать редко

18. State как immutable value

Если state содержит объект или массив, его нельзя обновлять мутацией старого значения.

Неправильно:


function MovingDot() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  function handlePointerMove(event: React.PointerEvent<HTMLDivElement>) {
    position.x = event.clientX;
    position.y = event.clientY;
  }

  return <div onPointerMove={handlePointerMove} />;
}

Этот код мутирует объект из прошлого render snapshot, но не вызывает setter. React не узнаёт, что нужно подготовить обновление.

Правильно:


function MovingDot() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  function handlePointerMove(event: React.PointerEvent<HTMLDivElement>) {
    setPosition({
      x: event.clientX,
      y: event.clientY,
    });
  }

  return <div onPointerMove={handlePointerMove} />;
}

Если нужно сохранить часть старого объекта:


setPerson({
  ...person,
  firstName: event.target.value,
});

Связь с update queue такая:


мутация старого объекта
→ React не получает update

setState(newObject)
→ React получает update
→ update попадает в очередь

Кроме того, если новое значение state идентично текущему по Object.is, React может пропустить re-render как оптимизацию. Поэтому для объектов и массивов важно передавать новое значение, а не тот же объект после мутации.

19. Что batching не делает

Batching часто понимают слишком широко. Нужно зафиксировать границы.


Batching ≠ немедленное изменение state-переменной в текущем handler-е

State внутри handler-а остаётся snapshot из текущего render-а.


Batching ≠ немедленное изменение DOM

DOM будет обновлён только позже, если React дойдёт до commit phase.


Batching ≠ объединение всех событий навсегда

React не batch-ит разные intentional user events как один бесконечный batch.


Batching ≠ пропуск render всегда

Batching уменьшает количество render-проходов, но не означает, что render не будет. Он означает, что несколько updates могут привести к одному render вместо нескольких.


Batching ≠ lanes

Batching группирует updates. Lanes и priority относятся к следующей статье: как React планирует работу.

20. Полный проход по диапазону

Пользователь кликает по кнопке. Браузер создаёт DOM event. React event system получает это событие и вызывает обработчик, который был передан в JSX как onClick.

Обработчик выполняется как обычная JavaScript-функция. Он может читать props и state из snapshot текущего render-а, может вызвать event.preventDefault() или event.stopPropagation(), может выполнить прикладную логику и может вызвать setState или dispatch.

Вызов setState не меняет текущую state-переменную внутри handler-а. Он ставит update в очередь React. Если передано значение, React воспринимает это как замену state на это значение. Если передана updater function, React позже применит её к pending state.

Если в рамках одного события вызывается несколько updates, React группирует их. Он ждёт завершения handler-а и затем обрабатывает updates вместе, чтобы избежать лишних render-проходов и промежуточных “полуготовых” состояний UI.

После batching эта статья заканчивается. Дальше React должен запланировать работу: определить приоритет, lane и перейти к render/reconciliation. Это уже следующая часть flow.

21. Что важно не перепутать


Browser DOM event ≠ React event object

Браузер создаёт native event. React передаёт handler-у React event object, который похож на DOM event, но является частью React event system.


Event handler ≠ render function

Render должен быть чистым. Event handler выполняется в ответ на действие пользователя и может иметь side effects.


onClick={handleClick} ≠ onClick={handleClick()}

В первом случае функция передана как handler. Во втором она вызвана во время render.


setState ≠ немедленно изменить переменную state

Setter запрашивает следующий render. Текущий handler продолжает видеть state snapshot из текущего render-а.


setState ≠ DOM mutation

Setter ставит update в очередь. DOM меняется только позже, на commit phase, если изменения действительно есть.


setCount(count + 1) × 3 ≠ +3

Если count в текущем snapshot равен 0, три таких вызова ставят три замены на 1.


setCount(c => c + 1) × 3 = +3

Updater functions применяются последовательно к pending state во время обработки очереди.


stopPropagation ≠ preventDefault

stopPropagation останавливает распространение события по React tree. preventDefault отменяет стандартное браузерное действие.


Batching ≠ scheduling

Batching группирует updates. Scheduling решает, как и когда выполнять React work. Это следующий этап.
