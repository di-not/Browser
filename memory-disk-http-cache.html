Ответ из Memory/Disk HTTP Cache

Коротко о блоке

Блок «Ответ из Memory/Disk HTTP Cache» описывает ситуацию, когда браузер может взять HTTP-ответ из собственного кэша, а не загружать ресурс заново из сети.

В схеме это выглядит так:

Можно использовать локальный ответ?
→ HTTP cache hit
→ Ответ из Memory/Disk HTTP Cache
→ HTTP Response

Если браузер уже загружал ресурс раньше и сохранённый ответ всё ещё можно использовать, он может вернуть его из HTTP Cache. В этом случае DNS, новое соединение и повторная передача ресурса могут не понадобиться.

1. Что такое HTTP Cache

HTTP Cache — это механизм браузера, который сохраняет HTTP responses и переиспользует их для следующих HTTP requests.

Пример:

Первый заход:
GET /styles.css
→ сервер вернул styles.css
→ браузер сохранил response в HTTP Cache

Повторный заход:
GET /styles.css
→ браузер проверил HTTP Cache
→ нашёл подходящий response
→ использовал его

Кэшироваться могут:

HTML
CSS
JavaScript
images
fonts
JSON
другие HTTP-ресурсы

Но будет ли ресурс сохранён и можно ли его использовать повторно, зависит от HTTP-заголовков.

2. Memory Cache и Disk Cache

В DevTools часто можно увидеть:

from memory cache
from disk cache

Это не два разных стандарта HTTP caching, а разные места, где браузер может физически хранить кэшированный response.

Memory Cache

Memory Cache — кэш в оперативной памяти.

Особенности:

  • очень быстрый;
  • обычно живёт недолго;
  • может очищаться при закрытии вкладки, навигации или нехватке памяти;
  • часто используется для ресурсов, которые недавно понадобились странице.

Пример:

страница загрузила /logo.png
потом этот же /logo.png понадобился снова
браузер может взять его из memory cache

Disk Cache

Disk Cache — кэш на диске.

Особенности:

  • медленнее memory cache, но обычно всё равно быстрее сети;
  • может переживать закрытие вкладки или браузера;
  • подходит для более долгого хранения ресурсов;
  • ограничен размером и политиками браузера.

Пример:

пользователь уже заходил на сайт вчера
браузер сохранил /app.js на диск
при новом заходе /app.js может быть взят из disk cache

Важное различие:

HTTP Cache → логический механизм кэширования HTTP responses
Memory/Disk Cache → возможные физические места хранения внутри браузера

То есть Memory Cache и Disk Cache — это детали реализации браузера. Основные правила всё равно задаются HTTP caching semantics и HTTP-заголовками.

3. HTTP cache hit и cache miss

HTTP cache hit — ситуация, когда браузер нашёл в HTTP Cache подходящий response для текущего request. 

Request
→ HTTP Cache
→ подходящий response найден
→ cache hit

Если response можно использовать сразу:

Request
→ HTTP Cache
→ fresh response
→ вернуть response без сети

Cache miss — ситуация, когда кэш не может отдать ответ.

Причины:

  • ресурса нет в кэше;
  • URL отличается;
  • метод запроса не подходит;
  • response нельзя кэшировать;
  • response устарел и требует проверки;
  • Vary говорит, что сохранённый response не подходит под текущие request headers.

4. Fresh и stale

Кэшированный response может быть fresh или stale.

Fresh response

Fresh response — свежий кэшированный ответ, который можно использовать без обращения к серверу.

Пример заголовка:

Cache-Control: max-age=3600

Это означает:

response можно считать свежим 3600 секунд

Если браузер запрашивает этот ресурс в течение 3600 секунд, он может использовать кэш без сети: 

Request
→ HTTP Cache
→ response fresh
→ вернуть response

Stale response

Stale response — устаревший кэшированный ответ.

Это не значит, что его нужно сразу удалить. Это значит, что перед использованием его обычно нужно проверить у сервера. 

Request
→ HTTP Cache
→ response stale
→ нужна revalidation

5. Revalidation

Revalidation — проверка у сервера, актуальна ли кэшированная копия.

Если response устарел, браузер может отправить условный запрос:

У меня есть версия ресурса.
Она всё ещё актуальна?

Если сервер отвечает:

304 Not Modified

это значит:

ресурс не изменился
можно использовать кэшированную копию

В этом случае body ресурса не скачивается заново.

Revalidation через ETag

Первый ответ сервера:

HTTP/1.1 200 OK
ETag: "abc123"
Cache-Control: no-cache

Браузер может сохранить response.

При повторном запросе он отправит:

If-None-Match: "abc123"

Если ресурс не изменился, сервер ответит:

HTTP/1.1 304 Not Modified

Браузер использует старое body из кэша.

Revalidation через Last-Modified

Первый ответ сервера:

Last-Modified: Tue, 15 Nov 2026 12:45:26 GMT

Повторный запрос может содержать:

If-Modified-Since: Tue, 15 Nov 2026 12:45:26 GMT

Если ресурс не изменился:

304 Not Modified

6. Главные HTTP-заголовки кэширования

Cache-Control

Cache-Control — главный HTTP-заголовок для управления кэшированием. 

Cache-Control: max-age=3600

Ответ можно использовать из кэша 3600 секунд.

Cache-Control: no-store

Ответ не должен сохраняться в кэше.

Cache-Control: no-cache

Название сбивает с толку. no-cache обычно не означает «не сохранять». Оно означает: 

можно сохранить, но перед повторным использованием
нужно проверить у сервера
Cache-Control: public, max-age=31536000, immutable

Подходит для версионированных статических файлов:

/app.8f3a2.js
/styles.91ab.css

Такой файл можно долго хранить в кэше, потому что при изменении содержимого меняется имя файла.

ETag

ETag — Entity Tag, идентификатор версии ресурса.

ETag: "abc123"

Если браузер позже хочет проверить ресурс, он отправляет:

If-None-Match: "abc123"

Last-Modified

Last-Modified — заголовок с датой последнего изменения ресурса. 

Last-Modified: Tue, 15 Nov 2026 12:45:26 GMT

Expires

Expires — старый способ указать дату, до которой response считается свежим. 

Expires: Wed, 21 Oct 2026 07:28:00 GMT

В современных приложениях чаще используют Cache-Control, потому что он гибче.

Vary

Vary говорит кэшу, какие request headers влияют на выбор response. 

Vary: Accept-Encoding

Это значит, что ответы для gzip, br и других вариантов сжатия могут отличаться. 

Vary: Accept-Language

Это значит, что ответ для ru и ответ для en могут быть разными.

7. Как браузер решает, можно ли использовать HTTP Cache

Упрощённая логика:

Request
→ есть ли response в HTTP Cache?
   ├─ нет
   │  → cache miss
   │  → идти в сеть
   │
   └─ да
      → подходит ли он под текущий request?
         ├─ нет
         │  → идти в сеть
         │
         └─ да
            → fresh или stale?
               ├─ fresh
               │  → вернуть response из cache
               │
               └─ stale
                  → revalidation
                     ├─ 304 Not Modified
                     │  → использовать cached response
                     └─ 200 OK
                        → сохранить новый response

8. Что значит «Ответ из Memory/Disk HTTP Cache»

Когда в DevTools видно:

from memory cache

или:

from disk cache

это обычно означает:

браузер не скачивал resource body заново из сети,
а использовал сохранённый response

Есть два важных сценария.

Fresh cache hit

Request
→ HTTP Cache
→ fresh response
→ вернуть response

В этом случае может не быть сетевого запроса вообще.

Revalidation

Request
→ HTTP Cache
→ stale response
→ conditional request
→ 304 Not Modified
→ использовать cached body

В этом случае сеть была использована, но полный ресурс не скачивался заново.

9. HTTP Cache и Service Worker Cache — не одно и то же

Важно отличать этот блок от Service Worker / Cache Storage. 

HTTP Cache
→ управляется браузером
→ работает по HTTP headers
→ связан с HTTP caching rules

Cache Storage
→ управляется JavaScript-кодом
→ часто используется Service Worker
→ разработчик сам пишет стратегию

Пример:

HTTP Cache:
сервер отправил Cache-Control: max-age=3600
браузер сам решает, что response можно использовать час

Cache Storage:
Service Worker сам положил Response в caches.open("app-v1")
и сам решил, когда его вернуть

10. Как этот блок связан с общей схемой

После разбора URL браузер проверяет:

Можно использовать локальный ответ?

Если есть подходящий HTTP cache hit:

Можно использовать локальный ответ?
→ HTTP cache hit
→ Ответ из Memory/Disk HTTP Cache
→ HTTP Response

Тогда браузер может не идти дальше в сетевую ветку:

DNS
connection
HTTP Request

Но если кэш не подходит:

HTTP cache miss
или stale response требует проверки

браузер продолжает сетевой путь.

11. Мини-глоссарий

HTTP Cache

Браузерный механизм хранения и повторного использования HTTP responses.

Memory Cache

Кэш в оперативной памяти браузера.

Disk Cache

Кэш на диске.

Cache hit

Ситуация, когда подходящий response найден в кэше.

Cache miss

Ситуация, когда подходящего response в кэше нет.

Fresh response

Кэшированный response, который ещё можно использовать без проверки у сервера.

Stale response

Кэшированный response, срок свежести которого истёк.

Revalidation

Проверка у сервера, актуальна ли кэшированная копия.

Cache-Control

HTTP-заголовок, задающий правила кэширования.

max-age

Директива Cache-Control, задающая время свежести response в секундах.

no-store

Директива, запрещающая сохранять response в кэше.

no-cache

Директива, требующая проверки response у сервера перед повторным использованием.

ETag

Entity Tag, идентификатор версии ресурса.

If-None-Match

Request header, через который браузер отправляет ETag для проверки актуальности.

Last-Modified

Response header с датой последнего изменения ресурса.

If-Modified-Since

Request header для проверки ресурса по дате последнего изменения.

304 Not Modified

HTTP status, означающий, что ресурс не изменился и можно использовать кэшированную копию.

Expires

Response header со временем, до которого response считается свежим.

Vary

Response header, указывающий, какие request headers влияют на выбор cached response.
