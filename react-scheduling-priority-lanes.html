React client update: Scheduling, priority и lanes

Коротко

Эта статья разбирает участок после очереди обновлений и batching, но до render/reconciliation:


Queue update
→ Batching
→ Schedule update / priority / lane
→ Render phase / Reconciliation

Главная идея:

После того как React получил updates и сгруппировал их, он должен решить, как срочно выполнять работу. Публичная модель React говорит о срочных обновлениях, несрочных Transition-обновлениях и возможности откладывать неважную часть UI. Внутри React эта система связана с приоритетами и lanes, но lanes не являются публичным API приложения.

Публичные инструменты, которые относятся к этой теме:


startTransition
useTransition
useDeferredValue
flushSync

startTransition, useTransition и useDeferredValue помогают React оставлять интерфейс отзывчивым, когда часть обновления может быть несрочной. flushSync делает противоположное: принудительно заставляет React синхронно сбросить update и обновить DOM, поэтому это редкий инструмент.

1. Граница статьи

Эта статья начинается здесь:


Queue update
→ Batching

и заканчивается здесь:


Schedule update / priority / lane
→ Render phase / Reconciliation

В предыдущей статье React уже получил updates:


function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
  setInput(event.target.value);
  setQuery(event.target.value);
}

React мог сгруппировать эти updates в batch. Теперь ему нужно решить, как выполнять работу дальше.

В этой статье не раскрываются подробно:


Fiber render phase
Reconciliation
Bailout
Commit phase
DOM mutations
useLayoutEffect
useEffect
Browser render pipeline

Они идут после scheduling.

2. Что значит “schedule update”

Schedule update означает, что React получил update и должен поставить работу на выполнение с подходящей срочностью.

React не обязан воспринимать все обновления одинаково. Некоторые изменения должны ощущаться мгновенными: ввод в поле, клик по кнопке, переключение checkbox. Другие изменения могут быть менее срочными: фильтрация большого списка, переход между вкладками с тяжёлым содержимым, обновление результатов поиска.

Смысловая модель:


update попал в очередь
→ updates сгруппированы batching
→ React определяет срочность работы
→ React выбирает, когда и какую работу выполнять
→ начинается render phase

Scheduling не равен batching.


Batching
→ какие updates обработать вместе

Scheduling
→ когда и с каким приоритетом выполнять React work

3. Почему React нужен scheduler

Если каждое обновление всегда выполнялось бы синхронно и до конца, тяжёлый render мог бы заблокировать главный поток браузера. Пока JavaScript занят, браузер хуже реагирует на ввод, клики и другие действия.

React 18 описывает concurrent rendering как механизм, который позволяет React готовить несколько версий UI одновременно, прерывать rendering, продолжать его позже или даже отказаться от начатой работы. Главное свойство concurrent rendering — interruptible rendering: React может начать render, поставить его на паузу, продолжить позже или выбросить незавершённую работу.

Для пользователя цель не в том, чтобы “React делал всё медленнее”, а в том, чтобы интерфейс оставался отзывчивым. Например, ввод в поле должен обновляться быстро, даже если связанный список результатов тяжёлый.

4. Срочные и несрочные обновления

Официальная документация React использует понятие Transition для несрочных обновлений.

Срочное обновление — это то, что пользователь ожидает увидеть немедленно как прямой отклик на действие.

Примеры:


набранный символ появляется в input
checkbox меняет checked-состояние
кнопка визуально нажалась
локальное раскрытие меню

Transition-обновление — это обновление, которое можно сделать несрочным, чтобы не блокировать более важные взаимодействия.

Примеры:


переключить вкладку с тяжёлым содержимым
обновить большой список после ввода
показать результаты фильтрации
перейти к новой части UI, если старый UI может оставаться видимым

React docs формулирует смысл Transitions так: они позволяют UI оставаться отзывчивым даже во время re-render. Если пользователь начал один переход, а потом передумал и нажал другую вкладку, он не должен ждать окончания первого тяжёлого render-а.

5. `startTransition`

startTransition — функция React, которая помечает state updates внутри переданного callback как Transition.


import { startTransition } from 'react';

function SearchPage() {
  const [input, setInput] = useState('');
  const [query, setQuery] = useState('');

  function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
    const nextValue = event.target.value;

    setInput(nextValue);

    startTransition(() => {
      setQuery(nextValue);
    });
  }

  return (
    <>
      <input value={input} onChange={handleChange} />
      <SearchResults query={query} />
    </>
  );
}

Здесь есть две разные задачи:


setInput(nextValue)
→ срочное обновление controlled input

setQuery(nextValue)
→ несрочное обновление результатов поиска

Если SearchResults тяжёлый, Transition позволяет React сохранить отзывчивость input. Ввод не должен ждать, пока тяжёлый список закончит re-render.

Что важно знать про `startTransition`

Callback внутри startTransition вызывается сразу. startTransition не откладывает сам callback как setTimeout. Он помечает updates, которые происходят во время его синхронного выполнения, как Transition.


startTransition(() => {
  setTab(nextTab);
});

Эти updates становятся несрочными Transition updates.

startTransition не возвращает pending state. Если нужно показывать, что transition ещё выполняется, обычно используют useTransition.

6. Ограничение: updates после async

В документации React есть важное предостережение: state updates после await нужно снова обернуть в startTransition, чтобы они были помечены как Transition.


startTransition(async () => {
  const data = await fetchData();

  startTransition(() => {
    setData(data);
  });
});

Это выглядит странно, но на момент текущей документации это известное ограничение. Если после async-запроса вызвать setState без повторного startTransition, update может не получить Transition-маркировку.

Практический вывод:


updates прямо внутри синхронного callback startTransition
→ помечаются как Transition

updates после await / setTimeout
→ нужно снова обернуть, если они тоже должны быть Transition

7. `useTransition`

useTransition — Hook, который возвращает два значения:


const [isPending, startTransition] = useTransition();

startTransition помечает updates как Transition, а isPending показывает, что Transition ещё находится в процессе.

Пример:


import { useTransition, useState } from 'react';

function Tabs() {
  const [tab, setTab] = useState('about');
  const [isPending, startTransition] = useTransition();

  function selectTab(nextTab: string) {
    startTransition(() => {
      setTab(nextTab);
    });
  }

  return (
    <>
      <button onClick={() => selectTab('about')}>About</button>
      <button onClick={() => selectTab('posts')}>Posts</button>

      {isPending && <p>Loading...</p>}

      <TabContent tab={tab} />
    </>
  );
}

Смысл:


пользователь нажал вкладку
→ React помечает update tab как Transition
→ старый UI может оставаться отзывчивым
→ isPending позволяет показать состояние ожидания
→ React завершает render новой вкладки

useTransition особенно полезен, когда нужно не только сделать update несрочным, но и показать пользователю, что переход ещё не завершён.

8. Что можно и нельзя оборачивать в Transition

Transition предназначен для state updates, которые можно сделать несрочными.

Хорошие кандидаты:


переключение тяжёлой вкладки
обновление большого списка
переключение route-like view
обновление результатов поиска

Плохие кандидаты:


value controlled input
checked checkbox
текст, который должен появиться сразу при вводе
любое обновление, где задержка ломает ощущение прямого отклика

Например, это плохая идея:


function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
  startTransition(() => {
    setInput(event.target.value);
  });
}

Controlled input должен обновляться синхронно с вводом. Если сделать его Transition, ввод может лагать или вести себя не так, как ожидает пользователь.

Правильная модель:


function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
  const nextValue = event.target.value;

  setInput(nextValue); // срочно

  startTransition(() => {
    setQuery(nextValue); // несрочно
  });
}

9. Transitions являются interruptible

Transition updates могут быть прерваны более срочными updates.

Пример:


пользователь ввёл "r"
→ запущен Transition для списка результатов

пользователь сразу ввёл "re"
→ срочное обновление input должно пройти быстро
→ старый Transition можно прервать или начать заново

React docs описывает это как возможность сохранять UI отзывчивым во время re-render. Если пользователь меняет решение, он не обязан ждать завершения предыдущего несрочного render-а.

Важно не делать из этого слишком внутренний вывод. Публичная гарантия для разработчика такая:


Transition помечает обновления как неблокирующие.
Более срочные обновления могут прерывать такую работу.

10. `useDeferredValue`

useDeferredValue — Hook, который позволяет отложить обновление неважной части UI.


const deferredValue = useDeferredValue(value);

Типичный пример — поиск.


import { useDeferredValue, useState } from 'react';

function SearchPage() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);

  return (
    <>
      <input
        value={query}
        onChange={event => setQuery(event.target.value)}
      />

      <SearchResults query={deferredQuery} />
    </>
  );
}

Здесь input использует обычный query и остаётся отзывчивым. Тяжёлый SearchResults получает deferredQuery, который может отставать от актуального значения.

Смысл:


query
→ срочно обновляет input

deferredQuery
→ может отставать
→ даёт React возможность отложить тяжёлую часть UI

11. Как работает `useDeferredValue` на уровне поведения

Когда значение, переданное в useDeferredValue, меняется, React делает текущий render с прежним deferred value, а затем планирует background re-render с новым deferred value.

Пример:


query = "react"
deferredQuery = "rea"

input уже показывает "react"
results ещё показывают "rea"

React планирует background render
→ когда он завершится, results получат "react"

Документация React подчёркивает, что background re-render прерываем. Если значение снова изменится, React может начать background re-render заново.

Это похоже на Transition по цели: сохранить отзывчивость важной части UI. Но API другой.


startTransition / useTransition
→ разработчик помечает конкретные state updates как Transition

useDeferredValue
→ разработчик откладывает использование конкретного value в части UI

12. `useDeferredValue` и stale UI

Deferred value может быть старым. Это не баг, а смысл Hook-а.

Пример:


function SearchPage() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);

  const isStale = query !== deferredQuery;

  return (
    <>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
      />

      <div style={{ opacity: isStale ? 0.5 : 1 }}>
        <SearchResults query={deferredQuery} />
      </div>
    </>
  );
}

Если query !== deferredQuery, можно визуально показать, что результаты ещё догоняют ввод.

Смысл:


input всегда актуален
results могут отставать
UI показывает stale-состояние

13. Ограничения `useDeferredValue`

React docs указывает несколько предостережений.

Если update уже находится внутри Transition, useDeferredValue возвращает новое значение и не создаёт отдельный deferred render, потому что update уже отложен.

Ещё одно важное ограничение: значение, которое передаётся в useDeferredValue, должно быть примитивом или объектом, созданным вне render. Если создавать новый объект прямо во время render и передавать его в useDeferredValue, он будет отличаться на каждом render-е, что может вызвать лишние background re-renders.

Плохо:


const deferredOptions = useDeferredValue({ query, limit: 20 });

Лучше:


const options = useMemo(
  () => ({ query, limit: 20 }),
  [query]
);

const deferredOptions = useDeferredValue(options);

Или проще передавать primitive value:


const deferredQuery = useDeferredValue(query);

14. `startTransition` vs `useTransition` vs `useDeferredValue`

`startTransition`

Используется, когда нужно пометить updates как Transition, но pending state не нужен или Hook недоступен.


startTransition(() => {
  setPage(nextPage);
});

`useTransition`

Используется внутри компонента, когда нужно и пометить updates как Transition, и получить isPending.


const [isPending, startTransition] = useTransition();

startTransition(() => {
  setPage(nextPage);
});

`useDeferredValue`

Используется, когда state update остаётся обычным, но конкретная часть UI должна использовать отложенную версию значения.


const deferredQuery = useDeferredValue(query);

Практическое различие:


Transition
→ “это обновление можно считать несрочным”

Deferred value
→ “эта часть UI может использовать старое значение, пока новое готовится”

15. `flushSync`

flushSync из react-dom делает противоположное Transition-подходу. Он заставляет React синхронно сбросить updates внутри callback и обновить DOM немедленно.


import { flushSync } from 'react-dom';

function handleClick() {
  flushSync(() => {
    setCount(c => c + 1);
  });

  // к этому месту React уже попытался обновить DOM
}

React docs прямо предупреждает: flushSync используется редко и может вредить производительности.

flushSync нужен для интеграции со сторонними API или браузерными API, которым нужен уже обновлённый DOM к определённому моменту.

Примерная ситуация:


нужно обновить state
→ React должен применить DOM changes прямо сейчас
→ после этого сторонний код читает DOM

В обычном React-коде лучше не использовать flushSync.

16. `flushSync` и batching

Обычный путь:


setState
→ queue update
→ batching
→ scheduling
→ render
→ commit

flushSync меняет этот путь:


flushSync callback
→ React синхронно сбрасывает updates внутри callback
→ DOM обновляется до выхода из flushSync

Это может сломать ожидание, что React сам выберет удобный момент. Поэтому flushSync может ухудшить производительность, может заставить React показать fallback для Suspense и может выполнить pending Effects до возврата из flushSync.

Правило:


flushSync
→ emergency tool

Transition / deferred value
→ способ дать React больше свободы планирования

17. Lanes: что это и как о них писать

Lane — внутренний термин React. Он относится к реализации приоритетов и групп работы внутри React reconciler. В исходном коде React есть файл ReactFiberLane.js, где определяются lane-типы и операции с ними.

Но lanes не являются публичным API приложения. Разработчик не пишет:


setState(value, lane);

и не должен строить бизнес-логику вокруг конкретных lane-значений.

React назначает обновлениям приоритет.

Во внутренней реализации React эта модель связана с lanes.

Публично разработчик работает не с lanes, а с APIs вроде startTransition, useTransition, useDeferredValue и flushSync.

18. Lanes и Transition

Несрочные Transition updates внутри React получают более низкую срочность, чем срочные пользовательские updates. На публичном уровне это выражается так:


Transition update
→ non-blocking
→ может быть прерван более срочным update
→ UI остаётся отзывчивым

На внутреннем уровне React может использовать lanes, чтобы хранить и обрабатывать разные категории работы.

Для практической статьи достаточно такой связи:


startTransition / useTransition
→ публично помечают updates как Transition

lanes
→ внутренняя модель, через которую React различает и обрабатывает приоритеты работы

19. Как scheduling подводит к render phase

Когда React решил, какую работу выполнять, начинается render phase.

Смысловой переход:


updates queued
→ updates batched
→ React assigned priority / scheduling decision
→ React starts render work
→ components are called
→ reconciliation compares result

Scheduling не вычисляет UI. Он только организует работу.


Scheduling
→ когда и с каким приоритетом выполнять

Render phase
→ вычислить следующий UI

Commit phase
→ применить изменения к DOM

20. Пример: input + тяжёлый список без Transition


function SearchPage() {
  const [query, setQuery] = useState('');

  function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
    setQuery(event.target.value);
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      <SlowList query={query} />
    </>
  );
}

Здесь одно значение query одновременно управляет input и тяжёлым списком.

При каждом вводе:


пользователь печатает
→ setQuery
→ React должен обновить input
→ React должен обновить SlowList

Если SlowList тяжёлый, ввод может ощущаться медленным.

21. Пример: input + Transition


function SearchPage() {
  const [input, setInput] = useState('');
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();

  function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
    const nextValue = event.target.value;

    setInput(nextValue);

    startTransition(() => {
      setQuery(nextValue);
    });
  }

  return (
    <>
      <input value={input} onChange={handleChange} />

      {isPending && <p>Updating results...</p>}

      <SlowList query={query} />
    </>
  );
}

Теперь input обновляется срочно, а список — как Transition.

Смысл:


input state
→ срочный отклик на ввод

query state
→ несрочное обновление результата

Это лучше отражает пользовательский приоритет: введённый символ должен появиться сразу, а результаты могут догнать позже.

22. Пример: input + `useDeferredValue`


function SearchPage() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  return (
    <>
      <input
        value={query}
        onChange={event => setQuery(event.target.value)}
      />

      <div style={{ opacity: isStale ? 0.5 : 1 }}>
        <SlowList query={deferredQuery} />
      </div>
    </>
  );
}

Здесь state один: query. Input всегда получает актуальное значение. SlowList получает отложенное значение.

Смысл:


query
→ актуальное значение для input

deferredQuery
→ отложенное значение для тяжёлой части UI

Этот подход удобен, когда не хочется разделять state на input и query, но нужно дать тяжёлой части UI возможность отставать.

23. Чем Transition отличается от debounce

Transition не является debounce.

Debounce обычно задерживает сам update или side effect на заданное время:


пользователь печатает
→ ждать 300ms
→ выполнить update / запрос

Transition не говорит “жди 300ms”. Он говорит React:


это обновление несрочное
его можно прервать более срочными обновлениями

useDeferredValue тоже не является debounce. Он не ждёт фиксированное время. Он позволяет React отложить re-render части UI и выполнить его, когда получится.

Если нужно ограничить сетевые запросы, debounce всё ещё может быть нужен. Если нужно сохранить отзывчивость UI при тяжёлом render-е, Transition или deferred value подходят лучше.

24. Suspense и Transitions

Transitions часто используются вместе с Suspense.

Если update внутри Transition приводит к тому, что часть UI suspends, React может не скрывать уже показанный UI мгновенно ради fallback, а сохранить более плавное поведение. Официальная документация Suspense показывает, что для переходов можно использовать startTransition, чтобы избежать замены уже видимого содержимого на fallback в некоторых сценариях.

Смысл:


срочное обновление, которое suspends
→ пользователь может резко увидеть fallback

Transition update, который suspends
→ React может сохранить уже видимый UI, пока готовится новый

Детали зависят от структуры Suspense boundaries и фреймворка. В статье про scheduling достаточно понять связь:


Transitions
→ помогают маркировать навигационные или тяжёлые обновления как несрочные

Suspense
→ может показать fallback, если содержимое ещё не готово

Transition + Suspense
→ помогают сделать ожидание менее резким для пользователя

25. Что важно не перепутать


Batching ≠ Scheduling

Batching группирует updates. Scheduling решает, как и когда выполнять работу.


Scheduling ≠ Render phase

Scheduling организует работу. Render phase вычисляет следующий UI.


Transition ≠ setTimeout

startTransition не откладывает callback по таймеру. Callback выполняется сразу, а updates внутри него помечаются как Transition.


Transition ≠ debounce

Transition не ждёт фиксированное время. Он меняет приоритет React work.


useTransition ≠ useDeferredValue

useTransition помечает updates. useDeferredValue возвращает отложенную версию value.


isPending ≠ loading data обязательно

isPending означает, что Transition ещё pending. Данные могут грузиться, но сам флаг относится к React Transition.


useDeferredValue может показывать stale UI

Это нормальное поведение. Старое значение может оставаться видимым, пока React готовит новое.


lanes ≠ публичный API

Lanes — внутренняя модель React. В приложении используются публичные API: startTransition, useTransition, useDeferredValue, flushSync.


flushSync ≠ обычный способ ускорить React

flushSync принудительно синхронен, редок и может ухудшить производительность.

26. Полный проход по диапазону

После пользовательского действия React получил один или несколько updates. Batching позволил обработать их вместе, если это безопасно. Теперь React должен решить, насколько срочно выполнять работу.

Срочные updates, связанные с непосредственным вводом пользователя, должны обновлять UI быстро. Несрочные updates можно пометить как Transition через startTransition или useTransition. Если нужно отложить не сам update, а использование значения в тяжёлой части UI, можно использовать useDeferredValue.

Внутри React эта модель связана с приоритетами и lanes. Но разработчик не управляет lanes напрямую. Он выражает намерение через публичные API: какие updates срочные, какие можно сделать Transition, какую часть UI можно отложить, и когда в редких случаях нужно принудительно синхронно обновить DOM через flushSync.

После scheduling React переходит к render phase / reconciliation: вызывает компоненты, вычисляет следующий UI и решает, какие изменения нужны. Это следующая статья.
