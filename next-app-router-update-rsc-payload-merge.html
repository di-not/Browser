Next.js App Router update: navigation, refresh, Server Action и RSC payload merge

1. Место этого блока в общей схеме

Эта статья разбирает участок Next.js App Router update, который начинается с выбора типа обновления и заканчивается слиянием нового RSC Payload с текущим клиентским деревом.

В схеме этот участок выглядит так:

Тип Next App Router update?
├─ Router navigation
├─ Refetch / revalidate
├─ Server Action
└─ RSC update
      ↓
Получить новый RSC / Flight Payload
      ↓
RSC payload merge
      ↓
React Fiber

Ветка Client state update → React Synthetic Event сюда не входит. Если пользователь нажал кнопку внутри Client Component, обработчик вызвал setState, и дальше всё пошло через обычный React-путь, это уже не App Router update в смысле этой статьи. Такой путь разбирается отдельно: React client update: user action → event handler → queue update → batching, Scheduling, priority и lanes, React Fiber: фаза render, reconciliation и bailout.

Здесь рассматривается другой тип обновления: когда источником следующего UI становится не локальный setState, а новый серверный результат App Router — RSC Payload.

2. Почему эта ветка отличается от обычного React update

В обычном клиентском React-обновлении всё начинается внутри браузера. Пользователь совершает действие, React вызывает обработчик события, обработчик ставит state update, React планирует работу, выполняет render/reconciliation и commit.

В App Router часть обновлений устроена иначе. Навигация, router.refresh(), revalidation или Server Action могут потребовать нового результата от сервера. В этом случае клиентская среда Next.js обращается к серверу или использует уже подготовленные данные из Router Cache, получает RSC Payload и передаёт этот результат в React.

Ключевая разница:

Обычный React update:
локальный state update → React Fiber → render/reconciliation → commit

Next App Router update:
серверный RSC Payload → merge с текущим деревом → React Fiber → render/reconciliation → commit

RSC payload merge не означает, что DOM уже изменился. Это означает, что Next.js и React получили новый серверный материал для дерева компонентов. Реальные DOM-изменения произойдут позже, если после Fiber/render/reconciliation/commit окажется, что DOM нужно менять.

Подробности commit-пути после Fiber разбираются отдельно: React update: commit phase, DOM mutations, useLayoutEffect, browser render и useEffect.

3. Что такое RSC Payload

RSC Payload — это React Server Component Payload. В документации Next.js он описывается как компактное бинарное представление отрендеренного дерева React Server Components.

В RSC Payload могут быть:

  • результат рендера Server Components;
  • места, где должны быть Client Components;
  • ссылки на JavaScript-файлы Client Components;
  • props, переданные из Server Components в Client Components.

Это не HTML-документ и не обычный JSON API для приложения. Это формат React Server Components, который нужен React на клиенте, чтобы согласовать Server Components и Client Components в одно дерево.

На initial load App Router может прислать HTML, чтобы быстро показать предварительный UI, и RSC Payload, чтобы React мог согласовать серверное и клиентское дерево. Эта часть уже относится к первичной загрузке Next.js и разобрана отдельно: Next.js App Router: потоковый HTML, RSC payload, ссылки на клиентские компоненты и первые JavaScript-фрагменты и Next.js App Router: гидратация Client Components.

В этой статье речь не о первом открытии страницы, а о последующих обновлениях: пользователь уже находится внутри App Router, приложение уже интерактивно, и какой-то App Router-механизм приводит к новому RSC Payload.

4. Server Components и Client Components в этом блоке

В App Router layout и page по умолчанию являются Server Components. Они могут получать данные на сервере, использовать секреты, работать ближе к базе данных или API и не увеличивать клиентский JavaScript bundle.

Client Components нужны там, где требуется интерактивность: state, обработчики событий, Effects, browser APIs или custom Hooks. Они обозначаются директивой 'use client'.

Эта граница важна для update-ветки. Когда App Router получает новый RSC Payload, он получает новый серверный результат для Server Components и информацию о местах Client Components. Но Client Components остаются клиентскими интерактивными частями. Их JavaScript должен быть доступен в браузере, а их локальное состояние может сохраняться или сбрасываться по обычным правилам React: тип, позиция, key, границы дерева.

Поэтому RSC Payload не “заменяет React”. Он приносит серверную часть результата, а React на клиенте всё равно должен совместить её с уже существующим клиентским деревом.

5. Router navigation

Router navigation — это переход между route внутри App Router без полной перезагрузки документа.

Обычно для переходов используется <Link>:

import Link from 'next/link';

export function Header() {
  return <Link href="/dashboard">Dashboard</Link>;
}

Для программной навигации внутри Client Component используется useRouter из next/navigation:

'use client';

import { useRouter } from 'next/navigation';

export function DashboardButton() {
  const router = useRouter();

  return (
    <button onClick={() => router.push('/dashboard')}>
      Dashboard
    </button>
  );
}

router.push() выполняет client-side navigation и добавляет запись в history stack. router.replace() тоже выполняет client-side navigation, но без добавления новой записи в историю.

В App Router routes рендерятся на сервере по умолчанию. Это означает, что для новой страницы клиенту часто нужен серверный результат. Но при client-side navigation браузер не обязан заново загружать полный HTML document. Вместо этого App Router получает или использует RSC Payload для нужного route.

Если payload для route уже есть в Router Cache, переход может использовать его без нового запроса к серверу. Если нужных route segments в cache нет, Next.js получает RSC Payload с сервера и заполняет Router Cache.

6. Prefetch и Router Cache

Next.js использует prefetching, чтобы сделать навигацию быстрее. <Link> может заранее подготовить route, когда ссылка появляется в viewport. Для ручного prefetch можно вызвать router.prefetch().

'use client';

import { useRouter } from 'next/navigation';

export function PricingPreview() {
  const router = useRouter();

  return (
    <div onMouseEnter={() => router.prefetch('/pricing')}>
      Pricing
    </div>
  );
}

Результат prefetch связан не просто с “HTML будущей страницы”. В App Router важен RSC Payload. Next.js хранит RSC Payload в клиентском Router Cache — in-memory cache, разделённом по route segments. Этот cache используется для уже посещённых route и для prefetched route.

Это объясняет, почему navigation-ветка в схеме сразу ведёт к Получить новый RSC / Flight Payload: “получить” здесь не всегда означает “сделать новый network request”. Иногда payload уже находится в Router Cache. Но для дальнейшей модели это неважно: App Router всё равно получает RSC Payload как вход для обновления дерева.

7. router.refresh()

router.refresh() обновляет текущий route. Он делает новый запрос к серверу, заново получает данные, заново рендерит Server Components и возвращает обновлённый RSC Payload.

'use client';

import { useRouter } from 'next/navigation';

export function RefreshButton() {
  const router = useRouter();

  return (
    <button onClick={() => router.refresh()}>
      Refresh
    </button>
  );
}

Важная особенность: клиент сливает обновлённый RSC Payload без потери незатронутого client-side React state, например useState, и browser state, например scroll position.

Но router.refresh() не означает автоматическую инвалидацию server-side cache. Он очищает Client Cache для текущего route, но если server-side данные остаются cached, результат может оказаться тем же. Для server-side cached data используются revalidatePath() или revalidateTag().

Такой путь в схеме соответствует ветке Refetch / revalidate → Получить новый RSC / Flight Payload.

8. revalidatePath и revalidateTag

revalidatePath() и revalidateTag() относятся к server-side revalidation. Они не вызываются из Client Components напрямую.

revalidatePath() инвалидирует cached data для конкретного path. В Server Functions он может обновить UI сразу, если пользователь сейчас находится на затронутом path. В Route Handlers он помечает path для revalidation, и фактическая revalidation произойдёт при следующем посещении этого path.

Пример Server Action:

'use server';

import { revalidatePath } from 'next/cache';

export async function createPost(formData: FormData) {
  await db.post.create({
    data: {
      title: String(formData.get('title')),
    },
  });

  revalidatePath('/posts');
}

revalidateTag() работает с cached data, помеченными tag. Это удобно, когда одни и те же данные используются в разных route.

Для схемы важно не перепутать два уровня:

  • router.refresh() — клиент просит обновить текущий route и получить новый RSC Payload;
  • revalidatePath() / revalidateTag() — сервер инвалидирует cached data;
  • после revalidation новый RSC Payload может прийти в ответе Server Action, при refresh или при следующей навигации.

9. Server Action request

Server Action — это React Server Function, вызываемая через механизмы React actions: например, через <form action>, <button formAction> или из client-side transition. В Next.js Server Actions часто используются для мутаций: создать запись, обновить профиль, удалить item, изменить cookies, инвалидировать cache.

Пример формы:

export default function Page() {
  async function createInvoice(formData: FormData) {
    'use server';

    const amount = formData.get('amount');
    // mutate data
    // revalidate cache
  }

  return <form action={createInvoice}>...</form>;
}

С точки зрения клиентской схемы это не обычный setState. Клиент отправляет action request на сервер. Сервер выполняет функцию, может изменить данные и cache, а затем Next.js может вернуть клиенту не только значение action, но и новый RSC Payload.

Важная официальная особенность Next.js: когда Server Action запускает немедленную revalidation, Next.js делает это в одном HTTP request. Он выполняет action, затем server-side re-render текущего route, и response возвращает и результат action, и новый RSC Payload в одном Flight stream.

Это значит: для обновления текущей страницы после action часто не нужен отдельный follow-up fetch. Серверная мутация, invalidation и новое серверное представление UI могут прийти за один roundtrip.

10. Server Action: return value и UI в одном ответе

Server Action может вернуть значение, которое прочитает клиентский код или useActionState. Но в App Router это не единственная часть ответа.

Если action вызывает updateTag, revalidatePath, refresh, меняет cookies или вызывает redirect, Next.js может включить в response заново отрендеренный RSC Payload.

Пример:

'use server';

import { revalidatePath } from 'next/cache';

export async function createPost(formData: FormData) {
  await db.post.create({
    data: {
      title: String(formData.get('title')),
    },
  });

  revalidatePath('/posts');

  return { ok: true };
}

Клиент может получить return value { ok: true }, а вместе с ним — обновлённый RSC Payload для текущего route. После этого App Router сливает payload с текущим деревом, и React продолжает обычный путь обновления.

revalidateTag() со stale-while-revalidate профилем является исключением: он помечает tag для background refresh и не включает немедленный re-render в action response. UI отразит изменения при последующем чтении.

11. Redirect после Server Action

Server Action может вызвать redirect().

'use server';

import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const post = await db.post.create({
    data: {
      title: String(formData.get('title')),
    },
  });

  redirect(`/posts/${post.id}`);
}

В этом случае результатом action становится не просто обновление текущего route. Router должен перейти к другому URL. В твоей схеме это правильно выделено развилкой: Redirect или обновлённый RSC payload?.

Если action вызывает redirect, response навигирует router и stream-ит RSC Payload для destination route. Поэтому дальше логически начинается навигационный путь: новый URL, новый route, новый серверный результат, новый payload.

Важно: redirect() прерывает выполнение как control-flow mechanism. Код после redirect() не должен рассматриваться как обычное продолжение action. Если destination нужны свежие данные, revalidation нужно сделать до redirect.

12. Получить новый RSC / Flight Payload

В схеме узел Получить новый RSC / Flight Payload объединяет несколько источников:

  • navigation запросила новый route;
  • router.refresh() запросил текущий route заново;
  • Server Action вернула обновлённый серверный результат;
  • cache revalidation сделала следующий серверный результат другим;
  • Router Cache уже содержал prefetched payload.

Слово Flight часто используется как название потока/формата, в котором передаётся React Server Components результат. В пользовательской документации Next.js чаще используется термин RSC Payload.

Для статьи важно понимать не внутренний протокол, а роль payload:

RSC Payload — это вход для React на клиенте. Он содержит серверный результат и информацию о Client Component boundaries. После его получения Next.js не “вставляет HTML в DOM”, а передаёт данные в React-дерево.

13. RSC payload merge

RSC payload merge — центральный узел этого блока.

На этом этапе клиентская среда Next.js/React берёт новый RSC Payload и совмещает его с текущим клиентским деревом. Server Components обновляются по серверному payload. Client Component boundaries остаются местами, где интерактивность живёт на клиенте.

Для router.refresh() документация Next.js прямо подчёркивает: клиент merge-ит обновлённый RSC Payload без потери незатронутого client-side React state и browser state. Это не значит, что любой state всегда сохраняется. Состояние сохраняется там, где React может сохранить соответствующие Client Components по обычным правилам дерева. Если route segment, key или граница компонента меняются так, что старая ветка перестаёт соответствовать новой, состояние может быть сброшено.

Поэтому правильная модель такая:

RSC Payload обновляет серверную часть дерева.
Client Components остаются клиентскими границами.
React решает, что сохранить и что заменить, по обычным правилам reconciliation.

RSC payload merge не является DOM mutation. DOM изменится только после того, как React обработает новое дерево через Fiber, render/reconciliation и commit.

14. Что происходит с Client Components

Client Components не становятся Server Components из-за RSC update. Они остаются клиентскими.

Пример:

// Server Component
export default async function Page() {
  const products = await getProducts();

  return <ProductList products={products} />;
}
'use client';

export function ProductList({ products }: { products: Product[] }) {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <ul>
      {products.map(product => (
        <li key={product.id} onClick={() => setSelectedId(product.id)}>
          {product.name}
        </li>
      ))}
    </ul>
  );
}

Если Server Action или router.refresh() приводит к новому списку products, RSC Payload может принести новые props для ProductList. После merge React обновит Client Component через обычный React-механизм. Локальный selectedId может сохраниться, если компонент остаётся той же веткой дерева. Но если route segment или key приводит к remount, состояние сбросится.

Важно не путать:

  • RSC Payload может изменить props, которые приходят в Client Component;
  • обработчики событий внутри Client Component всё равно существуют только в клиентском JavaScript;
  • локальный useState внутри Client Component не хранится на сервере;
  • Server Components не получают event handlers и не используют browser APIs.

15. Что происходит с URL, history и scroll

Для navigation App Router обновляет URL и history через router-механизм. router.push() добавляет запись в history stack, router.replace() заменяет текущую запись.

По умолчанию Next.js прокручивает страницу вверх при переходе на новый route. Это поведение можно отключить опцией scroll: false у router.push() или router.replace().

router.refresh() отличается от navigation. Он обновляет текущий route, но не означает переход на другой URL. При refresh Next.js сливает новый RSC Payload без потери browser state вроде scroll position, если соответствующая часть UI остаётся незатронутой.

Это важно для схемы: navigation и refresh могут оба привести к RSC Payload, но их смысл для URL/history разный.

16. Полная последовательность блока

Последовательность для Router navigation:

<Link> или router.push()
→ App Router client-side navigation
→ Router Cache hit или запрос к серверу
→ RSC Payload для нового route
→ RSC payload merge
→ React Fiber
→ render/reconciliation
→ commit

Последовательность для router.refresh():

router.refresh()
→ запрос текущего route на сервер
→ re-render Server Components
→ обновлённый RSC Payload
→ merge без потери незатронутого client/browser state
→ React Fiber
→ render/reconciliation
→ commit

Последовательность для Server Action:

<form action> / action вызвана из Client Component
→ POST request к серверу
→ Server Action выполняет мутацию
→ revalidatePath / updateTag / refresh / cookies / redirect при необходимости
→ return value и/или новый RSC Payload в ответе
→ RSC payload merge или navigation при redirect
→ React Fiber
→ render/reconciliation
→ commit

Главная модель: App Router update может начинаться не с setState, а с серверного payload. Но после RSC payload merge React всё равно возвращается к своему обычному механизму обновления UI.

17. Что важно не перепутать

RSC Payload — не HTML document. HTML может использоваться для initial preview, но RSC Payload нужен React на клиенте для согласования Server и Client Components.

RSC payload merge — не DOM mutation. Это обновление React-дерева на основе серверного payload. DOM меняется позже, на commit.

router.refresh() — не то же самое, что revalidatePath(). Refresh делает новый запрос текущего route и merge-ит обновлённый RSC Payload. revalidatePath() инвалидирует server-side cached data для path и работает только на сервере.

Server Action — не обычный API endpoint в пользовательской модели. Разработчик пишет серверную функцию, которую React/Next может вызвать через action-механизм; Next.js может вернуть и данные, и обновлённый UI в одном Flight stream.

Client Component не становится серверным после RSC update. Он остаётся клиентской границей с локальным state, effects и event handlers.

Client state update не входит в этот блок. Если update начинается с setState внутри Client Component, это обычная React ветка.
