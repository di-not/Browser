Next.js App Router: потоковый HTML, RSC payload, ссылки на клиентские компоненты и первые JavaScript-фрагменты

Коротко

В App Router начальный ответ сервера — это не только HTML. Next.js использует серверный HTML для быстрого видимого предварительного результата, RSC payload для описания результата Server Components и ссылки на JavaScript-файлы Client Components. Браузер может начать читать потоковый HTML, строить DOM и показывать части страницы до полной загрузки и гидратации клиентского JavaScript. Server Components могут быть видимы как HTML, но сами не гидрируются; гидратация нужна Client Components.

1. Что означает Server/streamed HTML + RSC payload + client refs

В App Router файлы layout и page по умолчанию являются Server Components. Они выполняются на сервере, могут получать данные рядом с серверным источником, использовать секреты без отправки их в браузер и уменьшать количество JavaScript, которое нужно загрузить на клиенте. Если компоненту нужны состояние, обработчики событий, API браузера или эффекты, он должен быть Client Component.

Стартовая запись:

Server/streamed HTML + RSC payload + client refs

означает, что серверный ответ для маршрута App Router состоит из нескольких связанных слоёв.

Server/streamed HTML — серверный HTML, который браузер может начать читать и показывать. Он может прийти сразу целиком или постепенно, фрагментами, если часть маршрута ждёт данные или временно приостановлена.

RSC payload — React Server Component Payload: специальное представление результата дерева Server Components. Оно нужно React и клиентской среде Next.js, чтобы позднее согласовать деревья Server и Client Components.

Client refs — ссылки на Client Components и их JavaScript-файлы. Они указывают, где в дереве находятся границы Client Components и какой JavaScript нужен для их клиентской части.

Это не три независимых ответа. Это части одной модели: HTML даёт быстрый видимый результат, RSC payload описывает дерево React Server Components, а ссылки на клиентские компоненты связывают места Client Components с их JavaScript-фрагментами.

2. Какие компоненты участвуют в этом диапазоне

В этом участке могут встречаться несколько видов компонентов. Их нужно различать, потому что они по-разному участвуют в HTML, RSC payload, JavaScript-фрагментах и гидратации.

Server Components

Server Component выполняется на сервере. В App Router это поведение по умолчанию для page, layout и обычных компонентов, которые не попали в клиентское поддерево модулей.

// app/page.tsx
export default async function Page() {
  const post = await getPost();

  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.description}</p>
    </main>
  );
}

Такой компонент может читать данные на сервере и вернуть UI. Его результат может попасть в HTML и RSC payload. Его компонентный код не нужен браузеру для интерактивности, если сам компонент не является Client Component.

Server Component подходит для серверного получения данных, работы с ключами API и другими секретами, уменьшения клиентского JavaScript и постепенной потоковой отправки содержимого в браузер.

Server Component не подходит для состояния, обработчиков событий, эффектов и API, которые существуют только в браузере. Эти возможности относятся к Client Components.

Client Components

Client Component нужен там, где UI должен быть интерактивным в браузере.

Файл Client Component начинается с директивы:

'use client';
// app/like-button.tsx
'use client';

import { useState } from 'react';

export default function LikeButton({ likes }: { likes: number }) {
  const [count, setCount] = useState(likes);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count} likes
    </button>
  );
}

Этот компонент использует useState и onClick, поэтому он должен быть Client Component. Для него нужен JavaScript в браузере. При начальной загрузке он может участвовать в серверном HTML-предпросмотре, но его интерактивность появится только после загрузки и гидратации соответствующего клиентского JavaScript.

Клиентское поддерево модулей

'use client' создаёт границу не просто в визуальном дереве JSX, а в дереве зависимостей модулей. Если файл помечен 'use client', он и его транзитивные зависимости становятся частью клиентского поддерева модулей.

Это значит:

// client-panel.tsx
'use client';

import Button from './button';
import formatDate from './format-date';

Button и formatDate могут не иметь собственной директивы 'use client', но если они импортированы из модуля с 'use client', они попадут в клиентское поддерево и будут выполняться на клиенте.

При этом один и тот же компонентный модуль без директивы может использоваться по-разному: если он импортирован из серверного кода, его использование может быть Server Component; если он импортирован из клиентского поддерева, его использование может стать клиентской частью. Поэтому нельзя определять тип компонента только по тому, как он выглядит в JSX.

loading.js и запасной UI

loading.js — специальный файл App Router для интерфейса загрузки сегмента маршрута. По умолчанию такой компонент является Server Component, но может быть Client Component, если внутри используется 'use client'.

// app/dashboard/loading.tsx
export default function Loading() {
  return <p>Loading dashboard...</p>;
}

loading.js автоматически создаёт границу Suspense для page и дочерних сегментов ниже текущего сегмента. Пока содержимое сегмента передаётся потоково, пользователь может видеть запасной UI. Когда содержимое готово, оно автоматически заменяет запасной UI.

Для текущей статьи важно не уходить глубоко в Suspense как React-тему, а зафиксировать его роль в HTML-потоке: запасной UI может быть частью раннего HTML, который браузер показывает до готовности более позднего сегмента.

Части маршрута

App Router разбивает маршрут на сегменты. В создании результата участвуют layout, page, вложенные layout, loading.js и другие специальные файлы. В текущем диапазоне важнее всего:

  • layout может формировать общий каркас страницы;
  • page формирует основное содержимое сегмента;
  • loading.js даёт запасной UI для сегмента;
  • вложенные сегменты могут передаваться потоково отдельно.

Из-за этого HTML не обязан ждать, пока вся страница будет полностью готова. Готовые части могут отправляться раньше, а приостановленные части — позже.

3. Как выглядит HTML, который приходит в браузер

Точный HTML, который генерирует Next.js, является внутренней деталью реализации: он может меняться между версиями, режимами сборки и настройками.

В потоковом ответе могут быть такие слои:

  1. Каркас документа: <!doctype html>, <html>, <head>, <body>.
  2. Метаданные и подсказки для загрузки ресурсов: title, meta, ссылки на таблицы стилей, предварительную загрузку и JavaScript.
  3. Статический каркас маршрута: общий layout, навигация, постоянные части страницы.
  4. Запасной HTML: интерфейс загрузки или запасной UI Suspense для сегмента, который ещё не готов.
  5. HTML для готовых Server Components и сегментов маршрута.
  6. Данные и служебные инструкции, связанные с RSC payload, которые нужны React и клиентской среде Next.js для дальнейшего согласования дерева.
  7. Ссылки на клиентские JavaScript-фрагменты для Client Components.

Упрощённый пример того, что пользователь может получить на раннем этапе:

<!doctype html>
<html lang="en">
  <head>
    <title>Dashboard</title>
    <link rel="stylesheet" href="/_next/static/css/app.css">
    <script src="/_next/static/chunks/main.js" defer></script>
  </head>
  <body>
    <nav>Dashboard</nav>
    <main>
      <p>Loading feed...</p>
    </main>
  </body>
</html>

Этот пример не описывает реальный внутренний формат Next.js. Он показывает идею: в раннем HTML уже может быть каркас документа, стили, ссылки на JavaScript-фрагменты, общий layout и запасной UI.

Позже поток может принести HTML для сегмента, который завершил асинхронную работу:

<article>
  <h2>Latest posts</h2>
  <p>Post content...</p>
</article>

В реальном потоковом ответе заменой запасного UI на готовое содержимое управляют React и клиентская среда Next.js через служебные инструкции.

4. Потоковый HTML

Потоковый HTML — HTML, который отправляется не одним блоком после полной готовности всего маршрута, а частями по мере готовности.

В традиционном варианте сервер мог бы сначала дождаться всех данных, собрать всю HTML-страницу и только потом отправить ответ. Потоковая передача меняет это поведение: сервер может отправить статический каркас и запасной UI раньше, а затем досылать готовые сегменты.

Упрощённая логика:

Сначала сервер может отдать layout и запасной UI:

// app/dashboard/loading.tsx
export default function Loading() {
  return <p>Loading feed...</p>;
}

Пользователь видит:

<nav>Dashboard</nav>
<main>
  <p>Loading feed...</p>
</main>

Когда данные для ленты готовы, сервер потоково отправляет содержимое:

<section>
  <h2>Feed</h2>
  <article>...</article>
</section>

После этого запасной UI заменяется готовым содержимым. Документация Next.js по потоковой передаче описывает, что запасной UI отправляется со статическим каркасом, а когда Promise завершается, React потоково отправляет готовый HTML на страницу.

Для диаграммы это соответствует идее:

HTML Parser читает server/streamed HTML
→ DOM создаётся по мере прихода HTML chunks
→ можно показать server HTML / stream segment

HTML chunk — фрагмент ответа, который браузер получил до полного окончания потока. Фрагмент не обязан быть самостоятельным полноценным HTML-документом. Это может быть часть <body>, служебные инструкции, данные, связанные с RSC payload, или содержимое для сегмента. Браузер и клиентская среда React/Next.js совместно доводят эти части до итогового DOM.

5. Как потоковый HTML связан с гидратацией

Потоковый HTML и гидратация связаны, но решают разные задачи.

Потоковый HTML отвечает за ранний показ результата. Сервер отправляет HTML частями, браузер читает эти части, постепенно строит DOM и может показывать пользователю каркас страницы, запасной UI или готовые сегменты маршрута до того, как весь JavaScript для Client Components загружен и выполнен.

Гидратация отвечает за интерактивность. После того как клиентский JavaScript загружен и выполнен, React может привязать обработчики событий к уже существующему DOM. В документации Next.js гидратация описывается именно как процесс прикрепления обработчиков событий к DOM, чтобы статический HTML стал интерактивным.

В App Router это особенно важно, потому что начальная загрузка разделена на несколько слоёв. HTML нужен для быстрого неинтерактивного предварительного результата. RSC payload нужен для согласования деревьев Server и Client Components. JavaScript нужен для гидратации Client Components и интерактивности.

Поэтому потоковый HTML можно считать подготовкой видимого DOM, а не самой гидратацией. Он даёт браузеру HTML, который можно показать. Гидратация приходит позже и подключает клиентскую логику только там, где есть Client Component.

Из этого следует важное правило для диаграммы: Server HTML виден не означает, что Client Components interactive. Видимость HTML может наступить раньше, чем гидратация клиентских частей.

6. RSC payload

RSC payload — React Server Component Payload.

Он не равен HTML. HTML нужен, чтобы пользователь быстро увидел предварительный результат маршрута. RSC payload нужен React и клиентской среде Next.js, чтобы понимать результат дерева Server Components и места, где должны быть Client Components.

В RSC payload входят:

  • результат Server Components;
  • заполнители для мест, где должны быть Client Components;
  • ссылки на JavaScript-файлы Client Components;
  • свойства, переданные от Server Component к Client Component.

Пример структуры приложения:

// app/post/page.tsx — Server Component
import LikeButton from './like-button';

export default async function Page() {
  const post = await getPost();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.text}</p>
      <LikeButton likes={post.likes} />
    </article>
  );
}
// app/post/like-button.tsx — Client Component
'use client';

import { useState } from 'react';

export default function LikeButton({ likes }: { likes: number }) {
  const [count, setCount] = useState(likes);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

Для такой страницы RSC payload на уровне смысла должен описать:

  • серверную часть article, h1, p;
  • место, где должен быть LikeButton;
  • ссылку на JavaScript, который нужен для LikeButton;
  • свойство likes, которое передаётся в LikeButton.

RSC payload не выполняется как обычный JavaScript-компонент. Это данные для клиентской работы React/Next.js. JavaScript для Client Components загружается отдельно как клиентские фрагменты.

7. Client refs: ссылки на клиентские компоненты

Client refs — ссылки на Client Components и их JavaScript-файлы. Они появляются в RSC payload, потому что дерево Server Components может содержать границы Client Components.

Ссылка на клиентский компонент отвечает на вопрос:

В каком месте общего дерева должен быть Client Component и какой JavaScript нужен, чтобы эта клиентская часть могла выполниться и пройти гидратацию?

Пример:

// Server Component
export default function Page() {
  return (
    <main>
      <h1>Product</h1>
      <AddToCartButton productId="p1" />
    </main>
  );
}
// Client Component
'use client';

export function AddToCartButton({ productId }: { productId: string }) {
  return <button onClick={() => addToCart(productId)}>Add to cart</button>;
}

<h1>Product</h1> может быть частью результата Server Component. AddToCartButton требует клиентский JavaScript, потому что у него есть обработчик события. Значит, для этой границы нужны ссылки на клиентский компонент и клиентские JavaScript-фрагменты.

Важное ограничение: свойства, которые Server Component передаёт Client Component, должны быть сериализуемыми React. Это связано с тем, что данные проходят через границу Server → Client.

8. HTML Parser читает server/streamed HTML

Когда браузер получает первые байты HTML-ответа, HTML parser начинает читать документ. Если ответ передаётся потоково, parser может работать постепенно: получил часть HTML — построил часть DOM, получил следующую часть — продолжил.

Пример логики:

Первые фрагменты могут дать документ и каркас:

<html>
  <head>
    <link rel="stylesheet" href="/_next/static/css/app.css">
  </head>
  <body>
    <nav>Dashboard</nav>
    <main>
      <p>Loading...</p>

DOM уже может содержать nav, main и запасной UI. Позже поток продолжится, и DOM будет дополнен или обновлён через потоковые инструкции.

Для браузера это всё ещё чтение HTML и построение DOM. Next.js добавляет к этому модель сегментов маршрута, RSC payload и границ Client Components, но базовая механика чтения HTML остаётся браузерной.

9. DOM создаётся по мере прихода HTML chunks

При потоковой передаче DOM может формироваться постепенно.

Ранний DOM может содержать:

  • элементы корневого layout;
  • навигацию;
  • общий каркас страницы;
  • запасной UI для приостановленного сегмента маршрута;
  • часть серверного содержимого, которая уже готова;
  • служебные элементы script и link для среды выполнения, CSS и JavaScript-фрагментов.

Поздний DOM может получить:

  • готовое содержимое сегмента вместо запасного UI;
  • дополнительные элементы из вложенного маршрута;
  • HTML, связанный с завершившимися асинхронными Server Components.

Такой подход особенно важен для ощущаемой скорости: пользователь видит не пустую страницу, а статический каркас или интерфейс загрузки, пока сервер готовит динамическое содержимое.

loading.js специально поддерживает этот паттерн: он позволяет показать мгновенное состояние загрузки с сервера, пока содержимое сегмента маршрута передаётся потоково, а готовое содержимое автоматически заменяет запасной UI.

10. Обнаружение CSS / client chunks / images / fonts

Пока HTML parser читает потоковый HTML, браузер и клиентская среда Next.js обнаруживают ресурсы, которые нужны странице.

CSS нужен для корректного визуального отображения. Эта тема относится к браузерному блоку CSS loading → CSSOM, который уже разобран во внутренней статье CSS loading и CSSOM.

Изображения и шрифты — обычные дополнительные ресурсы страницы. Они могут загружаться параллельно и влиять на событие load или визуальную стабильность, но в этой статье они не являются главной темой.

Клиентские JavaScript-фрагменты — главный объект для Next.js flow в этом диапазоне. Это JavaScript-файлы, которые нужны для Client Components и клиентской среды выполнения.

Обнаружение клиентских фрагментов может происходить на нескольких уровнях.

На браузерном уровне HTML содержит ссылки script, link, preload или modulepreload, и браузер может начать загрузку этих ресурсов.

На уровне фреймворка RSC payload и служебные манифесты Next.js связывают ссылки на Client Components с конкретными JavaScript-файлами. Эти детали не нужно рассматривать как публичный формат для ручного использования, но они объясняют, почему из дерева Server/Client Components получается набор нужных клиентских фрагментов.

Поэтому блок:

Client Component JS refs обнаружены
→ Client JS chunks загружаются

лучше понимать как смысловой шаг: Next.js знает, какие Client Components нужны для маршрута, и браузер начинает получать JavaScript, необходимый для клиентской части.

11. Откуда берётся клиентский bootstrap в HTML

Серверный HTML в App Router не состоит только из разметки, которую вернули Server Components. Server Component возвращает UI, но Next.js server renderer собирает вокруг этого результата полноценный HTML-документ или HTML-поток. В этот документ Next.js добавляет ссылки на CSS, ссылки на JavaScript-фрагменты и служебные данные, которые нужны клиентской части приложения.

Next.js формирует HTML/stream вокруг результата Server Components и вставляет туда ссылки на клиентские JavaScript-фрагменты и служебные данные. Браузер видит эти ссылки, скачивает соответствующие файлы и позже выполняет их.

Упрощённо в HTML могут встретиться ссылки на файлы вида:

<link rel="stylesheet" href="/_next/static/css/...css">

<script src="/_next/static/chunks/webpack-....js" async></script>
<script src="/_next/static/chunks/main-app-....js" async></script>
<script src="/_next/static/chunks/app/layout-....js" async></script>
<script src="/_next/static/chunks/app/page-....js" async></script>

Имена файлов здесь условные. В реальном проекте они зависят от версии Next.js, режима сборки, маршрута, сборщика и настроек. Их нельзя использовать как стабильный публичный API. Важно другое: браузер получает из HTML ссылки на JavaScript, который нужен клиентской части Next.js и Client Components.

Эти JavaScript-фрагменты можно разделить по смыслу на несколько категорий.

Первая категория — служебный код загрузки и выполнения модулей. Он помогает браузеру и сборщику понять, какие фрагменты уже загружены, какие нужны текущему маршруту, как зарегистрировать модуль и как получить экспорт Client Component.

Вторая категория — клиентская часть Next.js App Router. Она нужна, чтобы приложение могло работать в браузере: использовать данные App Router, читать RSC payload, поддерживать клиентскую навигацию, понимать текущий маршрут и подготовить React к работе с Client Components.

Третья категория — код самих Client Components и их зависимостей. Например, если в дереве есть кнопка с 'use client', её JavaScript должен быть доступен браузеру.

'use client';

export function AddToCartButton() {
  return <button onClick={() => addToCart()}>Add to cart</button>;
}

Когда JavaScript-файл с таким модулем загружен, это ещё не гидратация. Когда файл выполнен и модуль зарегистрирован, это тоже ещё не обязательно гидратация. Гидратация начинается тогда, когда React связывает Client Component с уже существующим DOM и подключает обработчики событий.

12. Client Component JS refs обнаружены

Когда в дереве есть граница Client Component, Next.js должен связать эту границу с JavaScript, который будет загружен на клиенте.

Граница Client Component появляется через 'use client'.

'use client';

export function Counter() {
  return <button onClick={() => {}}>Click</button>;
}

После такой границы всё, что импортируется и напрямую используется внутри клиентского поддерева, попадает в клиентский пакет JavaScript. Поэтому не нужно ставить 'use client' в каждом маленьком компоненте, если он уже находится внутри клиентского поддерева.

Но важно не спутать дерево модулей и дерево отображения. 'use client' определяет границу в дереве зависимостей модулей. Поэтому компонент, переданный как children в Client Component, может остаться Server Component, если он не импортирован внутрь клиентского поддерева модулей.

Пример идеи:

// Server Component
import ClientShell from './client-shell';
import ServerText from './server-text';

export default function Page() {
  return (
    <ClientShell>
      <ServerText />
    </ClientShell>
  );
}

Если ServerText не импортирован в клиентское поддерево модулей, он может оставаться Server Component, хотя визуально находится внутри ClientShell как дочерний элемент. Это важная часть композиции Server и Client Components.

14. Client JS chunks загружаются

После того как ссылки на Client Components известны, браузер начинает загружать JavaScript-фрагменты.

На этом этапе происходит получение файлов, но ещё не гидратация. Разница принципиальная.

Загруженный фрагмент — это файл, доступный браузеру. Выполненный фрагмент — это JavaScript-код, который уже был запущен. Гидратированный Client Component — это Client Component, к HTML которого React подключил обработчики событий и сделал его интерактивным.

Процесс можно описать как последовательность состояний. Сначала становится известно, что в дереве есть Client Component и для него нужен определённый JavaScript. Затем браузер получает соответствующий JavaScript-фрагмент. Только после этого клиентская среда Next.js и загруженный код могут выполниться, React сможет использовать RSC payload, а Client Component сможет пройти гидратацию.

На этом этапе ещё нет гидратации. Браузер только получает JavaScript-фрагменты, чтобы следующий шаг мог выполнить клиентскую среду Next.js и код Client Components.

15. Можно показать server HTML / stream segment?

Пока клиентский JavaScript загружается, браузер уже может иметь достаточно HTML и CSS, чтобы показать серверный результат.

Условие на уровне смысла:

  • DOM для каркаса или сегмента уже построен;
  • CSS для этого отображения загружен и обработан;
  • браузер может перейти к визуальному pipeline.

Здесь работает не гидратация Next.js, а браузерный rendering pipeline:

Style → Layout → Paint → Composite

Подробный разбор этого браузерного pipeline есть во внутренней статье Style → Layout → Paint → Composite.

Главная мысль для Next.js:

Серверный HTML или потоковый сегмент может стать видимым до того, как Client Components стали интерактивными.

Это не ошибка. Это один из смыслов потоковой передачи в App Router: пользователь получает видимое состояние раньше, чем полностью готова клиентская часть.

16. Server HTML виден

Server HTML виден означает, что браузер показал HTML, полученный от сервера.

Видимым может быть:

  • корневой layout;
  • общая навигация;
  • статический каркас страницы;
  • интерфейс загрузки;
  • запасной UI Suspense;
  • серверный сегмент маршрута;
  • HTML-предпросмотр маршрута, включая области, где позже будут Client Components.

Если сегмент приостановлен, пользователь может сначала увидеть запасной UI:

// app/feed/loading.tsx
export default function Loading() {
  return <p>Loading feed...</p>;
}

Потом, когда содержимое сегмента готово, запасной UI заменяется готовым HTML. Визуально это выглядит как постепенное раскрытие страницы: сначала каркас и состояние загрузки, затем основное содержимое.

17. Server Components видимы, но не гидрируются

Server Components видимы, но не гидрируются — одна из самых важных формулировок в этой ветке.

Server Component выполняется на сервере. Его результат может быть виден в браузере как HTML. Но сам Server Component не становится интерактивным браузерным компонентом, потому что его код не отправляется как клиентский компонентный код для гидратации.

Гидратация — это процесс, при котором React прикрепляет обработчики событий к уже существующему DOM, чтобы статический HTML стал интерактивным.

Разница:

Server Component:

  • выполняется на сервере;
  • может попасть в HTML и RSC payload;
  • может быть видимым пользователю;
  • не использует состояние, обработчики событий, эффекты и API, существующие только в браузере;
  • не гидрируется как Client Component.

Client Component:

  • имеет JavaScript в клиентском пакете;
  • может использовать состояние, эффекты, обработчики событий и API браузера;
  • может иметь серверный HTML-предпросмотр при начальной загрузке;
  • нуждается в гидратации, чтобы стать интерактивным.

18. Почему Server Components не гидрируются

Server Components не гидрируются не потому, что они “не важны”. Они уже сделали свою работу на сервере: получили данные, сформировали UI и помогли уменьшить JavaScript, который нужно отправить в браузер.

Гидратация решает другую задачу: сделать HTML интерактивным на клиенте. Для этого нужен компонентный код в браузере и обработчики событий. Server Component не содержит клиентских обработчиков событий и не выполняется в браузере как интерактивный компонент.

Правильная модель такая:

  • Server Component даёт видимый HTML и данные для RSC tree;
  • Client Component даёт интерактивную часть;
  • RSC payload связывает эти два мира;
  • клиентские JavaScript-фрагменты доставляют код для клиентских границ;
  • гидратация применяется к Client Components.

Фраза Server Components видимы, но не гидрируются означает: пользователь видит результат серверного render-а, но интерактивность подключается только там, где есть граница Client Component.

19. Client JS chunks загружены?

Проверка Client JS chunks загружены? относится только к состоянию файлов. На этом шаге браузер уже получил JavaScript-фрагменты, которые нужны клиентской части маршрута, или продолжает их ждать.

Клиентский JavaScript-фрагмент — это не просто “код React вообще”. В App Router он может включать код Client Components, их зависимости, части клиентской среды React/Next.js и служебный код, который нужен для дальнейшей работы приложения в браузере.

Важно разделять три состояния. Загруженный JavaScript-фрагмент — файл уже получен из сети или из cache. Выполненный JavaScript-фрагмент — браузер уже запустил код из этого файла. Готовая клиентская граница — React и Next.js уже знают, где в общем дереве находится соответствующая Client Component часть и какой клиентский модуль ей соответствует. Гидратация начинается позже; в этой статье она не разбирается.

Если фрагменты ещё не загружены, браузер ждёт их. Если они загружены, можно перейти к выполнению клиентской среды Next.js и клиентского кода.

Client JS chunks загружены?
├─ Нет → Ждать client JS chunks
└─ Да  → Запустить клиентский bootstrap Next.js / React

Здесь нет автоматического перехода к интерактивности. Доступность файла — только условие для следующих шагов.

20. Запустить клиентский bootstrap Next.js / React

После того как нужные клиентские JavaScript-фрагменты загружены, браузер может выполнить этот JavaScript. Этот шаг удобно называть запуском клиентского bootstrap Next.js / React.

Bootstrap здесь означает не один официальный файл с таким названием, а начальный запуск клиентской части приложения. Это технический этап: браузер выполняет код, который Next.js вставил в HTML как ссылки на JavaScript-фрагменты. Такой код поднимает клиентскую среду Next.js и React, регистрирует загруженные модули и делает доступными Client Components.

Эта стадия не должна пониматься как отдельный публичный lifecycle-этап Next.js. Официальная модель Next.js проще: HTML даёт быстрый неинтерактивный результат, RSC payload используется для согласования Server и Client деревьев, JavaScript используется для гидратации Client Components и интерактивности. Запуск bootstrap — это техническое условие, без которого JavaScript не смог бы начать эту работу.

На практике в этот момент могут произойти такие вещи:

  • браузер выполняет служебный код Next.js и React;
  • загрузчик модулей регистрирует уже полученные JavaScript-фрагменты;
  • становятся доступными экспорты Client Components;
  • клиентская часть Next.js получает возможность читать RSC payload;
  • React получает основу для дальнейшего перехода к клиентским границам.

Пример с модулем Client Component:

'use client';

console.log('LikeButton module evaluated');

export function LikeButton({ likes }: { likes: number }) {
  return (
    <button onClick={() => console.log('liked')}>
      {likes} likes
    </button>
  );
}

Когда JavaScript-фрагмент с этим модулем выполнен, верхний уровень модуля может выполниться:

LikeButton module evaluated

Это уже выполнение JavaScript, но ещё не обязательно гидратация. Модуль стал доступен клиентской среде, однако React ещё должен сопоставить Client Component с уже существующим DOM и подключить обработчики событий. Это сопоставление относится к гидратации, которая идёт позже.

Правильное различие такое:

загрузить JavaScript
→ браузер получил файл

выполнить JavaScript
→ код модуля и клиентская среда стали доступны

гидратировать Client Component
→ React связал Client Component с существующим DOM и подключил обработчики событий

21. DOMContentLoaded может быть поставлен в очередь

После выполнения соответствующих scripts браузер может поставить DOMContentLoaded в очередь, если HTML parsing уже завершён и выполнены scripts, которые должны завершиться до этого события.

В этой ветке DOMContentLoaded не является React-событием и не означает готовность Client Components. Это событие жизненного цикла документа. Оно показывает, что браузер дошёл до точки, где HTML-документ разобран, а нужные deferred/module scripts выполнены.

Для Next.js App Router важно не перепутать:

DOMContentLoaded может быть поставлен в очередь
≠
Client Components гидратированы
≠
все интерактивные части готовы

В этой статье DOMContentLoaded упоминается только как браузерная развилка после выполнения клиентских scripts. Подробный разбор DOMContentLoaded уже находится в отдельной внутренней статье: DOMContentLoaded → subresources → load.

22. Client runtime использует RSC payload

После запуска клиентского bootstrap клиентская среда Next.js / React получает возможность использовать RSC payload.

RSC payload был получен от сервера как специальное представление результата Server Components. Он содержит результат Server Components, места для Client Components, ссылки на JavaScript-файлы Client Components и свойства, переданные через границу Server → Client.

На клиенте этот payload нужен не для показа “ещё одного HTML”. HTML уже мог быть показан браузером раньше. Payload нужен React и Next.js, чтобы восстановить смысловую структуру дерева: какие части пришли с сервера, где находятся клиентские границы, какие данные нужно передать в Client Components и какой клиентский модуль соответствует этим местам.

Пример:

// Server Component
import LikeButton from './like-button';

export default async function Page() {
  const post = await getPost();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.text}</p>
      <LikeButton likes={post.likes} />
    </article>
  );
}

В HTML пользователь может уже видеть h1, p и даже HTML-предпросмотр кнопки. Но RSC payload дополнительно сообщает клиентской среде, что LikeButton — это граница Client Component, к ней нужно передать likes, а для её клиентской части нужен конкретный JavaScript-модуль.

На этом этапе клиентская среда не выполняет Server Components заново в браузере. Server Components уже выполнились на сервере. Клиент использует payload как описание того, как серверный результат связан с клиентскими границами.

Запуск bootstrap и использование RSC payload можно описать так:

HTML уже мог быть показан
→ JavaScript-фрагменты Next.js / React загружены
→ браузер выполняет клиентский bootstrap
→ клиентская среда получает доступ к RSC payload
→ React/Next понимают, где находятся Client Component boundaries

23. Reconcile / merge Server и Client trees

Reconcile / merge Server и Client trees — это согласование Server и Client деревьев в одно React-дерево на клиенте.

В официальных материалах Next.js эта идея описана так: RSC Payload содержит references к местам Client Components, а на клиенте React использует RSC Payload, чтобы согласовать Server и Client Components в единое дерево. В русской статье удобно писать “согласовать и объединить”, потому что речь не о механическом склеивании HTML-строк, а о восстановлении React-структуры.

Здесь участвуют два разных представления:

  1. Server tree — результат Server Components, уже вычисленный на сервере и описанный в RSC payload.
  2. Client tree — клиентские модули и Client Components, JavaScript которых уже загружен и выполнен.

Согласование нужно, потому что в App Router приложение не является полностью клиентским деревом и не является только HTML. Оно состоит из Server Components, которые выполнялись на сервере, и Client Components, которые должны существовать как клиентские границы.

Пример смысловой структуры:

Server Component: Page
├─ Server Component result: <h1>Product</h1>
├─ Server Component result: <p>Description</p>
└─ Client Component boundary: AddToCartButton

После использования RSC payload клиентская среда понимает, что h1 и p — часть серверного результата, а AddToCartButton — место, где начинается клиентская часть. Само подключение этой клиентской части к DOM относится к следующему этапу, который в этой статье не раскрывается.

24. Границы Client Components

Границы Client Components — точки в общем дереве, где серверная часть передаёт управление клиентской части.

Граница Client Component появляется из-за директивы 'use client'. Она говорит React и Next.js: начиная с этого модуля, код должен быть частью клиентского поддерева модулей и должен быть доступен браузеру.

Пример:

// app/product/add-to-cart-button.tsx
'use client';

export function AddToCartButton({ productId }: { productId: string }) {
  return <button onClick={() => addToCart(productId)}>Add to cart</button>;
}

Если Server Component использует этот компонент:

// app/product/page.tsx
import { AddToCartButton } from './add-to-cart-button';

export default async function Page() {
  const product = await getProduct();

  return (
    <main>
      <h1>{product.title}</h1>
      <AddToCartButton productId={product.id} />
    </main>
  );
}

то в общем дереве появляется граница:

Server Component: Page
├─ server result: <h1>{product.title}</h1>
└─ Client Component boundary: AddToCartButton

Граница означает несколько вещей.

Во-первых, JavaScript для этой клиентской части должен попасть в клиентские фрагменты. Во-вторых, свойства, переданные через эту границу, должны быть сериализуемыми React. В-третьих, Server Component может передать Client Component данные, но не может передать несериализуемую функцию как обычное свойство через эту границу. В-четвёртых, Client Component может использовать состояние, эффекты, обработчики событий и API браузера, потому что её код будет выполняться в браузере.

Граница Client Component не означает, что всё визуально вложенное внутрь неё автоматически стало Client Component. Важна граница в дереве зависимостей модулей. Компонент, переданный через children, может оставаться Server Component, если он не импортирован в клиентское поддерево модулей.

Пример:

// Server Component
import ClientShell from './client-shell';
import ServerText from './server-text';

export default function Page() {
  return (
    <ClientShell>
      <ServerText />
    </ClientShell>
  );
}

ServerText визуально находится внутри ClientShell, но если он не импортирован клиентским модулем, он может оставаться Server Component. Это не просто синтаксическая тонкость. Это объясняет, почему границы Client Components определяются не только JSX-вложенностью, а тем, какие модули входят в клиентское поддерево.

На этом статья останавливается. Следующий этап после границ Client Components — гидратация Client Components, но она здесь намеренно не раскрывается.

25. Полный проход по диапазону

Next.js начинает на сервере. Маршрут разбит на сегменты, а работа по созданию результата может быть разделена между layout, page и границами Suspense. Server Components превращаются в RSC payload. Client Components и RSC payload используются для предварительного HTML, который браузер сможет быстро показать как неинтерактивный результат.

Если часть маршрута готова сразу, она может попасть в ранний HTML. Если часть маршрута ждёт асинхронную работу, App Router может показать интерфейс загрузки или запасной UI Suspense и потоково отправить готовое содержимое позже. Поэтому HTML приходит фрагментами: сначала каркас или запасной UI, потом готовые сегменты.

Next.js server renderer формирует HTML/stream вокруг результата Server Components. В этом HTML могут быть ссылки на CSS, ссылки на JavaScript-фрагменты Next.js и Client Components, а также служебные inline scripts и данные, связанные с потоковой передачей и RSC payload. Поэтому браузер не только строит DOM из HTML, но и обнаруживает ресурсы, которые нужны клиентской части приложения.

Когда DOM и стили готовы, браузер может показать серверный HTML или отдельный потоковый сегмент. Это не означает, что вся страница интерактивна. Server Components могут быть видны как HTML, но они не гидрируются.

После загрузки клиентских JavaScript-фрагментов браузер может запустить клиентский bootstrap Next.js / React. Это не отдельная пользовательская стадия и не сама гидратация, а выполнение JavaScript, который делает доступными клиентскую среду, загрузчик модулей, код Client Components и работу с RSC payload. После выполнения соответствующих scripts DOMContentLoaded может быть поставлен в очередь, если выполнены браузерные условия этого события.

Затем клиентская среда использует RSC payload, чтобы понять серверный результат, места клиентских границ, переданные свойства и ссылки на клиентские модули. React согласует Server и Client деревья в одно общее дерево и выходит к границам Client Components. На этом диапазон статьи заканчивается: сама гидратация Client Components идёт дальше по ветке и здесь не раскрывается.

26. Что важно не перепутать

Server HTML ≠ RSC payload

HTML нужен для видимого предварительного результата. RSC payload нужен React и клиентской среде Next.js для дерева Server/Client Components.

RSC payload ≠ client JavaScript

RSC payload описывает результат Server Components, заполнители Client Components, ссылки на клиентские компоненты и свойства. Client chunks — это JavaScript-файлы, которые браузер загружает и выполняет.

Client refs ≠ загруженные chunks

Ссылки указывают, какой JavaScript нужен. Фрагменты ещё должны быть загружены.

Client JS chunks загружены ≠ client chunks выполнены

Файл может быть получен браузером, но код из него ещё может не быть запущен.

Запустить клиентский bootstrap Next.js / React ≠ гидратация

Запуск клиентского bootstrap означает, что браузер выполняет JavaScript, который поднимает клиентскую среду Next.js / React и делает доступными клиентские модули. Гидратация Client Components — следующий отдельный этап.

Client runtime использует RSC payload ≠ повторно выполняет Server Components в браузере

Server Components уже выполнились на сервере. Клиентская среда использует payload как описание серверного результата и клиентских границ.

Reconcile / merge Server и Client trees ≠ изменить весь DOM заново

Речь о согласовании React-дерева: серверный результат и клиентские границы должны стать единым деревом для дальнейшей клиентской работы.

Границы Client Components ≠ гидратация Client Components

Границы показывают, где в дереве начинается клиентская часть. Гидратация этих границ — следующий этап, который здесь не раскрывается.

Потоковый HTML ≠ гидратация

Потоковый HTML позволяет раньше показать DOM, полученный с сервера. Гидратация позже подключает интерактивность к Client Components.

Server HTML виден ≠ страница полностью interactive

HTML может быть показан как неинтерактивный предварительный результат.

Server Components видимы ≠ Server Components гидрируются

Server Components видимы как результат серверного render-а. Гидратация нужна Client Components.

'use client' boundary ≠ просто визуальная вложенность

Граница проходит по дереву зависимостей модулей. Визуальная вложенность через children не всегда делает дочерний элемент Client Component.
