React SPA до React Render Phase: HTML shell, root container и запуск приложения

Коротко

Этот материал описывает начальную часть client-rendered React SPA: от получения HTML shell до вызова root.render(<App />). Следующий блок, React Render Phase, сюда не входит.

React SPA scenario
→ HTML shell + root container + React bundle ref
→ HTML Parser читает HTML shell
→ DOM создаётся: появляется root container
→ Обнаружение CSS / React module bundle / assets
→ React bundle обычно defer/module
→ Bundle загружается параллельно
→ Можно показать HTML shell?
→ Style → Layout → Paint → Composite
→ HTML shell виден
→ Parser дошёл до конца HTML shell
→ React bundle загружен?
→ Выполнить React bundle
→ React bootstrap
→ createRoot(root).render(App)
→ React Render Phase

На этом участке браузер ещё не выполняет React Render Phase. Сначала он получает HTML shell, создаёт DOM-узел для React-приложения, обнаруживает CSS и JavaScript bundle, может показать HTML shell обычным браузерным rendering pipeline, а затем выполняет JavaScript-код, который вызывает createRoot(...).render(...).

1. Что означает React SPA в этом диапазоне

React SPA — это React-приложение, которое запускается в браузере и отображает интерфейс внутри DOM-узла, уже существующего в HTML.

В client-rendered React SPA сервер обычно отдаёт не полный готовый React UI, а HTML shell: минимальный HTML-документ, в котором есть контейнер для приложения и ссылка на JavaScript-код приложения.

Пример минимального shell:

<body>
  <div id="root"></div>
  <script type="module" src="/assets/index.js"></script>
</body>

<div id="root"></div> — это DOM-узел, в который позже будет отображено React-приложение. В React API такой узел передают в createRoot(domNode). Официальная документация React описывает createRoot как API, который создаёт React root для отображения React-компонентов внутри DOM-узла браузера. Источник: React docs — createRoot.

2. HTML shell + root container + React bundle ref

HTML shell — минимальный HTML-каркас страницы, который браузер получает до выполнения React-приложения.

Обычно в нём есть три важные части:

root container
→ DOM-элемент, куда React позже отобразит приложение

React bundle ref
→ ссылка на JavaScript-файл приложения

CSS / metadata / assets
→ стили, метаданные, изображения или другие ресурсы

Пример:

<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <title>React SPA</title>
    <link rel="stylesheet" href="/assets/index.css">
    <script type="module" src="/assets/index.js"></script>
  </head>
  <body>
    <div id="root"></div>
  </body>
</html>

root container и React root — разные вещи.

root container
→ обычный DOM-элемент из HTML, например <div id="root"></div>

React root
→ объект, который React создаёт через createRoot(domNode)

Документация React показывает такой порядок: сначала код получает DOM-узел, затем вызывает createRoot(domNode), затем вызывает root.render(<App />). Источник: React docs — createRoot.

3. HTML Parser читает HTML shell

После получения HTML браузер начинает читать HTML shell. В процессе чтения он строит DOM.

DOM — объектное представление HTML-документа: элементы, текст и связи между ними. В этом диапазоне ключевой результат — появление DOM-узла, который будет контейнером React-приложения.

<body>
  <div id="root"></div>
</body>

После чтения этой части HTML в документе появляется элемент div#root.

Document
└─ html
   └─ body
      └─ div#root

Позже стартовый JavaScript-код сможет найти этот элемент:

const domNode = document.getElementById('root');

React docs для добавления React в существующий проект показывает похожую идею: в HTML уже есть элемент, например <nav id="navigation">, а React-компонент затем отображается внутри этого существующего DOM-элемента. Источник: React docs — Add React to an Existing Project.

4. Обнаружение CSS / React module bundle / assets

Пока HTML parser читает shell, браузер встречает ссылки на ресурсы.

Пример CSS:

<link rel="stylesheet" href="/assets/index.css">

Пример JavaScript bundle:

<script type="module" src="/assets/index.js"></script>

Пример изображения:

<img src="/assets/logo.svg" alt="Logo">

Asset — дополнительный файл, который нужен странице или приложению: изображение, шрифт, CSS-файл, JavaScript-файл и так далее.

В этом месте важно только разделить роли:

HTML parser
→ создаёт DOM

CSS
→ относится к браузерному блоку загрузки CSS и CSSOM

React bundle
→ JavaScript-файл, который позже выполнит стартовый код приложения

assets
→ дополнительные ресурсы страницы или приложения

CSS-ветка в этой статье не раскрывается подробно: она уже разобрана отдельно в материале CSS loading и CSSOM. Здесь достаточно зафиксировать связь: если в shell есть CSS, браузер должен обработать CSS-часть перед корректным показом shell.

5. React bundle обычно defer/module

В диаграмме React bundle обозначен как обычно defer/module.

Это не правило React API. Для браузера React bundle — обычный JavaScript-ресурс. Его поведение при загрузке зависит от того, как он подключён в HTML.

Пример module script:

<script type="module" src="/assets/index.js"></script>

Современные React-проекты часто используют сборщик: Vite, Parcel, Rsbuild или другой инструмент. Сборщик берёт исходные файлы приложения и готовит файлы для браузера. React docs рекомендует начинать с Vite, Parcel или Rsbuild, если проект не требует React Native. Источник: React docs — Build a React app from Scratch.

Точное поведение defer, module, async и parser-blocking scripts относится к браузерной механике script loading. Эта тема уже разобрана отдельно: Vanilla JS: app script, parsing, DOM и render.

6. Bundle загружается параллельно

Bundle загружается параллельно означает, что загрузка JavaScript-файла может идти одновременно с чтением HTML shell.

Упрощённая последовательность:

parser встретил <script type="module" src="/assets/index.js">
→ браузер начал загрузку JavaScript-файла
→ parser продолжает читать HTML shell
→ DOM продолжает строиться

Здесь важно различать загрузку и выполнение.

загрузка bundle
→ файл получается из сети или cache

выполнение bundle
→ JavaScript-код запускается в браузере

До выполнения bundle React-приложение ещё не стартовало. Браузер только получает JavaScript-файл, который позже вызовет React API.

7. Можно показать HTML shell?

После того как у браузера появилась DOM-структура shell и необходимые стили для текущего отображения, он может перейти к показу shell.

DOM с root container
+ CSS для shell
→ можно показать HTML shell

Это браузерный вопрос, а не React-вопрос. На этом этапе React-компоненты ещё не обязаны быть отрисованы React-ом.

Показ shell происходит через браузерный pipeline:

Style
→ Layout
→ Paint
→ Composite

Этот pipeline подробно разобран в отдельном материале Style → Layout → Paint → Composite. В этой статье достаточно одного различия:

браузерный render
≠
React Render Phase

Браузерный render превращает DOM и CSS в пиксели. React Render Phase — это работа React, которая начинается после вызова root.render(<App />).

8. HTML shell виден

HTML shell виден означает, что браузер уже показал то, что было в HTML shell.

Если shell содержит видимую разметку вне React root, пользователь может увидеть её до запуска React-приложения.

<body>
  <header>Loading app...</header>
  <div id="root"></div>
</body>

Если shell содержит только пустой root container, пользователь может увидеть пустую область или почти пустую страницу.

<body>
  <div id="root"></div>
</body>

В таком случае фраза HTML shell виден не означает, что React UI уже виден. Она означает только то, что браузер смог отобразить текущий HTML shell.

React UI появится внутри root только после выполнения стартового кода React и вызова root.render(<App />). React docs прямо говорит: после создания root нужно вызвать root.render, чтобы отобразить React-компонент внутри него. Источник: React docs — createRoot.

9. Parser дошёл до конца HTML shell

Когда HTML parser дошёл до конца shell, структура начального HTML-документа построена.

HTML Parser читает HTML shell
→ DOM создаётся: появляется root container
→ Parser дошёл до конца HTML shell

К этому моменту DOM-узел #root уже должен существовать, если он был в HTML.

document.getElementById('root')

Этот вызов может вернуть DOM-узел, который затем передаётся в createRoot.

Если DOM-узел не найден, createRoot не получает корректный контейнер. В troubleshooting для createRoot React docs описывает ошибку Target container is not a DOM element: она означает, что переданное значение не является DOM-элементом. Источник: React docs — createRoot.

10. React bundle загружен?

После завершения parsing-а shell браузер может быть в одном из двух состояний:

bundle уже загружен
→ можно выполнить JavaScript-код

bundle ещё не загружен
→ нужно ждать завершения загрузки

В диаграмме это отражено так:

React bundle загружен?
├─ Нет → Ждать React bundle
└─ Да  → Выполнить React bundle

Эта проверка всё ещё относится к загрузке JavaScript-ресурса. React Render Phase ещё не началась.

11. Выполнить React bundle

Когда React bundle загружен и наступил момент выполнения script, браузер запускает JavaScript-код приложения.

Типичный стартовый файл client-rendered React SPA выглядит так:

import { createRoot } from 'react-dom/client';
import App from './App.js';

const domNode = document.getElementById('root');
const root = createRoot(domNode);
root.render(<App />);

В этом коде есть две разные части:

обычный JavaScript startup
→ imports, поиск DOM-узла, выполнение entry file

React startup
→ createRoot(domNode), затем root.render(<App />)

Выполнить React bundle ещё не означает, что React UI уже появился на экране. На этом шаге браузер только выполняет JavaScript-файл. React начинает свою работу тогда, когда выполненный код доходит до вызова root.render(<App />).

React docs показывает именно такой стартовый порядок: импортировать createRoot из react-dom/client, найти DOM-узел, создать root и вызвать root.render(<App />). Источник: React docs — createRoot.

12. DOMContentLoaded может быть поставлен в очередь

После выполнения React bundle в диаграмме есть отдельная подветка:

Выполнить React bundle
→ DOMContentLoaded может быть поставлен в очередь
→ DOMContentLoaded

Это браузерная часть, а не React-этап. DOMContentLoaded связан с готовностью HTML-документа и script-ов, которые должны выполниться до этого события.

Для React SPA с bundle, подключённым как defer или type="module", важна такая последовательность:

HTML parser дошёл до конца HTML shell
→ deferred/module scripts загружены и выполнены
→ DOMContentLoaded может быть поставлен в очередь

Поэтому DOMContentLoaded в этой ветке логично расположен после выполнения React bundle. Если bundle относится к deferred/module scripts, браузер не должен поставить DOMContentLoaded раньше выполнения этого script.

Эта подветка не означает, что DOMContentLoaded ждёт завершения React Render Phase или появления React UI на экране. Она означает только готовность браузера поставить событие после завершения parsing-а и выполнения соответствующих scripts.

Подробности про DOMContentLoaded, load и ожидание subresources уже разобраны отдельно: DOMContentLoaded → subresources → load.

13. React bootstrap

React bootstrap — стартовая инициализация React-приложения в браузере.

В этом диапазоне bootstrap состоит из трёх действий:

найти DOM-узел root container
→ создать React root
→ вызвать root.render(<App />)

Пример:

const domNode = document.getElementById('root');
const root = createRoot(domNode);
root.render(<App />);

domNode — обычный DOM-элемент из HTML. В этой ветке таким элементом обычно является div#root.

<div id="root"></div>

root — React root, созданный через createRoot(domNode). React docs описывает createRoot как API, который создаёт root для отображения React-компонентов внутри DOM-узла браузера. После создания root нужно вызвать root.render, чтобы React отобразил компонент внутри этого root. Источник: React docs — createRoot.

Если DOM-узел не найден или в createRoot передано не DOM-значение, React не получает корректный контейнер. В troubleshooting для createRoot React docs описывает ошибку Target container is not a DOM element: она означает, что переданное значение не является DOM-элементом. Источник: React docs — createRoot.

14. createRoot(root).render(App)

В диаграмме запись:

createRoot(root).render(App)

сжато обозначает две операции.

Первая операция создаёт React root:

const root = createRoot(domNode);

Вторая операция просит React отобразить приложение внутри этого root:

root.render(<App />);

Это граница перед React Render Phase.

root.render(<App />)
→ React Render Phase

React docs по Render and Commit описывает дальнейшую работу React как процесс из трёх шагов:

Trigger
→ Render
→ Commit

В этой статье важна только граница: вызов root.render(<App />) является trigger для initial render. Сама React Render Phase, вызов компонентов, последующий commit и DOM mutations относятся к следующему участку диаграммы. Источник: React docs — Render and Commit.

15. Что важно не перепутать

HTML shell виден ≠ React UI виден

Браузер может показать HTML shell до выполнения React bundle. Если внутри shell только пустой root container, React UI ещё не отображён.

Browser render ≠ React render

Style → Layout → Paint → Composite выполняет браузер. React Render Phase выполняет React.

Bundle загружен ≠ bundle выполнен

Загрузка означает, что JavaScript-файл получен. Выполнение означает, что код из этого файла запущен.

Bundle выполнен ≠ React Render Phase завершена

После запуска bundle код ещё должен дойти до root.render(<App />). Этот вызов только запускает React-процесс; он не является завершением React render/commit-работы.

DOMContentLoaded может быть поставлен в очередь ≠ React UI уже виден

DOMContentLoaded относится к parsing-у HTML и выполнению соответствующих scripts. Оно не означает, что React уже выполнил Render Phase, Commit Phase и что React UI уже появился на экране.

React bundle обычно defer/module ≠ правило React

Это относится к подключению script в HTML. React не определяет сам способ загрузки <script>.

root container ≠ React root

Root container — DOM-элемент из HTML. React root — объект, созданный через createRoot(domNode).

createRoot ≠ hydrateRoot

createRoot используется для client-rendered приложения. React docs отдельно указывает: если HTML был сгенерирован react-dom/server, нужно использовать hydrateRoot, а не createRoot. Источники: React docs — createRoot, React docs — hydrateRoot.
