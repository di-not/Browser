HTTP Request — запрос

Коротко о блоке

HTTP Request — это сообщение, которое клиент отправляет серверу. В сценарии браузера клиентом обычно является браузер, а сервером — origin server, CDN, reverse proxy или другой HTTP-сервер на пути запроса.

В общей схеме этот блок идёт после подготовки соединения:

Переиспользовать TCP/TLS/QUIC соединение
или
Открыть TCP/TLS или QUIC соединение
→ HTTP Request — запрос

На этом этапе браузер уже знает:

scheme
host / authority
port
path
query

и может сформировать HTTP-запрос.

HTTP Request — это не «страница открылась», а конкретное сообщение от клиента к серверу: что нужно получить или сделать, к какому ресурсу обратиться, какие метаданные передать и, иногда, какое тело запроса отправить.

1. HTTP как request-response протокол

HTTP работает по модели request-response:

client → HTTP Request → server
client ← HTTP Response ← server

Для навигации по URL браузер обычно отправляет request, а сервер возвращает response.

Примеры HTTP-клиентов:

браузер
fetch() в JavaScript
curl
мобильное приложение
серверное приложение

Примеры серверной стороны:

origin server
CDN
reverse proxy
backend service
edge server

2. Из чего состоит HTTP Request

Логически HTTP Request содержит:

method
request target
HTTP version / protocol framing
headers
optional body

Для HTTP/1.1 запрос выглядит текстово:

GET /products?page=2 HTTP/1.1
Host: example.com
Accept: text/html
Accept-Language: ru,en;q=0.9
Accept-Encoding: gzip, br, zstd
User-Agent: ...
Cookie: session=abc

Упрощённая структура:

start line
headers
empty line
optional body

3. Start line в HTTP/1.1

В HTTP/1.1 первая строка request называется request line.

Пример:

GET /products?page=2 HTTP/1.1

Она состоит из:

method         = GET
request target = /products?page=2
HTTP version   = HTTP/1.1

Method

HTTP method указывает действие, которое клиент хочет выполнить.

Основные методы:

GET     → получить представление ресурса
HEAD    → получить headers как у GET, но без body ответа
POST    → отправить данные на обработку
PUT     → заменить ресурс
PATCH   → частично изменить ресурс
DELETE  → удалить ресурс
OPTIONS → узнать доступные возможности/настройки

Для обычного перехода по ссылке или ввода URL в адресную строку чаще всего используется: 

GET

Например:

GET /about HTTP/1.1
Host: example.com

Request target

Request target — цель запроса внутри HTTP-сообщения.

Для обычного запроса к origin server в HTTP/1.1 это чаще всего: 

path + query

Пример URL:

https://example.com/products?page=2#reviews

В request target попадёт:

/products?page=2

А fragment/hash:

#reviews

в HTTP Request не отправляется.

HTTP version

В HTTP/1.1 request line содержит версию:

HTTP/1.1

В HTTP/2 и HTTP/3 сообщение не передаётся как текстовая строка GET /path HTTP/1.1. Там используются бинарные фреймы и pseudo-headers, но семантика остаётся похожей.

4. HTTP/2 и HTTP/3: pseudo-headers

В HTTP/2 и HTTP/3 вместо текстовой start line используются специальные pseudo-headers.

Пример логического представления:

:method: GET
:scheme: https
:authority: example.com
:path: /products?page=2
accept: text/html
accept-language: ru,en;q=0.9
cookie: session=abc

Главные pseudo-headers:

:method    → HTTP method, например GET
:scheme    → scheme, например https
:authority → host/authority, например example.com
:path      → path + query

Смысл остаётся тем же:

что сделать?
по какой схеме?
к какому authority?
какой path + query запросить?

5. Headers

HTTP headers — это метаданные запроса или ответа. В request они помогают серверу понять контекст запроса и выбрать правильный response.

Формат HTTP/1.x:

Header-Name: value

Примеры request headers:

Host: example.com
Accept: text/html
Accept-Language: ru,en;q=0.9
Accept-Encoding: gzip, br, zstd
User-Agent: ...
Cookie: session=abc
Referer: https://example.com/
Cache-Control: max-age=0
If-None-Match: "abc123"

Host

Host указывает имя хоста и, при необходимости, порт.

Пример:

Host: example.com

или:

Host: example.com:8443

Host важен, потому что один IP-адрес может обслуживать несколько сайтов. Сервер использует Host, чтобы понять, к какому виртуальному хосту относится запрос.

В HTTP/2 и HTTP/3 аналогичную роль обычно выполняет pseudo-header: 

:authority: example.com

Accept

Accept сообщает серверу, какие media types клиент готов получить.

Пример:

Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8

Для навигации браузер обычно указывает, что предпочитает HTML-документ.

Accept-Language

Accept-Language сообщает предпочтительные языки пользователя.

Пример:

Accept-Language: ru,en;q=0.9

Сервер может использовать это для выбора локализованной версии ответа.

Accept-Encoding

Accept-Encoding сообщает, какие алгоритмы сжатия клиент поддерживает.

Пример:

Accept-Encoding: gzip, br, zstd

Сервер может вернуть сжатый ответ, например:

Content-Encoding: br

User-Agent

User-Agent содержит информацию о клиенте: браузере, платформе и иногда движке.

Пример:

User-Agent: Mozilla/5.0 ...

Этот заголовок исторически используется для совместимости, аналитики и feature detection, но полагаться на него как на точное описание возможностей браузера часто ненадёжно.

Cookie

Cookie отправляет серверу подходящие cookies для данного запроса.

Пример:

Cookie: session=abc; theme=dark

Cookies прикрепляются браузером автоматически, если они подходят по правилам domain, path, secure, SameSite и другим атрибутам.

Referer

Referer сообщает адрес страницы, с которой был сделан переход или запрос.

Пример:

Referer: https://example.com/catalog

Название исторически написано как Referer, а не Referrer.

Cache-related request headers

Браузер может отправлять заголовки, связанные с кэшированием и проверкой актуальности ресурса.

Примеры:

Cache-Control: max-age=0
If-None-Match: "abc123"
If-Modified-Since: Tue, 15 Nov 2026 12:45:26 GMT

Эти заголовки важны для revalidation: браузер спрашивает сервер, изменилась ли сохранённая копия ресурса.

6. Body запроса

Body — тело запроса.

У обычного GET при навигации body обычно отсутствует.

Пример GET без body:

GET /products?page=2 HTTP/1.1
Host: example.com

У POST, PUT, PATCH body часто есть.

Пример POST:

POST /api/login HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 44

{"email":"user@example.com","password":"..."}

Если request содержит body, важны representation headers:

Content-Type: application/json
Content-Length: 44

7. Content-Type в request

Content-Type в request описывает тип данных в body.

Примеры:

Content-Type: application/json
Content-Type: application/x-www-form-urlencoded
Content-Type: multipart/form-data
Content-Type: text/plain

Для обычного GET-навигационного запроса body обычно нет, поэтому Content-Type для тела запроса обычно не нужен.

8. Safe, idempotent и cacheable методы

HTTP-методы имеют семантику.

Safe

Safe method — метод, который должен только читать данные и не должен менять состояние сервера.

Примеры:

GET
HEAD
OPTIONS

Важно: “safe” означает намерение семантики метода, а не гарантию, что сервер физически ничего не сделает.

Idempotent

Idempotent method — метод, повторное выполнение которого должно иметь тот же эффект, что и одно выполнение.

Примеры:

GET
HEAD
PUT
DELETE
OPTIONS

POST обычно не считается idempotent.

Cacheable

Некоторые ответы на запросы могут кэшироваться, если это разрешено правилами HTTP caching.

Чаще всего кэшируют responses на GET.

9. Request при navigation, subresource и fetch()

HTTP Request может возникнуть в разных сценариях.

Navigation request

Пользователь вводит URL или кликает ссылку:

адресная строка / link click
→ navigation request

Обычно:

GET HTML document

Subresource request

HTML-документ содержит ресурсы:

<link rel="stylesheet" href="/styles.css">
<script src="/app.js"></script>
<img src="/logo.png">

Браузер делает отдельные HTTP requests:

GET /styles.css
GET /app.js
GET /logo.png

fetch/XHR request

JavaScript может инициировать request:

fetch("/api/users")

Тогда request может быть:

GET /api/users
POST /api/login
PATCH /api/profile

10. Credentials, cookies и CORS

Некоторые запросы включают credentials:

cookies
HTTP authentication
TLS client certificates

При same-origin запросах cookies обычно могут отправляться автоматически, если подходят по правилам cookies.

Для cross-origin запросов включаются правила CORS. CORS — это механизм на основе HTTP headers, который позволяет серверу указать, каким origin разрешён доступ к ресурсу.

В некоторых cross-origin сценариях браузер перед основным запросом отправляет preflight request методом OPTIONS.

11. Что HTTP Request не делает

Важно не путать этапы.

HTTP Request не выполняет DNS. DNS уже был нужен раньше, чтобы получить IP.

HTTP Request не открывает соединение. Соединение уже открыто или выбрано раньше.

HTTP Request не рендерит страницу. Он только отправляет сообщение серверу.

HTTP Request не содержит fragment/hash из URL.

12. Как блок связан с общей схемой

В общей схеме:

Открыть / переиспользовать соединение
→ HTTP Request
→ HTTP Response

HTTP Request использует результаты предыдущих этапов:

scheme      → :scheme в HTTP/2/3, выбор HTTPS ранее
host/port   → Host или :authority
path/query  → request target или :path
cookies     → Cookie header, если применимо
cache state → conditional headers, если нужна revalidation

13. Мини-глоссарий

HTTP

Hypertext Transfer Protocol — протокол прикладного уровня для передачи ресурсов и сообщений между клиентом и сервером.

HTTP Request

Сообщение клиента серверу.

Method

Действие, которое клиент хочет выполнить: GET, POST, PUT, DELETE и т.д.

GET

Метод для получения представления ресурса.

POST

Метод для отправки данных на обработку.

Request target

Цель запроса. Для обычного HTTP/1.1 запроса чаще всего path + query.

Header

Метаданные HTTP-сообщения.

Body

Тело запроса или ответа.

Host

HTTP/1.1 header, указывающий host и, при необходимости, port.

:authority

HTTP/2/HTTP/3 pseudo-header, близкий по роли к Host.

:path

HTTP/2/HTTP/3 pseudo-header, содержащий path + query.

Cookie

Request header, через который браузер отправляет cookies серверу.

CORS

Cross-Origin Resource Sharing — механизм HTTP headers для разрешения cross-origin доступа.

Preflight

Предварительный CORS-запрос методом OPTIONS.
