Что может вызвать re-render в React и как избегать лишних re-render-ов

Коротко

В React “перерисовка” в разговорной речи обычно означает re-render: React снова вызывает компонент, чтобы вычислить, каким должен быть UI. Это не то же самое, что браузерный Paint, и не то же самое, что DOM mutation.

Базовая формула:


React re-render
→ React снова вызывает компонент
→ компонент возвращает новый JSX snapshot
→ React сравнивает новый результат с предыдущим
→ если есть отличия, позже на commit phase React применит изменения к DOM

Поэтому важно разделять:


re-render компонента
≠
DOM точно изменился

DOM mutation
≠
браузер уже нарисовал пиксели

browser paint
≠
React render phase

Главные причины re-render-а:


state update
dispatch из useReducer
re-render родителя
новые props
изменение context
изменение external store snapshot
изменение key/type, приводящее к remount
state update из Effect
setState во время render в редком разрешённом паттерне
class setState / forceUpdate
root render со стороны React DOM
development-only проверки Strict Mode

Главный принцип оптимизации:

Не нужно пытаться убрать каждый re-render. Re-render сам по себе не всегда проблема. Оптимизировать стоит там, где есть измеримая задержка, лишняя работа дорогого поддерева или цепочки обновлений.

1. Что именно считать “перерисовкой”

В React есть несколько разных уровней, которые часто смешивают.

React render

React render — это вызов компонентов React, чтобы понять, что должно быть на экране.


function Profile({ name }: { name: string }) {
  return <h1>{name}</h1>;
}

Когда React вызывает Profile снова, это re-render компонента.

React commit

Commit phase — этап, где React применяет найденные изменения к DOM.


render phase
→ React вычислил следующий UI

commit phase
→ React применил DOM mutations, если они нужны

Browser render pipeline

После DOM mutations браузер может выполнить свою визуальную работу:


Style
→ Layout
→ Paint
→ Composite

Это уже браузерная часть, а не React render phase.

Поэтому в статье слово “re-render” означает именно повторный вызов компонента React, а не repaint пикселей.

2. State update через `useState`

Самая прямая причина re-render-а — вызов setter-функции из useState.


import { useState } from 'react';

export function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Когда вызывается setCount, React получает запрос обновить state и render-ит компонент снова с новым значением.

Важно:


setCount(...)
→ не меняет переменную count внутри уже выполняющегося handler-а
→ ставит update для следующего render-а

Если внутри handler-а после setCount прочитать count, там всё ещё будет старое значение из текущего render snapshot.

Как избежать лишних re-render-ов при state update

Не хранить в state то, что можно вычислить во время render

Плохо:


function FullName({ firstName, lastName }: Props) {
  const [fullName, setFullName] = useState('');

  useEffect(() => {
    setFullName(firstName + ' ' + lastName);
  }, [firstName, lastName]);

  return <p>{fullName}</p>;
}

Здесь props изменились → компонент отрендерился → Effect вызвал setFullName → компонент отрендерился ещё раз.

Лучше:


function FullName({ firstName, lastName }: Props) {
  const fullName = firstName + ' ' + lastName;
  return <p>{fullName}</p>;
}

State нужен для данных, которые React должен помнить между render-ами и которые не могут быть просто вычислены из текущих props/state.

Не обновлять state, если значение не изменилось

React сравнивает новое значение state с текущим через Object.is. Если значение идентично, React может пропустить re-render компонента и его children как оптимизацию.


setCount(count); // обычно не приведёт к полезному обновлению

Для объектов и массивов это особенно важно:


Object.is({}, {}) // false

Даже если два объекта выглядят одинаково, это разные ссылки.

Держать state ближе к месту использования

Плохо:


function App() {
  const [hovered, setHovered] = useState(false);

  return (
    <>
      <BigTree />
      <Card hovered={hovered} onHoverChange={setHovered} />
    </>
  );
}

Если transient state поднят слишком высоко, re-render родителя может затрагивать слишком большое поддерево.

Лучше держать такой state внутри Card, если он не нужен выше:


function Card() {
  const [hovered, setHovered] = useState(false);
  return <div onMouseEnter={() => setHovered(true)} />;
}

Сгруппировать связанные state-переменные

Если две переменные всегда обновляются вместе, можно хранить их как один state-объект или reducer-state. Это не всегда уменьшает число render-ов из-за batching, но часто упрощает модель и уменьшает вероятность цепочек синхронизации через Effects.

Плохо:


const [x, setX] = useState(0);
const [y, setY] = useState(0);

function move(point: Point) {
  setX(point.x);
  setY(point.y);
}

В React 18 эти updates обычно batch-ятся, но иногда модель лучше выразить как одно состояние:


const [position, setPosition] = useState({ x: 0, y: 0 });

function move(point: Point) {
  setPosition({ x: point.x, y: point.y });
}

3. `dispatch` через `useReducer`

useReducer тоже вызывает re-render, когда dispatch приводит к новому state.


import { useReducer } from 'react';

type State = { count: number };
type Action = { type: 'increment' } | { type: 'reset' };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 };
    case 'reset':
      return { count: 0 };
  }
}

export function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });

  return (
    <button onClick={() => dispatch({ type: 'increment' })}>
      {state.count}
    </button>
  );
}

dispatch похож на setter из useState: он ставит update, React позже вызывает reducer, получает next state и render-ит компонент с новым state.

Если reducer вернул то же значение state по Object.is, React может пропустить re-render как оптимизацию.

Как избежать лишних re-render-ов при `useReducer`

Reducer должен возвращать старый state, если изменений нет

Плохо:


function reducer(state: State, action: Action): State {
  if (action.type === 'noop') {
    return { ...state };
  }

  return state;
}

{ ...state } создаёт новый объект, поэтому для React это новое значение.

Лучше:


function reducer(state: State, action: Action): State {
  if (action.type === 'noop') {
    return state;
  }

  return state;
}

Не мутировать state внутри reducer

Плохо:


function reducer(state: State, action: Action): State {
  state.count++;
  return state;
}

Здесь ссылка на объект та же. React может не увидеть полезного изменения, а логика становится нечистой.

Лучше:


function reducer(state: State, action: Action): State {
  return { count: state.count + 1 };
}

Использовать reducer для сложной логики, а не для всего подряд

useReducer полезен, когда много связанных переходов состояния. Если state простой, useState обычно понятнее. Избыточный reducer не уменьшает re-render-ы сам по себе.

4. Re-render родителя

По умолчанию, когда компонент re-render-ится, React re-render-ит и его дочерние компоненты.


function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(count + 1)}>
        {count}
      </button>
      <ExpensiveChild />
    </>
  );
}

При изменении count re-render-ится Parent. ExpensiveChild тоже может быть вызван снова, даже если он не использует count.

Это нормально: React снова вычисляет UI поддерева. Если результат не изменился, DOM может остаться прежним.

Как избежать лишних re-render-ов из-за родителя

Держать state ниже

Если state нужен только маленькой части, переместить его в эту часть.

Плохо:


function Page() {
  const [input, setInput] = useState('');

  return (
    <>
      <Header />
      <Sidebar />
      <SearchBox value={input} onChange={setInput} />
      <BigContent />
    </>
  );
}

Лучше:


function Page() {
  return (
    <>
      <Header />
      <Sidebar />
      <SearchBox />
      <BigContent />
    </>
  );
}

function SearchBox() {
  const [input, setInput] = useState('');
  return <input value={input} onChange={e => setInput(e.target.value)} />;
}

Теперь ввод в SearchBox не заставляет Page снова вычислять всё дерево из-за этого state.

Использовать композицию через `children`

Если компонент-обёртка имеет свой локальный state, но визуально содержит большое поддерево, можно передать поддерево как children.


function Panel({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false);

  return (
    <section>
      <button onClick={() => setOpen(o => !o)}>Toggle</button>
      {open && children}
    </section>
  );
}

Так React лучше понимает, что children не зависят от локального state обёртки, если они были созданы выше.

Использовать `memo` для дорогих дочерних компонентов


const ExpensiveChild = memo(function ExpensiveChild({ value }: { value: string }) {
  return <SlowList value={value} />;
});

memo говорит React: если props не изменились, компонент обычно можно не re-render-ить при re-render-е родителя.

Ограничения:


memo — оптимизация, не гарантия
memo не блокирует own state updates
memo не блокирует context updates, которые компонент сам читает
props сравниваются shallow через Object.is, если не передан custom compare

5. Props изменились

Компонент может re-render-иться, когда получает новые props от родителя.


function Parent() {
  const [name, setName] = useState('Ada');
  return <Greeting name={name} />;
}

function Greeting({ name }: { name: string }) {
  return <h1>Hello, {name}</h1>;
}

Если name изменился, Greeting должен получить новый props и вычислить новый UI.

Тонкость: для обычного компонента без memo сам факт re-render-а родителя уже обычно приводит к re-render-у child. Но при оптимизации через memo именно изменение props становится критерием, по которому React решает, можно ли пропустить re-render child.

Как избежать лишних re-render-ов из-за props

Передавать минимальные props

Плохо:


<Profile person={person} />

если person пересоздаётся часто, хотя child нужен только name.

Лучше:


<Profile name={person.name} />

Примитивные props проще сравнивать и легче стабилизировать.

Не создавать новые объекты и массивы без необходимости

Плохо для memoized child:


const Child = memo(function Child({ options }: { options: Options }) {
  return <Chart options={options} />;
});

function Parent() {
  return <Child options={{ theme: 'dark' }} />;
}

На каждом render-е создаётся новый объект.

Лучше:


const options = useMemo(() => ({ theme: 'dark' }), []);

return <Child options={options} />;

Или ещё лучше — передать отдельные примитивы, если возможно:


<Child theme="dark" />

Стабилизировать функции через `useCallback`, когда они ломают memo

Плохо:


const ProductList = memo(function ProductList({ onBuy }: Props) {
  // ...
});

function Page({ productId }: { productId: string }) {
  return (
    <ProductList
      onBuy={() => {
        buy(productId);
      }}
    />
  );
}

Каждый render создаёт новую функцию, поэтому memo у child не поможет.

Лучше:


const handleBuy = useCallback(() => {
  buy(productId);
}, [productId]);

return <ProductList onBuy={handleBuy} />;

useCallback нужен не “для всех функций”, а когда стабильная ссылка действительно важна: например, функция передаётся memoized child или используется как dependency другого Hook.

Использовать `useMemo` для дорогих вычислений

Если re-render неизбежен, можно не повторять дорогую чистую калькуляцию при неизменных dependencies.


const visibleTodos = useMemo(
  () => filterTodos(todos, tab),
  [todos, tab]
);

Это не предотвращает re-render компонента. Это предотвращает повторное выполнение дорогой калькуляции внутри re-render-а, пока dependencies не изменились.

6. Изменение context

Компонент, который читает context через useContext, re-render-ится, когда значение ближайшего Provider для этого context меняется.


const ThemeContext = createContext('light');

function Button() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Save</button>;
}

Если Provider получил новое value, React re-render-ит consumers, которые читают этот context. Значения сравниваются через Object.is.

Важно:


memo не блокирует re-render из-за context,
если memoized component сам читает этот context

Частый источник лишних context re-render-ов

Плохо:


function App() {
  const [currentUser, setCurrentUser] = useState(null);

  function login(response: Response) {
    setCurrentUser(response.user);
  }

  return (
    <AuthContext value={{ currentUser, login }}>
      <Page />
    </AuthContext>
  );
}

На каждом render-е App создаёт новый объект и новую функцию. Для context это новое value, поэтому consumers могут re-render-иться.

Как избежать лишних context re-render-ов

Мемоизировать объект Provider value


const login = useCallback((response: Response) => {
  setCurrentUser(response.user);
}, []);

const authValue = useMemo(
  () => ({ currentUser, login }),
  [currentUser, login]
);

return (
  <AuthContext value={authValue}>
    <Page />
  </AuthContext>
);

Разделять contexts

Плохо:


<AppContext value={{ theme, currentUser, locale, permissions }}>
  <App />
</AppContext>

Любое изменение одного поля меняет весь объект context value.

Лучше разделить по независимым областям:


<ThemeContext value={theme}>
  <AuthContext value={authValue}>
    <LocaleContext value={locale}>
      <App />
    </LocaleContext>
  </AuthContext>
</ThemeContext>

Тогда компонент, читающий только theme, не обязан re-render-иться из-за изменения auth.

Разделить consumer на внешний и memoized child

Если компоненту нужна только часть context, можно прочитать context во внешнем компоненте, а в memoized child передать минимальный prop.


function GreetingContainer() {
  const theme = useContext(ThemeContext);
  return <Greeting theme={theme} name="Ada" />;
}

const Greeting = memo(function Greeting({
  theme,
  name,
}: {
  theme: string;
  name: string;
}) {
  return <h1 className={theme}>Hello, {name}</h1>;
});

Это помогает, когда большой child не должен зависеть от всего context.

7. Изменение external store snapshot

Если компонент подписан на внешний store через useSyncExternalStore, React может re-render-ить его при изменении snapshot.


const todos = useSyncExternalStore(
  todosStore.subscribe,
  todosStore.getSnapshot
);

React вызывает subscribe, чтобы подписаться на store. Когда store меняется, callback уведомляет React. React снова вызывает getSnapshot. Если новый snapshot отличается от предыдущего через Object.is, компонент re-render-ится.

Примеры external store:


Redux-подобный store
Zustand-подобный store
самописная подписка
browser API с mutable value и events

Как избежать лишних re-render-ов из external store

Возвращать стабильный snapshot, если данные не изменились

Плохо:


function getSnapshot() {
  return { todos: store.todos };
}

Каждый вызов возвращает новый объект.

Лучше кэшировать snapshot и возвращать тот же объект, пока данные не изменились.


let lastTodos = store.todos;
let lastSnapshot = { todos: lastTodos };

function getSnapshot() {
  if (store.todos !== lastTodos) {
    lastTodos = store.todos;
    lastSnapshot = { todos: lastTodos };
  }

  return lastSnapshot;
}

Делать snapshot immutable

Если store внутри mutable, snapshot должен отражать изменения безопасно. Иначе React может не заметить изменение или получить несовместимое значение.

Не пересоздавать `subscribe` внутри компонента без необходимости

Плохо:


function Todos() {
  const todos = useSyncExternalStore(
    callback => todosStore.subscribe(callback),
    todosStore.getSnapshot
  );

  return <List todos={todos} />;
}

Функция subscribe создаётся заново на каждом render-е. React может переподписываться.

Лучше вынести:


const subscribe = (callback: () => void) => {
  return todosStore.subscribe(callback);
};

function Todos() {
  const todos = useSyncExternalStore(
    subscribe,
    todosStore.getSnapshot
  );

  return <List todos={todos} />;
}

8. Изменение `key` или типа компонента

Изменение key или типа компонента — это не обычный re-render. Это может привести к remount: React удаляет старый компонент в этой позиции и создаёт новый. State при этом сбрасывается.

Пример намеренного reset:


function ProfilePage({ userId }: { userId: string }) {
  return <ProfileForm key={userId} userId={userId} />;
}

Когда userId изменится, ProfileForm будет создан заново, и его state сбросится.

Как случайно вызвать remount

Менять key без необходимости

Плохо:


<Item key={Math.random()} />

Каждый render создаёт новый key, React считает это новым компонентом.

Лучше:


<Item key={item.id} />

Определять компонент внутри другого компонента

Плохо:


function Parent() {
  function Child() {
    return <input />;
  }

  return <Child />;
}

На каждом render-е Parent создаётся новая функция Child. React может воспринимать это как новый тип компонента в этой позиции, из-за чего state child будет сбрасываться.

Лучше:


function Child() {
  return <input />;
}

function Parent() {
  return <Child />;
}

Как избегать лишних remount


использовать стабильные keys
не использовать Math.random() / Date.now() как key
не менять key, если не нужен reset state
не определять компонентные функции внутри других компонентов
сохранять одинаковый тип компонента в той же позиции дерева

9. Effect вызывает state update

useEffect сам по себе не re-render-ит компонент. Но если внутри Effect вызвать setState, это создаст новый update.

Плохо:


function ProductList({ products, filter }: Props) {
  const [visibleProducts, setVisibleProducts] = useState([]);

  useEffect(() => {
    setVisibleProducts(filterProducts(products, filter));
  }, [products, filter]);

  return <List items={visibleProducts} />;
}

Здесь происходит лишняя цепочка:


props изменились
→ render со старым visibleProducts
→ commit
→ Effect
→ setVisibleProducts
→ второй render

Лучше:


function ProductList({ products, filter }: Props) {
  const visibleProducts = filterProducts(products, filter);
  return <List items={visibleProducts} />;
}

Если фильтрация дорогая:


const visibleProducts = useMemo(
  () => filterProducts(products, filter),
  [products, filter]
);

Как избегать лишних re-render-ов из Effects

Удалять Effects, которые только синхронизируют derived state

Если значение можно вычислить из текущих props/state во время render, Effect не нужен.

Переносить action-specific логику в event handler

Плохо:


useEffect(() => {
  if (isSubmitted) {
    sendAnalytics();
  }
}, [isSubmitted]);

Лучше:


function handleSubmit() {
  sendAnalytics();
  setSubmitted(true);
}

Если действие вызвано конкретным событием пользователя, часто логика должна быть в handler-е, а не в Effect.

Разделять независимые Effects

Плохо:


useEffect(() => {
  connectChat(roomId);
  logVisit(pageId);
}, [roomId, pageId]);

Если pageId изменился, effect переподключит чат, хотя чат зависит только от roomId.

Лучше:


useEffect(() => {
  return connectChat(roomId);
}, [roomId]);

useEffect(() => {
  logVisit(pageId);
}, [pageId]);

Убирать лишние object/function dependencies

Плохо:


const options = { roomId };

useEffect(() => {
  const connection = createConnection(options);
  connection.connect();
  return () => connection.disconnect();
}, [options]);

options новый на каждом render-е, значит effect будет re-run-иться.

Лучше создать объект внутри Effect:


useEffect(() => {
  const options = { roomId };
  const connection = createConnection(options);
  connection.connect();
  return () => connection.disconnect();
}, [roomId]);

Или стабилизировать через useMemo, если объект нужен ещё где-то.

Использовать ref для данных, которые не должны влиять на UI

Если значение нужно хранить между render-ами, но его изменение не должно re-render-ить компонент, можно использовать useRef.


const lastClickTimeRef = useRef<number | null>(null);

function handleClick() {
  lastClickTimeRef.current = Date.now();
}

Изменение ref.current не вызывает re-render.

10. `setState` во время render

Обычно вызывать setState во время render нельзя. Но React допускает редкий паттерн: вызвать setter во время render текущего компонента, чтобы сохранить информацию о предыдущих render-ах. React отбросит текущий output и сразу попробует render снова с новым state.

Пример из практического смысла:


function CountLabel({ count }: { count: number }) {
  const [prevCount, setPrevCount] = useState(count);
  const [trend, setTrend] = useState<'up' | 'down' | null>(null);

  if (prevCount !== count) {
    setPrevCount(count);
    setTrend(count > prevCount ? 'up' : 'down');
  }

  return (
    <p>
      {count} {trend}
    </p>
  );
}

Это редкий паттерн. В большинстве случаев лучше вычислять значение во время render или использовать key для reset.

Как избегать лишних render retry


не вызывать setState во время render без необходимости
не синхронизировать props в state через Effect, если можно вычислить во время render
использовать key для намеренного reset state
проверять условие перед setState, иначе можно получить бесконечный цикл

11. `root.render`

На уровне React DOM можно явно вызвать render root-а:


const root = createRoot(document.getElementById('root')!);
root.render(<App />);

При повторном вызове root.render React обновляет root tree, сравнивая новый React tree с предыдущим. В обычном приложении фреймворк или bootstrap-код вызывает root render, а внутри UI обновления обычно идут через state/context/store.

Это причина re-render-а на уровне корня, но в повседневном компонентном коде она встречается реже, чем state/props/context.

Как избегать лишних root render


не вызывать root.render в цикле или на каждое внешнее событие без необходимости
подключать внешние данные через useSyncExternalStore или state management
оставить root.render для bootstrap / смены верхнеуровневого tree

12. Strict Mode в development

<StrictMode> может вызвать дополнительные render-вызовы в development. Это нужно, чтобы найти нечистые компоненты, неправильные Effects и проблемы с ref callbacks.

Пример:


<StrictMode>
  <App />
</StrictMode>

В development React может вызвать компонентную функцию дважды. Это не production-поведение и не причина оптимизировать компонент через memo.

Правильная реакция:


не убирать Strict Mode ради “меньше render-ов”
сделать render logic чистой
убрать мутации props/state
правильно реализовать cleanup Effects

Если компонент ломается от повторного вызова, это признак нечистой логики.

13. Что не вызывает re-render само по себе

Изменение `ref.current`


const countRef = useRef(0);

function handleClick() {
  countRef.current++;
}

Изменение ref не вызывает re-render. Если значение должно отображаться на экране, нужен state.

Изменение локальной переменной


let count = 0;

function handleClick() {
  count++;
}

Локальная переменная не сохраняется между render-ами и не сообщает React, что нужно render-ить компонент снова.

Мутация объекта state без setter

Плохо:


const [person, setPerson] = useState({ name: 'Ada' });

function rename() {
  person.name = 'Grace';
}

React не получает update. Нужно создать новый объект и передать его в setter:


setPerson({
  ...person,
  name: 'Grace',
});

Мутация массива state без setter

Плохо:


items.push(newItem);

Лучше:


setItems([...items, newItem]);

Ручное изменение DOM


ref.current!.textContent = 'Changed';

Ручное изменение DOM не является React state update. Оно может быть перезаписано React на следующем commit. Для UI, которым управляет React, лучше выражать изменения через state/props.

14. `memo`: когда помогает и когда нет

memo помогает пропустить re-render child, когда parent re-render-ится, но props child не изменились.


const Greeting = memo(function Greeting({ name }: { name: string }) {
  return <h1>Hello, {name}</h1>;
});

memo полезен, если:


компонент часто получает те же props
parent re-render-ится часто
render child дорогой
props можно сделать стабильными

memo не поможет, если:


сам компонент обновляет собственный state
компонент читает context, который изменился
props каждый раз новые по ссылке
render дешёвый и нет заметной проблемы

Custom compare


const Chart = memo(ChartImpl, (prevProps, nextProps) => {
  return prevProps.dataPoints.length === nextProps.dataPoints.length;
});

Custom compare опасен, если он не сравнивает все props, включая функции. Функции часто закрывают старые props/state, и неправильное сравнение может привести к stale closures.

Использовать custom compare стоит только тогда, когда сравнение реально дешевле render-а и корректность легко доказать.

15. `useMemo`: что он оптимизирует

useMemo кэширует результат вычисления между re-render-ами.


const visibleTodos = useMemo(
  () => filterTodos(todos, tab),
  [todos, tab]
);

Он помогает:


не повторять дорогую чистую калькуляцию
сохранять стабильную ссылку на объект/массив для memoized child
сохранять стабильную dependency для другого Hook

Он не делает компонент “не re-render-ящимся”. Компонент всё равно может быть вызван снова. Просто внутри этого вызова React может вернуть кэшированное значение.

Не нужно оборачивать каждую переменную в useMemo. Это добавляет сложность. Начинать лучше с чистой архитектуры state и измерений.

16. `useCallback`: что он оптимизирует

useCallback кэширует определение функции между re-render-ами.


const handleSubmit = useCallback((orderDetails: OrderDetails) => {
  post('/buy', { productId, orderDetails });
}, [productId]);

Он полезен, когда:


функция передаётся в memoized child
функция используется как dependency другого Hook
нужна стабильная ссылка для внешней подписки

Он не предотвращает создание логики вообще и не ускоряет автоматически каждый компонент. Если child не memoized и функция не dependency, useCallback часто не даёт пользы.

17. React Compiler

React Compiler может автоматически применять часть мемоизации, уменьшая необходимость ручных memo, useMemo и useCallback. Но это не отменяет базовую модель:


state/props/context/store updates
→ могут вызвать re-render

чистый render
→ обязателен

лишние derived state Effects
→ всё ещё плохая модель

измерение производительности
→ всё ещё важно

Если проект не использует React Compiler или использует его частично, обычные правила оптимизации остаются актуальными.

18. Как диагностировать лишние re-render-ы

React DevTools Profiler

Использовать Profiler, если взаимодействие заметно тормозит. Он показывает, какие части дерева render-ились и сколько времени это заняло.

`<Profiler>`

Для программного измерения можно использовать компонент <Profiler>:


<Profiler id="SearchResults" onRender={onRender}>
  <SearchResults />
</Profiler>

Callback получает информацию о commit-е и времени render-а. Это инструмент измерения, а не способ оптимизации сам по себе.

Локальный `console.log`

Для учебного анализа можно временно поставить:


console.log('Render ExpensiveChild');

Но на него нельзя слепо полагаться в development, потому что Strict Mode может вызывать render дважды.

19. Что важно не перепутать


re-render ≠ DOM mutation

Компонент может быть вызван снова, но React может не изменить DOM.


DOM mutation ≠ browser paint

После DOM changes браузер ещё выполняет свой rendering pipeline.


state update ≠ мгновенное изменение state-переменной

Setter обновляет state для следующего render-а.


same state value может быть skipped

Если новое значение идентично текущему через Object.is, React может пропустить re-render как оптимизацию.


memo ≠ защита от всех re-render-ов

memo касается props от родителя. Own state и context всё равно могут вызвать re-render.


useMemo ≠ запрет re-render-а

useMemo кэширует вычисление, но компонент всё равно может быть вызван.


useCallback ≠ ускоритель по умолчанию

Он полезен, когда важна стабильная ссылка на функцию.


ref update ≠ re-render

ref.current можно менять без re-render-а, поэтому ref не подходит для данных, которые должны отображаться на экране.


key change ≠ обычный re-render

Изменение key может сбросить state и создать компонент заново.


Strict Mode double render ≠ production render

Это development-only проверка чистоты.

20. Итоговая модель

React re-render может начаться, когда React получил сигнал, что входные данные для UI могли измениться:


state
props
context
external store snapshot
root tree
key/type identity
class forceUpdate

Но хороший React-код не пытается вручную запретить все re-render-ы. Он делает другое:


держит render чистым
хранит минимальный state
не дублирует derived data в state
не создаёт цепочки Effects
локализует state
разделяет context
использует memo/useMemo/useCallback только там, где есть причина
измеряет performance перед оптимизацией

Правильная финальная формула:


Re-render — это способ React заново вычислить UI.
Лишним он становится не сам по себе,
а когда из-за структуры state, props, context или Effects
React вынужден часто пересчитывать дорогие части дерева без необходимости.
