Локального ответа нет → DNS → соединение

Границы блока

Этот материал описывает ветку, которая начинается после проверки локального ответа и продолжается до момента, когда браузер готов отправить HTTP-запрос. 

IP для hostname уже известен?
├─ No
│  └─ DNS Resolution — получение IP
│     └─ Есть переиспользуемое соединение?
│        ├─ Yes → Переиспользовать TCP/TLS/QUIC соединение
│        └─ No  → Открыть TCP/TLS или QUIC соединение
│
└─ Yes
   └─ Есть переиспользуемое соединение?
      ├─ Yes → Переиспользовать TCP/TLS/QUIC соединение
      └─ No  → Открыть TCP/TLS или QUIC соединение

1. Общий смысл ветки

Если браузер не может получить ответ локально — через Service Worker, Cache Storage или HTTP Cache, — он должен перейти к сетевой подготовке.

На этом этапе браузер уже знает компоненты URL:

scheme
hostname
port
path
query

Например:

https://example.com/products?page=2

Даёт:

scheme   = https
hostname = example.com
port     = 443 по умолчанию
path     = /products
query    = ?page=2

Чтобы реально подключиться к серверу, браузеру нужен не просто hostname, а сетевое направление: 

IP-адрес + порт + подходящий транспорт

Поэтому дальше браузер решает три задачи:

1. Известен ли IP для hostname?
2. Если IP неизвестен, получить его через DNS.
3. Есть ли уже подходящее соединение, или нужно открыть новое?

2. «IP для hostname уже известен?»

Определение

Блок «IP для hostname уже известен?» означает проверку: есть ли у браузера, операционной системы или сетевой инфраструктуры уже известный IP-адрес для имени хоста.

Пример:

hostname = example.com

Браузеру нужно узнать:

example.com → IP-адрес

Если это соответствие уже известно, браузер может не выполнять новый DNS lookup.

Откуда IP может быть уже известен

IP-адрес может быть известен из нескольких мест:

DNS cache браузера
DNS cache операционной системы
DNS cache локального резолвера
предыдущие запросы к тому же hostname
dns-prefetch
preconnect
сам URL уже содержит IP-адрес

Пример URL, где DNS для домена не нужен:

http://192.168.0.1/admin

Здесь вместо доменного имени уже указан IPv4-адрес:

192.168.0.1

Hostname и IP — не одно и то же

hostname — это имя хоста из URL:

example.com

IP-адрес — это сетевой адрес, к которому можно подключаться:

93.184.216.34

или IPv6:

2606:2800:220:1:248:1893:25c8:1946

Доменное имя удобно человеку, но сетевому стеку нужен IP-адрес.

3. DNS Resolution — получение IP

Что такое DNS

DNS расшифровывается как Domain Name System.

DNS — это иерархическая и децентрализованная система имён для интернет-ресурсов. Она связывает доменные имена с ресурсными записями, в том числе с IP-адресами.

Главная идея:

человек использует: example.com
компьютеру нужен: IP-адрес
DNS помогает получить: example.com → IP

Что такое DNS Resolution

DNS Resolution — процесс получения IP-адреса по hostname.

Пример:

hostname = example.com
DNS resolution
→ 93.184.216.34

После этого браузер может готовить соединение к:

93.184.216.34:443

Порт при этом берётся не из DNS, а из URL:

явно указанный port
или порт по умолчанию из scheme

Например:

https → 443
http  → 80

DNS работает не со всем URL

DNS не получает полный URL:

https://example.com/products?page=2#reviews

DNS работает только с именем:

example.com

DNS не обрабатывает:

scheme = https
path = /products
query = ?page=2
fragment = #reviews

DNS lookup и DNS cache

DNS lookup — операция поиска DNS-записей для hostname.

Перед реальным запросом к DNS-серверам браузер или ОС могут проверить кэши: 

browser DNS cache
OS DNS cache
resolver cache

Если запись найдена и ещё актуальна, новый DNS-запрос может не понадобиться.

TTL

TTL — Time To Live.

В DNS TTL показывает, сколько времени запись может храниться в кэше.

Пример:

example.com A 93.184.216.34 TTL=300

Упрощённо:

этот результат можно кэшировать 300 секунд

После истечения TTL запись считается устаревшей, и её может потребоваться получить заново.

A, AAAA и CNAME

A record — DNS-запись, которая указывает IPv4-адрес. 

example.com A 93.184.216.34

AAAA record — DNS-запись, которая указывает IPv6-адрес. 

example.com AAAA 2606:2800:220:1:248:1893:25c8:1946

CNAME — Canonical Name record.

CNAME указывает, что одно доменное имя является алиасом другого.

www.example.com CNAME example.com
example.com A 93.184.216.34

Упрощённо:

www.example.com → example.com → IP

Что получается после DNS

После DNS resolution браузер получает один или несколько IP-адресов.

Например:

example.com
→ 93.184.216.34
→ 2606:2800:220:1:248:1893:25c8:1946

Дальше браузер выбирает подходящий адрес и переходит к проверке соединения.

4. «Есть переиспользуемое соединение?»

Определение

После того как IP-адрес уже известен или получен через DNS, браузер проверяет: 

Есть ли уже открытое соединение, которое подходит для этого запроса?

Если подходящее соединение есть, браузер может использовать его повторно. Если нет — нужно открыть новое.

Почему соединение переиспользуют

Открытие нового соединения занимает время.

Для HTTPS поверх TCP обычно нужны:

TCP handshake
TLS handshake
согласование протокола

Если соединение уже открыто и подходит, браузер может избежать части этих затрат и быстрее перейти к отправке запроса.

Persistent connection

Persistent connection — постоянное соединение, которое остаётся открытым между несколькими последовательными запросами.

Вместо модели:

один запрос → одно соединение → закрыть

используется модель:

открыть соединение
→ отправить запрос
→ получить ответ
→ оставить соединение открытым
→ использовать для следующих запросов

Keep-Alive

Keep-Alive — разговорное название идеи переиспользования соединения.

Смысл:

не закрывать соединение сразу после ответа,
а оставить его для следующих запросов

В HTTP/1.1 persistent connections являются обычной моделью.

HTTP/2 multiplexing

В HTTP/2 несколько запросов и ответов могут передаваться как потоки внутри одного соединения.

Упрощённо:

одно TCP/TLS соединение
→ несколько HTTP/2 streams
→ CSS, JS, images и API-запросы могут идти через одно соединение

Это называется multiplexing.

Преимущество:

меньше новых соединений
меньше handshake overhead
лучшее использование одного соединения

HTTP/3 и QUIC

В HTTP/3 используется QUIC.

HTTP/3 сохраняет HTTP semantics, но переносит их поверх QUIC вместо TCP. QUIC предоставляет транспорт с потоками, multiplexing, flow control, защищённостью и более быстрой установкой соединения в некоторых сценариях.

Упрощённо:

HTTP/1.1 → обычно TCP
HTTP/2   → обычно TCP + TLS
HTTP/3   → QUIC поверх UDP

Когда соединение можно переиспользовать

Для базового понимания:

соединение можно переиспользовать,
если оно уже открыто и подходит для нужного сетевого назначения

На практике браузер учитывает больше факторов:

scheme
hostname / authority
port
HTTP version
TLS certificate
proxy settings
network partitioning
состояние соединения
лимиты браузера
возможность connection coalescing

Для собеседования обычно достаточно сказать:

Браузер проверяет, есть ли уже открытое соединение к подходящему origin/authority. Если есть — использует его, если нет — открывает новое.

5. Ветка: «Переиспользовать TCP/TLS/QUIC соединение»

Что происходит

Если подходящее соединение уже существует, браузер не открывает новое. 

Есть переиспользуемое соединение?
→ Yes
→ Переиспользовать TCP/TLS/QUIC соединение

Пример:

браузер загрузил HTML с https://example.com
HTML ссылается на /style.css и /app.js
если соединение к example.com:443 ещё открыто,
браузер может использовать его для следующих ресурсов

Что такое TCP

TCP — Transmission Control Protocol.

TCP — транспортный протокол, который обеспечивает надёжную доставку данных между двумя узлами.

Он используется как транспортная основа для HTTP/1.1 и HTTP/2 в типичных браузерных сценариях.

Что такое TLS

TLS — Transport Layer Security.

TLS обеспечивает защищённое соединение поверх недоверенной сети.

Он даёт:

шифрование
проверку подлинности сервера
защиту целостности данных

HTTPS — это HTTP, защищённый TLS.

Что такое QUIC

QUIC — транспортный протокол поверх UDP.

QUIC включает:

защищённость
потоки
multiplexing
flow control
connection migration
низкую задержку установки соединения

HTTP/3 работает поверх QUIC.

6. Ветка: «Открыть TCP/TLS или QUIC соединение»

Что происходит

Если подходящего соединения нет, браузер открывает новое.

Есть переиспользуемое соединение?
→ No
→ Открыть TCP/TLS или QUIC соединение

Выбор зависит от схемы URL, поддержки сервера, поддержки браузера и доступного HTTP-протокола.

HTTPS через TCP + TLS

Для обычного HTTPS поверх TCP упрощённая цепочка такая:

DNS → IP
TCP handshake
TLS handshake
соединение готово

TCP handshake

TCP handshake — процедура установления TCP-соединения.

Упрощённо:

client → SYN
server → SYN-ACK
client → ACK

После этого TCP-соединение считается установленным.

TLS handshake

TLS handshake — процедура установки защищённого соединения.

На этом этапе браузер и сервер:

согласуют параметры шифрования
проверяют сертификат сервера
получают ключи для защищённой передачи

После успешного TLS handshake можно безопасно передавать HTTP-данные.

HTTP/3 через QUIC

Если используется HTTP/3, браузер открывает QUIC-соединение.

Упрощённо:

DNS → IP
QUIC connection setup
соединение готово

QUIC работает поверх UDP и включает встроенную защиту соединения. Поэтому для HTTP/3 не используется отдельная TCP-связка.

Как браузер выбирает TCP/TLS или QUIC

Браузер может использовать HTTP/3, если:

браузер поддерживает HTTP/3
сервер поддерживает HTTP/3
браузер знает, что сервер поддерживает HTTP/3
сеть не блокирует UDP/QUIC

Если HTTP/3 не используется, для HTTPS обычно применяется TCP + TLS.

7. Как выглядит вся третья ветка

IP уже известен

локального ответа нет
→ IP для hostname уже известен
→ проверить переиспользуемое соединение
   ├─ соединение есть
   │  → переиспользовать TCP/TLS/QUIC соединение
   │
   └─ соединения нет
      → открыть TCP/TLS или QUIC соединение

IP неизвестен

локального ответа нет
→ IP для hostname неизвестен
→ DNS Resolution
→ получить IP
→ проверить переиспользуемое соединение
   ├─ соединение есть
   │  → переиспользовать TCP/TLS/QUIC соединение
   │
   └─ соединения нет
      → открыть TCP/TLS или QUIC соединение

8. Что важно не перепутать

DNS не получает HTML

DNS не загружает страницу и не получает HTML.

DNS только помогает получить IP-адрес по hostname:

example.com → IP

DNS не знает path и query

DNS не обрабатывает:

/products
?page=2
#reviews

Он работает только с именем:

example.com

IP не равен серверному маршруту

IP указывает, куда подключиться на сетевом уровне. А path и query будут важны позже, когда браузер сформирует HTTP Request.

Переиспользуемое соединение не равно кэшу

Кэш — это сохранённый response.

Переиспользуемое соединение — это уже открытый сетевой канал.

HTTP Cache → можно ли не идти в сеть за response
Connection reuse → можно ли не открывать новое соединение

QUIC не является «просто TCP быстрее»

QUIC не использует TCP. Он работает поверх UDP и реализует собственные механизмы потоков, защиты, контроля потока и восстановления.

9. Мини-глоссарий

Hostname

Имя хоста из URL, например example.com.

IP address

Сетевой адрес устройства или сервера. Бывает IPv4 или IPv6.

DNS

Domain Name System — система сопоставления доменных имён с IP-адресами и другими записями.

DNS Resolution

Процесс получения IP-адреса по hostname.

DNS lookup

Операция поиска DNS-записей для доменного имени.

DNS cache

Кэш результатов DNS lookup.

TTL

Time To Live — время, в течение которого DNS-запись может считаться актуальной в кэше.

A record

DNS-запись, указывающая IPv4-адрес.

AAAA record

DNS-запись, указывающая IPv6-адрес.

CNAME

Canonical Name record — DNS-запись, которая делает одно имя алиасом другого.

TCP

Transmission Control Protocol — надёжный транспортный протокол.

TLS

Transport Layer Security — протокол защиты соединения.

HTTPS

HTTP поверх TLS.

QUIC

Транспортный протокол поверх UDP, используемый HTTP/3.

UDP

User Datagram Protocol — транспортный протокол без установки TCP-соединения; QUIC работает поверх UDP.

Persistent connection

Соединение, которое остаётся открытым и может использоваться для нескольких запросов.

Keep-Alive

Идея сохранения соединения открытым для повторного использования.

Multiplexing

Передача нескольких потоков запросов и ответов через одно соединение.

HTTP/2

Версия HTTP, которая поддерживает multiplexing поверх одного соединения.

HTTP/3

Версия HTTP, которая работает поверх QUIC.
