React initial render: первый render компонента, commit, useLayoutEffect и useEffect

Коротко

Initial render — первый render компонента. В client-rendered React SPA он запускается после создания React root и вызова root.render(<App />).

import { createRoot } from 'react-dom/client';
import App from './App.js';

const root = createRoot(document.getElementById('root'));
root.render(<App />);

В этой статье рассматривается начальный путь:

root.render(<App />)
→ React Render Phase
→ React Commit Phase
→ DOM mutations
→ useLayoutEffect
→ browser paint
→ useEffect

Во время Render Phase React вызывает компоненты и вычисляет, какой UI нужен. Во время Commit Phase React применяет результат к DOM. useLayoutEffect выполняется после commit, но до того, как браузер перерисует экран. useEffect относится к passive effects и обычно выполняется после paint.

1. Что запускает initial render

React docs в статье Render and Commit описывает две причины render-а компонента:

это первый render компонента
state компонента или одного из его предков обновился

Здесь речь о первом варианте.

Для первого render-а нужно вызвать root.render(...).

const root = createRoot(document.getElementById('root'));
root.render(<App />);

createRoot(domNode) создаёт React root для DOM-узла браузера. DOM-узел — это обычный элемент документа, например <div id="root"></div>. React root — объект React, через который React управляет содержимым внутри этого DOM-узла.

<body>
  <div id="root"></div>
</body>
const domNode = document.getElementById('root');
const root = createRoot(domNode);

После создания root вызов root.render(<App />) просит React отобразить компонент App внутри этого root. Это и есть trigger initial render — запуск первого render-а. Источник: React docs — createRoot.

2. Trigger initial render

Trigger — момент, когда React получает задачу подготовить UI.

root.render(<App />)
→ trigger initial render

Вызов root.render(<App />) не означает, что DOM уже изменён в эту же секунду как готовый результат. Он запускает React-процесс.

Удобно разделять три шага:

Trigger
→ React получил запрос отобразить App

Render
→ React вызывает компоненты и вычисляет результат

Commit
→ React применяет результат к DOM

React docs использует именно такую модель: trigger, render, commit. Источник: React docs — Render and Commit.

3. Что такое React Render Phase

React Render Phase — этап, на котором React вызывает компоненты.

Компонент в React — функция, которая возвращает описание UI.

function App() {
  return <h1>Hello</h1>;
}

Когда React выполняет initial render, он начинает с root component.

root.render(<App />)
→ React вызывает App()

Если App возвращает другие компоненты, React вызывает и их.

function App() {
  return (
    <main>
      <Header />
      <Profile />
    </main>
  );
}

function Header() {
  return <h1>Hello</h1>;
}

function Profile() {
  return <p>User profile</p>;
}

Упрощённая последовательность:

React вызывает App()
→ App возвращает JSX с Header и Profile
→ React вызывает Header()
→ Header возвращает JSX
→ React вызывает Profile()
→ Profile возвращает JSX
→ React получает описание всего UI

React docs описывает это как рекурсивный процесс: если компонент возвращает другой компонент, React render-ит следующий компонент, затем следующий, пока не поймёт, что нужно показать на экране. Источник: React docs — Render and Commit.

4. JSX — это описание UI, а не DOM

Компоненты возвращают JSX.

function App() {
  return <button>Save</button>;
}

JSX — синтаксис для описания UI. Он похож на HTML, но JSX не является DOM-узлом.

JSX
→ описание того, что React должен показать

DOM
→ реальные узлы документа в браузере

Во время Render Phase React собирает описание:

какие элементы нужны
какой текст нужен
какие props переданы компонентам
какие атрибуты должны быть у элементов
какие дочерние элементы вложены внутрь

DOM-изменения на этом этапе ещё не выполняются. React сначала вычисляет результат, а затем применяет его на Commit Phase.

5. Render Phase должна быть чистой

React docs говорит, что компоненты и Hooks должны быть чистыми. Это значит: при одинаковых входных данных компонент должен возвращать одинаковый результат и не должен менять внешний мир во время render-а. Источник: React docs — Components and Hooks must be pure.

Входные данные компонента:

props
state
context

Чистый компонент:

function Greeting({ name }) {
  return <h1>Hello, {name}</h1>;
}

Он получает name и возвращает JSX. Он не изменяет DOM вручную, не меняет глобальные переменные, не запускает таймеры и не подписывается на внешние системы.

Нечистый компонент:

let counter = 0;

function BadComponent() {
  counter = counter + 1;
  return <h1>{counter}</h1>;
}

Этот компонент меняет переменную, которая существовала до render-а. Такой код нарушает правило чистоты.

Side effect — побочный эффект: действие, которое меняет что-то вне текущего вычисления.

Примеры побочных эффектов:

изменить DOM вручную
изменить глобальную переменную
запустить timer
отправить network request
подписаться на внешний источник
изменить объект, который существовал до render-а

React docs объясняет: rendering logic может выполняться несколько раз, поэтому побочные эффекты во время render-а могут приводить к ошибкам и непредсказуемому поведению. Источники: Components and Hooks must be pure, Keeping Components Pure.

6. Что React делает на initial render

Для initial render React вызывает root component и все вложенные компоненты, которые нужны для описания UI.

Пример:

function App() {
  return (
    <section>
      <h1>Dashboard</h1>
      <button>Refresh</button>
    </section>
  );
}

На Render Phase React получает описание:

section
→ h1 с текстом Dashboard
→ button с текстом Refresh

После Render Phase React знает, какие DOM-узлы нужно создать на Commit Phase.

Render Phase
→ вычислить, какой UI нужен

Commit Phase
→ создать и вставить DOM-узлы

Для initial render React docs говорит, что React использует DOM API appendChild() для размещения созданных DOM nodes на экране. Источник: React docs — Render and Commit.

7. React Commit Phase и DOM mutations внутри root

Commit Phase — этап, на котором React применяет результат render-а к DOM.

До commit в HTML может быть только пустой root container:

<body>
  <div id="root"></div>
</body>

После initial commit внутри root появляются DOM-узлы, соответствующие результату React render-а.

<body>
  <div id="root">
    <section>
      <h1>Dashboard</h1>
      <button>Refresh</button>
    </section>
  </div>
</body>

DOM mutations внутри root — это изменения, которые React делает в DOM-дереве внутри контейнера, переданного в createRoot.

root container
→ <div id="root"></div>

DOM mutations внутри root
→ React добавляет, обновляет или удаляет DOM-узлы внутри этого контейнера

Для initial render главная mutation — вставить в пустой контейнер DOM-узлы, которые соответствуют результату Render Phase.

React Render Phase
→ React понял, что нужен section, h1 и button

React Commit Phase
→ React создал DOM-узлы
→ React вставил их внутрь root container

Упрощённо это можно представить так:

до commit:

<div id="root"></div>


после commit:

<div id="root">
  <section>
    <h1>Dashboard</h1>
    <button>Refresh</button>
  </section>
</div>

React docs для initial render говорит, что React использует DOM API appendChild() для размещения созданных DOM nodes на экране. Это не значит, что в реальном приложении всегда вызывается только один appendChild(). Важна идея: на initial commit React создаёт нужные DOM-узлы и помещает их в DOM внутри root container. Источник: React docs — Render and Commit.

React не начинает автоматически управлять всем HTML-документом. Он управляет содержимым внутри того DOM-узла, для которого создан root.

<body>
  <header>Outside React root</header>

  <div id="root">
    <!-- React управляет этой областью -->
  </div>
</body>

В этом примере <header> находится вне root container. React не изменяет его как часть этого root. React работает внутри #root, потому что именно этот DOM-узел был передан в createRoot(domNode).

8. useLayoutEffect после commit и до paint

useLayoutEffect — Hook для кода, который должен выполниться после того, как React уже применил изменения к DOM, но до того, как браузер перерисует экран.

React docs формулирует это так: useLayoutEffect — версия useEffect, которая срабатывает до browser repaint. После commit компонента в DOM и до repaint браузера React запускает setup-функцию useLayoutEffect. Источник: React docs — useLayoutEffect.

Последовательность для initial render:

React Render Phase
→ React Commit Phase
→ DOM уже изменён
→ useLayoutEffect
→ browser paint

setup — функция, которую передают в Hook.

useLayoutEffect(() => {
  // setup logic
}, []);

Пустой массив зависимостей [] означает, что для обычного production-сценария этот layout effect связан с mount компонента: он запускается после того, как компонент оказался в DOM.

Пример задачи для useLayoutEffect — измерить DOM-элемент до того, как пользователь увидит финальный результат.

import { useLayoutEffect, useRef, useState } from 'react';

function Tooltip() {
  const ref = useRef(null);
  const [height, setHeight] = useState(0);

  useLayoutEffect(() => {
    const rect = ref.current.getBoundingClientRect();
    setHeight(rect.height);
  }, []);

  return <div ref={ref}>Tooltip height: {height}</div>;
}

Почему здесь нужен именно useLayoutEffect:

React вставил DOM
→ можно измерить элемент
→ effect запускается до paint
→ state update из useLayoutEffect обрабатывается до paint
→ пользователь не видит промежуточное положение

React docs предупреждает: useLayoutEffect может ухудшить производительность, потому что блокирует browser repaint. Поэтому его нужно использовать только там, где важно выполнить код до paint, например для измерения layout и позиционирования UI. Источник: React docs — useLayoutEffect.

9. Browser paint после layout effects

После commit и layout effects браузер может обновить изображение на экране.

React Commit Phase
→ useLayoutEffect
→ browser paint

Browser paint — не то же самое, что React render. React render — это вызов компонентов. Browser paint — это визуальное обновление экрана после того, как DOM уже изменён.

В общей браузерной схеме это соответствует блоку:

Style
→ Layout
→ Paint
→ Composite

Подробно этот браузерный pipeline разобран отдельно: Style → Layout → Paint → Composite.

Для этой статьи важно только место в последовательности:

DOM изменён React-ом
→ layout effects уже выполнены
→ браузер может показать результат пользователю

10. useEffect после paint

useEffect — Hook для синхронизации компонента с внешними системами.

React docs описывает useEffect как способ подключить компонент к внешней системе: browser API, network, сторонней библиотеке, подписке, timer-у и другим системам вне React. Источник: React docs — useEffect.

Пример:

useEffect(() => {
  const connection = createConnection(serverUrl, roomId);
  connection.connect();

  return () => {
    connection.disconnect();
  };
}, [serverUrl, roomId]);

Здесь:

setup
→ подключиться к внешней системе

cleanup
→ отключиться от внешней системы

Для initial render с пустым массивом зависимостей:

useEffect(() => {
  console.log('Component mounted');
}, []);

effect запускается после того, как компонент был отрендерен и React применил изменения к DOM. В обычном случае useEffect не должен блокировать paint. React docs говорит: если effect делает визуальную работу и заметна задержка до его выполнения, нужно заменить useEffect на useLayoutEffect, потому что useLayoutEffect выполняется до browser paint. Источник: React docs — useEffect.

Упрощённая последовательность:

React Render Phase
→ React Commit Phase
→ useLayoutEffect
→ browser paint
→ useEffect

React DevTools Performance tracks описывает passive effects как фазу, где React выполняет такие Hooks, как useEffect; обычно это происходит после paint. Есть исключение для некоторых пользовательских взаимодействий, когда passive effects могут выполниться раньше paint. Для initial render React SPA обычно полезно держать в голове основную модель: useEffect не используется для блокирования первого paint. Источник: React docs — React Performance tracks.

11. useLayoutEffect и useEffect: главное различие

Оба Hook-а запускаются не во время Render Phase.

Render Phase
→ компоненты должны быть чистыми
→ useLayoutEffect и useEffect здесь не выполняются

Различие в моменте после commit:

useLayoutEffect
→ после DOM mutations
→ до browser paint
→ может читать layout и синхронно повлиять на то, что увидит пользователь

useEffect
→ passive effect
→ обычно после browser paint
→ подходит для синхронизации с внешними системами,
   которая не должна блокировать показ экрана

Если нужно измерить DOM и избежать видимого скачка интерфейса, подходит useLayoutEffect.

Если нужно подключиться к внешней системе, отправить analytics event, подписаться на событие, запустить timer или синхронизироваться после отображения, обычно подходит useEffect.

React docs прямо советует предпочитать useEffect, когда возможно, потому что useLayoutEffect может ухудшить производительность. Источник: React docs — useLayoutEffect.

12. Strict Mode в development

В development-сборке при включённом <StrictMode> React может выполнять дополнительные проверки.

React docs по Strict Mode говорит, что в development компоненты могут re-render-иться дополнительный раз, а Effects могут запускаться дополнительный раз, чтобы найти ошибки, связанные с нечистым render-ом и отсутствующим cleanup. Источник: React docs — StrictMode.

Это не меняет основную production-модель:

render
→ commit
→ layout effects
→ paint
→ passive effects

Но при чтении логов в development можно увидеть больше вызовов, чем ожидалось. Это нормально для Strict Mode и нужно для поиска ошибок.

13. React UI виден и готовность к действиям пользователя

После Commit Phase, layout effects и браузерного paint пользователь может увидеть React UI.

React Commit Phase
→ DOM mutations внутри root
→ useLayoutEffect
→ browser paint
→ React UI виден пользователю

React UI виден пользователю означает, что браузер уже показал DOM-результат, который React вставил внутрь root container.

<div id="root">
  <section>
    <h1>Dashboard</h1>
    <button>Refresh</button>
  </section>
</div>

Но видимость UI и готовность к действиям пользователя — не одно и то же. Для React-приложения важна ещё готовность event handlers.

Event handler — функция, которую React вызовет при пользовательском событии: клике, вводе текста, отправке формы, фокусе и так далее.

function App() {
  function handleClick() {
    console.log('Refresh clicked');
  }

  return <button onClick={handleClick}>Refresh</button>;
}

В React обработчики событий обычно задаются прямо в JSX через props вроде onClick, onChange, onSubmit, onFocus.

<button onClick={handleClick}>Refresh</button>

Когда React выполнил render и commit, DOM-элемент уже находится внутри root, а React знает, какой handler связан с этим элементом. Поэтому после commit и показа UI можно говорить, что React UI готов к действиям пользователя.

React UI виден пользователю
→ React UI interactive / event handlers готовы
→ React SPA готово к действиям пользователя

React UI interactive означает: пользователь может взаимодействовать с элементами, а React сможет обработать соответствующие события через event handlers, объявленные в компонентах.

Важно не смешивать три состояния:

HTML shell виден
→ браузер показал начальный HTML

React UI виден
→ React сделал commit, браузер показал DOM внутри root

React UI interactive
→ event handlers для React UI готовы к пользовательским событиям

Для initial render эти состояния могут идти последовательно:

HTML shell виден
→ React bundle выполнен
→ root.render(<App />)
→ React Render Phase
→ React Commit Phase
→ DOM mutations внутри root
→ browser paint
→ React UI виден пользователю
→ React UI interactive / event handlers готовы
→ React SPA готово к действиям пользователя

Если компонент содержит только статическую разметку без custom React handlers, UI всё равно может быть видимым и часть нативного поведения браузера будет работать. Но фраза React SPA готово к действиям пользователя в этой диаграмме означает, что React-приложение не только показало UI, но и готово обрабатывать пользовательские действия через свои React event handlers.

React docs описывает event handlers как функции, которые запускаются в ответ на взаимодействия пользователя: click, hover, focus input и другие события. Источник: React docs — Responding to Events.

14. Initial render и управление root

После initial render React управляет содержимым внутри root container.

<body>
  <header>Outside React root</header>
  <div id="root">
    <!-- React управляет этой областью -->
  </div>
</body>

Элементы вне root container остаются обычной частью HTML-документа. React не начинает автоматически управлять всем документом; он управляет тем DOM-узлом, для которого был создан root.

React docs по createRoot говорит: для приложения, полностью построенного на React, обычно нужен один root для root component. Если на странице есть несколько независимых областей, можно создать отдельный root для каждой из них. Источник: React docs — createRoot.

15. Что важно не перепутать

initial render ≠ re-render

Initial render — первый render компонента. Re-render — последующий render после обновления state, props или context.

root.render(<App />) ≠ DOM уже полностью готов как результат

root.render(<App />) запускает React-процесс. React должен вызвать компоненты, вычислить UI и применить результат к DOM.

React Render Phase ≠ browser rendering

React Render Phase — вызов компонентов. Browser rendering — визуальное обновление экрана после DOM-изменений.

JSX ≠ DOM

JSX описывает UI. DOM-узлы создаются и вставляются позже.

useLayoutEffect ≠ useEffect

useLayoutEffect выполняется после commit, но до browser paint. useEffect обычно выполняется после paint.

DOM mutations внутри root ≠ управление всем document

React применяет изменения внутри DOM-узла, переданного в createRoot. Элементы вне этого root не становятся частью этого React root.

React UI виден ≠ React SPA готово к действиям пользователя

UI может быть показан после browser paint, но готовность SPA означает ещё и готовность React event handlers к пользовательским событиям.

Effect ≠ render logic

Render должен быть чистым. Effects нужны для синхронизации с внешними системами после render/commit.

initial render ≠ hydration

В этой статье рассматривается createRoot. Для HTML, уже сгенерированного React на сервере, React docs указывает использовать hydrateRoot, а не createRoot. Источник: React docs — hydrateRoot.
