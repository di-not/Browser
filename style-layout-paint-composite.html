Style → Layout → Paint → Composite

Коротко о блоке

Style → Layout → Paint → Composite — это упрощённая схема browser rendering pipeline: процесса, в котором браузер подготавливает визуальный кадр страницы.

В упрощённом виде:

JavaScript / CSS
→ Style
→ Layout
→ Paint
→ Composite

Начальный пункт JavaScript / CSS означает причину изменения: скрипт мог изменить DOM, CSS мог изменить состояние элемента, анимация могла обновить значение свойства, или браузер мог выполнять первый показ страницы после загрузки HTML и CSS.

Браузер не обязан каждый раз проходить все стадии. Если изменилась геометрия элемента, обычно нужны Layout, Paint и Composite. Если изменился только цвет, Layout можно пропустить. Если меняется transform или opacity в подходящих условиях, браузер может обойтись стадией Composite.

1. Границы блока

Блок начинается, когда браузеру нужно подготовить новый визуальный результат.

Причины могут быть такими:

DOM построен впервые
CSS загружен или изменён
DOM изменился
изменился class или style
сработала CSS-анимация
изменился размер окна
пользователь выполнил действие

Блок заканчивается, когда браузер подготовил кадр для вывода на экран. 

Style
→ Layout
→ Paint
→ Composite
→ кадр готов к отображению

Эта схема относится к браузеру. Код приложения может быть причиной изменения, но вычисление стилей, layout, paint и compositing выполняет браузер.

2. Style

Style — стадия вычисления стилей для элементов. Браузер берёт DOM и CSS-данные, находит подходящие правила для элементов и получает итоговые значения CSS-свойств.

Пример:

<h1 class="title">Hello</h1>
h1 {
  color: black;
}

.title {
  color: red;
}

CSS-правило — это блок вида .title { color: red; }.

Selector описывает, к каким элементам применяется правило. В примере .title выбирает элементы с классом title.

Declaration — пара свойство: значение. В примере это color: red.

На стадии Style браузер делает примерно следующее:

DOM-элемент
+ CSS-правила
+ inline styles
+ базовые стили браузера
→ выбрать подходящие декларации
→ применить cascade
→ получить computed styles

Cascade — алгоритм выбора итогового значения, когда к одному элементу подходят несколько CSS-деклараций.

Computed styles — вычисленные стили элемента. Это уже не исходный CSS-текст, а итоговые значения свойств после применения правил, наследования и каскада.

Например, для элемента может получиться:

display: block
color: red
font-size: 32px
margin-block-start: 0.67em

Style не вычисляет координаты и размеры элементов. На этой стадии браузер отвечает на вопрос “какие стили применяются?”, а не “где элемент находится на экране?”.

Что происходит с CSSOM при изменении стилей

CSSOM — объектная модель CSS: представление таблиц стилей и CSS-правил в виде объектов.

Важно различать две разные операции:

изменить CSSOM
≠
пересчитать стили элементов

Если меняется класс элемента:

button.classList.add("active");

CSSOM обычно не строится заново. CSS-правила уже есть в CSSOM. Изменился DOM-элемент: у него появился новый class. Поэтому браузер заново проверяет, какие существующие CSS-правила теперь подходят этому элементу, и пересчитывает его computed styles. 

class изменился
→ CSSOM тот же
→ Style recalculation для затронутых элементов

Если меняется inline style:

button.style.opacity = "0.5";

Это обновляет inline style конкретного элемента. Браузеру снова нужно пересчитать computed styles для этого элемента, но это не означает пересборку всех stylesheet-ов документа. 

style="" изменился
→ inline declarations элемента изменились
→ Style recalculation

CSSOM обновляется, когда меняется сама таблица стилей или набор правил.

document.styleSheets[0].insertRule(".active { color: red }");

В этом случае меняется CSSStyleSheet: объект таблицы стилей получает новое правило. После такого изменения браузер должен учитывать обновлённые правила при следующем вычислении стилей. 

CSS rule добавлено / удалено / изменено
→ CSSOM обновился
→ Style recalculation

Поэтому в pipeline стадия называется Style, а не Build CSSOM: при обычном обновлении страницы браузер чаще использует уже существующий CSSOM и пересчитывает применимые стили.

3. Layout

Layout — стадия вычисления геометрии элементов: их размеров и положения.

После Style браузер знает, какие стили применяются. Но ему ещё нужно понять, какие boxes появляются на странице, какого они размера и где находятся. 

computed styles
+ размер viewport
+ правила layout
→ размеры и координаты boxes

Viewport — видимая область страницы в окне браузера.

Box — прямоугольная область, которую элемент занимает в layout.

Пример:

.container {
  width: 600px;
}

.item {
  width: 50%;
}

Чтобы вычислить ширину .item, браузеру нужно знать ширину родителя .container. Поэтому layout зависит от контекста: изменение одного элемента может повлиять на дочерние элементы, соседние блоки и переносы строк.

Свойства, которые часто могут затрагивать layout:

width
height
top
left
margin
padding
display
position

Повторный layout часто называют reflow. Это повторное вычисление размеров и положения после изменения страницы.

Пример:

box.style.width = "400px";

Такое изменение может потребовать нового layout, потому что изменилась геометрия элемента.

4. Paint

Paint — стадия подготовки рисования. После Layout браузер знает, где находятся элементы и какого они размера. Теперь нужно подготовить, что именно будет нарисовано.

Paint включает визуальные части:

текст
цвета
фоны
границы
тени
изображения
части form controls

Результат этой стадии можно представить как записи о том, что и в каком порядке нужно нарисовать. 

layout data
→ paint records

Изменения, которые меняют внешний вид, но не меняют геометрию, могут пропустить Layout, но потребовать Paint.

Примеры:

color
background-color
background-image
box-shadow

Если меняется цвет текста, браузеру не нужно заново рассчитывать положение элементов. Но затронутую область нужно заново нарисовать.

5. Composite

Composite — стадия сборки итогового кадра. Браузер может разделять страницу на слои, а затем собирать эти слои в правильном порядке. 

layers
→ compositing
→ final frame

Layer — отдельная поверхность, которую браузер может подготовить независимо от других частей страницы.

Compositing важен, когда элементы перекрываются или перемещаются.

Пример:

.modal {
  position: fixed;
  z-index: 100;
}

Модальное окно должно оказаться поверх основного контента. На стадии Composite браузер собирает подготовленные части изображения в итоговый кадр с правильным порядком наложения.

Composite не равен Paint.

Paint
→ подготовить, что нужно нарисовать

Composite
→ собрать подготовленные слои в итоговый кадр

В Chromium часть этой работы может выполнять compositor thread — отдельный поток, связанный со сборкой кадра.

6. Три типичных пути pipeline

Визуальное изменение может пройти по разным путям.

Полный путь

JavaScript / CSS
→ Style
→ Layout
→ Paint
→ Composite

Такой путь нужен, когда изменение влияет на геометрию.

Примеры:

width
height
left
top
margin
padding

Без Layout

JavaScript / CSS
→ Style
→ Paint
→ Composite

Такой путь возможен, когда изменение влияет на внешний вид, но не меняет размеры и положение.

Примеры:

color
background-color
box-shadow

Только Composite после Style

JavaScript / CSS
→ Style
→ Composite

Такой путь возможен, когда изменение не требует ни нового layout, ни нового paint.

Типичные свойства:

transform
opacity

Эти свойства часто используют для анимаций, потому что браузер может изменить способ сборки слоя без полного пересчёта геометрии и без перерисовки содержимого.

Но это не означает, что transform и opacity всегда бесплатны. Итоговая стоимость зависит от реализации браузера, количества слоёв, размера элемента и сложности страницы.

7. Кадры и jank

Экран обновляется с определённой частотой. Для 60 Hz это примерно 60 обновлений в секунду. 

1000 ms / 60 ≈ 16.67 ms на кадр

Части этого времени нужны самому браузеру, поэтому для плавной анимации полезно целиться примерно в 10 ms работы на кадр.

Jank — видимые рывки или пропуски кадров. Он появляется, когда браузер не успевает подготовить следующий кадр вовремя.

Причины могут быть разными:

долгий JavaScript
дорогой Style
дорогой Layout
большой Paint
слишком много layers
дорогие визуальные эффекты

Производительность интерфейса зависит не только от JavaScript, но и от того, какие стадии rendering pipeline запускает изменение.

8. Первичный render и обновления

Rendering pipeline используется и при первом показе страницы, и при последующих изменениях.

Первичный render:

HTML разобран
CSS разобран
DOM / CSSOM готовы
→ Style
→ Layout
→ Paint
→ Composite

Обновление страницы:

DOM или стили изменились
→ нужные стадии pipeline
→ новый кадр

Браузер старается выполнять только необходимую работу. Небольшое изменение не обязано пересчитывать всю страницу. Но некоторые изменения распространяются широко: например, изменение ширины контейнера может повлиять на дочерние элементы, переносы строк и положение соседних блоков.

9. Что важно не перепутать

Style ≠ CSS parsing

CSS parsing строит или обновляет CSSOM. Style вычисляет применимые стили для DOM-элементов.

CSSOM rebuild ≠ style recalculation

CSSOM меняется, когда меняются stylesheet-ы или CSS-правила. Style recalculation происходит, когда браузер заново вычисляет стили элементов.

class change ≠ CSSOM rebuild

Изменение класса обычно не меняет CSSOM. Оно меняет DOM-элемент, после чего браузер пересчитывает применимые стили.

Layout ≠ Paint

Layout вычисляет размеры и положение. Paint подготавливает рисование визуальных частей.

Paint ≠ Composite

Paint описывает, что нужно нарисовать. Composite собирает подготовленные части в итоговый кадр.

Composite-only ≠ бесплатно

Composite обычно дешевле layout и paint, но слои требуют памяти и управления.

DOM update ≠ полный pipeline всей страницы

Браузер может ограничить работу затронутыми частями, но масштаб пересчёта зависит от конкретного изменения.
