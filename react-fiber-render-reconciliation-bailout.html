React Fiber: фаза render, reconciliation и bailout

Коротко

Эта статья разбирает участок клиентского обновления React после планирования работы и до фазы commit:

Участок Смысл
Schedule update / priority / lane React уже получил обновление и выбрал работу для выполнения.
Render phase / Reconciliation React вычисляет следующий UI и сравнивает его с предыдущим деревом.
Bailout React может сократить работу для ветки, если её результат можно переиспользовать.
Commit phase Следующий этап: React применит подготовленные изменения к DOM.

Главная идея:

Fiber — внутренняя структура React, через которую React хранит дерево работы, строит новую версию UI, проходит по дереву вниз и вверх, отмечает нужные операции флагами и передаёт готовую работу в commit.

1. Что React называет render

В документации React rendering описывается как вызов компонентов для вычисления того, что должно быть показано на экране.

На начальном render React вызывает корневой компонент. При последующих обновлениях React вызывает компонент, чьё состояние изменилось, и затем рекурсивно обрабатывает компоненты, которые он возвращает.

Пример:


function App() {
  return (
    <main>
      <Header />
      <Counter />
    </main>
  );
}

При обработке такого дерева React должен получить результат для App, Header и Counter. Результатом работы компонента является описание UI: React elements, созданные из JSX.

Важно: фаза render не должна менять внешний мир. Компонент должен быть чистым: если props, состояние и context не изменились, результат должен быть тем же. Запросы, подписки, запись в DOM, аналитика и другие побочные действия не должны выполняться во время render.

2. Render React и отрисовка браузера — разные вещи

Слово “render” в React легко перепутать с браузерной отрисовкой. Это разные этапы.

Этап Кто выполняет Что происходит
Фаза render React React Вызывает компоненты, получает React elements, сравнивает деревья, готовит работу для commit.
Фаза commit React React DOM Применяет подготовленные изменения к DOM.
Визуальный конвейер браузера Браузер Пересчитывает стили, раскладку, рисует и композитит слои.

Если компонент “отрендерился”, это означает, что React вызвал его функцию и вычислил результат. Это ещё не означает, что DOM изменился или браузер уже нарисовал новые пиксели.

3. React element

React element — это объект JavaScript, который описывает желаемый UI.

JSX:


<h1 className="title">Hello</h1>

на уровне смысла описывает:


{
  type: 'h1',
  props: {
    className: 'title',
    children: 'Hello'
  }
}

Для пользовательского компонента:


<Greeting name="Ada" />

React element описывает тип компонента и props:


{
  type: Greeting,
  props: {
    name: 'Ada'
  }
}

React element — это описание. Он сам не хранит всю внутреннюю рабочую информацию React: состояние Hooks, связь с родителем, соседями, альтернативным деревом, lanes и флаги будущих операций. Для этой внутренней работы React использует Fiber.

4. Reconciliation как часть фазы render

После вызова компонента React получает новый результат — React elements. Этот результат описывает, каким должен быть UI после текущего обновления. Дальше React должен сопоставить это новое описание с уже существующим деревом.

Это сопоставление и называется reconciliation.

Во время reconciliation React не меняет DOM. Он работает с внутренним деревом React: сравнивает новые React elements с текущими Fiber-узлами, решает, какие части дерева можно переиспользовать, какие нужно создать заново, какие удалить и где нужно подготовить работу для commit.

Важный момент: новая рабочая версия Fiber-дерева не появляется до reconciliation как уже готовая структура. Она формируется во время reconciliation. React проходит по текущему дереву, получает новый результат render-а и постепенно строит work-in-progress tree.

На этом этапе React принимает несколько ключевых решений:

Что сравнивается Что решает React
Тип элемента Можно ли переиспользовать старый узел или нужно заменить ветку.
key Как сопоставить старых и новых детей в списке.
Позиция в дереве Сохраняется ли состояние компонента.
Props и текст Нужно ли подготовить обновление для commit.
Наличие работы в поддереве Нужно ли идти глубже или можно сделать bailout.

Поэтому reconciliation — это не отдельный этап после построения Fiber-дерева. Это работа, во время которой React строит рабочую версию Fiber-дерева на основе нового результата render-а и текущего дерева.

5. Fiber

Fiber — это внутренняя рабочая структура React. Приложение не создаёт Fiber-узлы вручную и не управляет ими через публичный API.

Удобная модель:

Понятие Роль
React element Публичное описание того, какой UI нужен.
Fiber Внутренний узел работы React для обработки этого описания.
DOM node Реальный узел браузерного DOM, который появляется или обновляется на commit.

Fiber-узел может связывать:

Поле / связь Смысл
тип Какой компонент или host-элемент обрабатывается: функция, класс, div, button и т.д.
props Входные данные текущего узла.
состояние Данные Hooks или class component state.
return Родительский Fiber-узел.
child Первый дочерний Fiber-узел.
sibling Следующий соседний Fiber-узел.
alternate Связь со второй версией того же узла: текущей или рабочей.
lanes Приоритеты/группы работы, связанные с этим узлом или его поддеревом.
flags Что нужно сделать с этим узлом на commit.
subtreeFlags Есть ли commit-работа глубже в поддереве.

Конкретные поля Fiber являются внутренней реализацией React. Они полезны для понимания механики, но не являются стабильным пользовательским API.

6. Зачем React нужен Fiber

Fiber нужен React для того, чтобы представлять rendering как работу над отдельными узлами дерева.

Без такой структуры React было бы сложнее:

  • приоритизировать разные обновления;
  • прерывать несрочную работу;
  • продолжать работу позже;
  • переиспользовать уже готовые части дерева;
  • хранить связь между предыдущей и новой версией UI;
  • собрать список будущих операций commit.

Публично это проявляется в возможностях React 18+ вроде concurrent rendering, transitions и прерываемой работы. Внутри это опирается на то, что React не просто рекурсивно вызывает компоненты “в один проход”, а работает с деревом Fiber-узлов.

7. Два дерева: текущее и рабочее

Во время обновления React держит две связанные версии дерева.

Дерево Что означает
current tree Текущее дерево. Оно уже прошло commit и соответствует UI, который пользователь сейчас видит.
work-in-progress tree Рабочее дерево. React строит его во время новой фазы render.

Связь между двумя версиями узла хранится через alternate.

Представим компонент:


function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

До клика текущее дерево можно представить так:

Дерево Состояние
current tree Counter(count: 0) и DOM-кнопка с текстом 0.

После клика React получает update setCount(1). Он не меняет текущее дерево сразу. Он создаёт или переиспользует рабочую версию:

Дерево Состояние
current tree Всё ещё соответствует экрану: Counter(count: 0).
work-in-progress tree Вычисляется новая версия: Counter(count: 1).

Пока рабочее дерево не прошло commit, пользователь продолжает видеть UI из текущего дерева.

После успешного commit рабочее дерево становится новым текущим деревом. Прежняя текущая версия остаётся связанной через alternate и может быть переиспользована для следующего обновления.

8. Почему это похоже на double buffering

Эту модель часто называют double buffering.

Смысл:

  1. У React есть committed-версия дерева, которая соответствует экрану.
  2. Для нового обновления React строит вторую версию дерева.
  3. Если работа завершилась успешно, React применяет изменения.
  4. После commit новая версия становится текущей.

Это помогает понять два важных свойства.

Во-первых, фаза render может быть прервана. Если React начал строить рабочее дерево, но пришло более срочное обновление, незавершённая работа может быть отложена или выброшена. Пользователь при этом продолжает видеть committed UI из текущего дерева.

Во-вторых, commit должен быть целостным. Пользователь не должен увидеть половину старого дерева и половину незавершённого рабочего дерева. Поэтому рабочая версия становится видимой только после commit.

9. Единица работы

Fiber-узел можно понимать как единицу работы.

Для каждого Fiber-узла React должен ответить на вопросы:

  • есть ли у этого узла обновления для текущих lanes;
  • нужно ли вызвать компонент;
  • можно ли переиспользовать старый результат;
  • какие дочерние узлы должны быть созданы, обновлены, удалены или переиспользованы;
  • нужно ли поставить флаги для commit;
  • есть ли работа глубже в поддереве.

Фаза render состоит из последовательной обработки таких единиц работы.

10. Спуск вниз и подъём вверх

Во время фазы render React обходит Fiber-дерево. Упрощённо обход состоит из двух частей:

Часть обхода Что происходит
Спуск вниз React начинает работу над узлом: проверяет входы, вызывает компонент или обрабатывает host-элемент, согласует детей.
Подъём вверх React завершает узел: подготавливает host-детали, собирает флаги поддерева и возвращается к родителю.

На уровне внутренних функций React этому соответствуют две большие идеи:

Внутренний этап Смысл
beginWork Начать обработку Fiber-узла и определить, к каким детям идти дальше.
completeWork Завершить Fiber-узел после обработки его детей и подготовить данные для commit.

Эти имена относятся к внутреннему коду React. В статье они используются как модель, а не как публичный API.

11. Reconciliation

Reconciliation — это центральная работа фазы render. React сравнивает новый результат render-а с текущим Fiber-деревом и формирует work-in-progress tree.

Во время reconciliation React работает не с HTML-строками и не с DOM напрямую. Он сравнивает React elements и Fiber-узлы.

React elements показывают, каким должен быть UI после текущего обновления. Текущие Fiber-узлы показывают, что уже было в дереве до этого обновления. Результатом сравнения становится рабочая версия Fiber-дерева и набор признаков будущей работы для commit.

Если React видит, что элемент того же типа находится в той же позиции, он может переиспользовать существующую структуру и сохранить состояние. Если тип или key изменились, React воспринимает это как другую ветку. Если изменились props, текст или состав детей, React не меняет DOM сразу, а помечает соответствующие Fiber-узлы для будущей commit-работы.

Для React DOM это означает: reconciliation вычисляет, что должно измениться в DOM, но сами DOM-операции выполняются позже, на commit phase.

12. Пример обхода дерева

Возьмём дерево:


function App() {
  return (
    <main>
      <Header />
      <Counter />
    </main>
  );
}

Упрощённое дерево:


App
  main
    Header
    Counter

Порядок работы можно представить таблицей:

Шаг Направление Узел Что делает React
1 Вниз App Начинает обработку App, вызывает компонент, получает main.
2 Вниз main Начинает обработку host-узла, переходит к детям.
3 Вниз Header Начинает обработку Header, получает его результат.
4 Вверх Header Завершает Header, возвращается к соседу.
5 Вниз Counter Начинает обработку Counter, применяет update состояния, получает новый JSX.
6 Вверх Counter Завершает Counter, поднимает информацию о работе выше.
7 Вверх main Завершает main, собирает сведения о дочерних изменениях.
8 Вверх App Завершает App, рабочее дерево готово к commit.

Такой обход объясняет, почему React может ставить флаги на конкретных узлах, а затем поднимать информацию о них к родителям через subtreeFlags.

13. Что происходит на спуске вниз

Во время спуска React начинает работу над Fiber-узлом.

Для function component это может включать:

  • применить pending updates к состоянию;
  • подготовить dispatcher Hooks;
  • вызвать функцию компонента;
  • получить React elements;
  • сравнить новых детей с текущими детьми;
  • создать, переиспользовать или пометить дочерние Fiber-узлы.

Пример:


function Counter() {
  const [count, setCount] = useState(0);

  return <button>{count}</button>;
}

Если пришёл update count: 1, на спуске для Counter React вызывает функцию компонента и получает:


<button>1</button>

Теперь React должен согласовать этот новый результат со старым деревом, где было:


<button>0</button>

Если тип button тот же, React переиспользует соответствующий Fiber-узел для кнопки и позже подготовит обновление текстового содержимого.

14. Что происходит на подъёме вверх

Когда дети узла обработаны, React поднимается обратно.

На подъёме React завершает Fiber-узел:

  • для host-узла может подготовить создание или обновление DOM-экземпляра;
  • сравнить старые и новые props host-узла;
  • отметить флаги, если нужны изменения;
  • собрать subtreeFlags из дочерних узлов;
  • передать информацию о работе родителю.

Пример host-узла:


<button disabled={false}>Save</button>

поменялся на:


<button disabled={true}>Save</button>

Во время reconciliation React понимает: это тот же button, но prop disabled изменился. DOM ещё не меняется. Рабочий Fiber получает признак того, что на commit нужно обновить DOM-свойство.

15. Флаги Fiber

Флаги — это внутренние признаки работы, которую нужно выполнить на commit.

Упрощённые смысловые категории:

Флаг Что означает по смыслу
Placement Нужно вставить новый узел или поддерево.
Update Нужно обновить существующий узел: props, текст, ref или другую host-деталь.
ChildDeletion Нужно удалить дочерний узел или поддерево.
Ref Нужно прикрепить или обновить ref.
Passive Есть пассивный effect, который будет обработан после commit.
Layout Есть layout-работа, связанная с commit.

Названия флагов относятся к внутренней реализации. В разных версиях React набор и детали флагов могут меняться. Для статьи важен принцип: render phase не выполняет DOM-операции, а помечает, что нужно сделать позже.

16. `flags` и `subtreeFlags`

У Fiber-узла есть два разных уровня признаков.

Поле Смысл
flags Работа, относящаяся к самому узлу.
subtreeFlags Признак, что работа есть где-то ниже в поддереве.

Пример:


App
  main
    Counter
      button

Если изменился текст внутри button, конкретная работа находится на узле кнопки или текстового потомка. Но commit-этапу нужно быстро понять, что в поддереве App вообще есть работа.

Для этого React на подъёме собирает сведения снизу вверх:

Узел Что происходит
button Получает флаг обновления.
Counter Получает информацию, что в его поддереве есть работа.
main Получает информацию, что в его поддереве есть работа.
App Получает информацию, что ниже есть commit-работа.

Идея subtreeFlags: commit phase может пропускать целые поддеревья, если в них нет флагов. Это уменьшает обход там, где нечего применять.

17. Пример: изменение текста

Компонент:


function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

Пользователь нажал кнопку. Было 0, стало 1.

Во время фазы render:

Шаг Что делает React
1 Берёт текущее дерево, где Counter соответствует значению 0.
2 Создаёт рабочую версию дерева.
3 На Fiber-узле Counter применяет update состояния.
4 Вызывает Counter() и получает JSX с текстом 1.
5 Сравнивает старый и новый результат. Тип button тот же.
6 Понимает, что текстовый потомок изменился.
7 Помечает рабочее дерево флагом будущего обновления.
8 Поднимает информацию о флаге вверх через subtreeFlags.

DOM-кнопка на этом этапе ещё не изменилась. Текст будет обновлён на commit phase.

18. Пример: вставка элемента

Компонент:


function List({ items }: { items: string[] }) {
  return (
    <ul>
      {items.map(item => (
        <li key={item}>{item}</li>
      ))}
    </ul>
  );
}

Было:


['A', 'B']

Стало:


['A', 'B', 'C']

Во время reconciliation React сопоставляет элементы по key.

Ключ Старое дерево Новое дерево Решение
A Есть Есть Переиспользовать.
B Есть Есть Переиспользовать.
C Нет Есть Создать новый Fiber и пометить на вставку.

Для C появится работа на commit: нужно вставить новый DOM-узел li.

19. Пример: удаление элемента

Было:


['A', 'B', 'C']

Стало:


['A', 'C']

React снова сопоставляет элементы по key.

Ключ Старое дерево Новое дерево Решение
A Есть Есть Переиспользовать.
B Есть Нет Пометить на удаление.
C Есть Есть Переиспользовать.

Удаление также не выполняется во время render. React только отмечает, что на commit нужно удалить соответствующий узел или поддерево.

20. Пример: изменение типа

Было:


<div className="card" />

Стало:


<section className="card" />

Тип host-элемента изменился: было div, стало section.

React не может просто изменить тип существующего DOM-узла. Он должен заменить ветку:

Старое Новое Решение
div section Удалить старую ветку и создать новую.

Для пользовательских компонентов правило похожее:


<Profile />

заменяется на:


<Settings />

Тип компонента изменился. React воспринимает это как другой компонент в этой позиции. State старого компонента не сохраняется.

21. Reconciliation работает с деревьями, а не HTML-строками

React не сравнивает готовые HTML-строки.

Он сравнивает результат render-а: React elements и соответствующие Fiber-узлы.

Пример:


function App({ name }: { name: string }) {
  return <h1>Hello, {name}</h1>;
}

Было:


<h1>Hello, Ada</h1>

Стало:


<h1>Hello, Grace</h1>

React видит:

Часть Результат сравнения
Тип h1 Не изменился.
Props h1 Не изменились или изменились частично.
Текстовый потомок Изменился.

Значит, не нужно пересоздавать весь h1; достаточно подготовить обновление текста.

22. Позиция компонента и сохранение состояния

React связывает состояние с позицией компонента в дереве.

Пример:


function App() {
  return (
    <div>
      <Counter />
      <Counter />
    </div>
  );
}

Это два независимых Counter. Они имеют одинаковый тип, но разные позиции, поэтому состояние у них разное.

Если компонент остаётся в той же позиции и имеет тот же тип, состояние обычно сохраняется.


function App({ show }: { show: boolean }) {
  return (
    <div>
      {show ? <Counter /> : <Counter />}
    </div>
  );
}

Хотя JSX написан в двух ветках, на этой позиции в дереве всё равно оказывается Counter. Для React это тот же тип в той же позиции.

Если структура меняется, состояние может сброситься:


function App({ wrap }: { wrap: boolean }) {
  return wrap ? (
    <section>
      <Counter />
    </section>
  ) : (
    <Counter />
  );
}

Теперь Counter находится в другой позиции дерева. Для React это может быть уже другая ветка.

23. `key` и идентичность

key помогает React сопоставлять элементы между старым и новым деревом.

Правильно:


function TodoList({ todos }: { todos: Todo[] }) {
  return (
    <ul>
      {todos.map(todo => (
        <li key={todo.id}>{todo.text}</li>
      ))}
    </ul>
  );
}

key должен быть стабильным и связанным с идентичностью данных.

Плохо:


{todos.map(todo => (
  <li key={Math.random()}>{todo.text}</li>
))}

Такой ключ меняется при каждом render. React будет считать элементы новыми, из-за чего возможны лишние удаления/вставки и сброс состояния.

Индекс массива как key допустим только для простых статичных списков. Если элементы добавляются, удаляются или переставляются, индекс может привести к неправильному сохранению состояния между элементами.

24. Intentional reset через `key`

Изменение key можно использовать намеренно, чтобы сбросить состояние.


function ProfilePage({ userId }: { userId: string }) {
  return <ProfileForm key={userId} userId={userId} />;
}

Когда userId меняется, меняется и key. React воспринимает это как новый ProfileForm, создаёт его заново и сбрасывает внутреннее состояние формы.

Это не обычный re-render. Это remount: старая ветка удаляется, новая создаётся.

25. Bailout

Bailout — это сокращение работы. React может понять, что для части дерева результат можно переиспользовать, и не идти глубже в эту ветку.

Bailout может происходить на разных уровнях:

Ситуация Почему возможен bailout
Setter получил то же значение состояния Новое значение совпадает с текущим через Object.is.
Компонент обёрнут в memo, props не изменились React может переиспользовать результат компонента.
Поддерево не содержит работы для текущих lanes React может не заходить глубже при этом проходе.
JSX-узел был закэширован через useMemo React видит тот же неизменяемый объект JSX.

Bailout не означает, что update “исчез”. React мог получить update, проверить входы и решить, что конкретную ветку можно переиспользовать.

26. Bailout при одинаковом state

Пример:


function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(0)}>
      {count}
    </button>
  );
}

Если count уже равен 0, React получает update с тем же значением. React использует сравнение Object.is и может пропустить дальнейшую работу для этой ветки.

С объектами нужно быть внимательным:


const [person, setPerson] = useState({ name: 'Ada' });

setPerson(person);

Здесь ссылка та же. React видит то же значение.

А здесь ссылка новая:


setPerson({ name: 'Ada' });

Даже если содержимое такое же, объект новый, поэтому для React это новое значение.

27. `memo` как способ помочь bailout

memo позволяет React пропустить повторный вызов компонента, если родитель re-render-ился, но props компонента не изменились.


import { memo } from 'react';

const Greeting = memo(function Greeting({ name }: { name: string }) {
  return <h1>Hello, {name}</h1>;
});

Если родитель re-render-ится, но name остался тем же, React обычно может переиспользовать прежний результат Greeting.

Ограничения:

  • memo не блокирует обновления собственного состояния компонента;
  • memo не блокирует изменение context, который компонент сам читает;
  • memo бесполезен, если props каждый раз новые по ссылке;
  • memo — оптимизация, а не гарантия.

28. Почему `memo` может не сработать

Частая причина — новый объект или новая функция в props.


const Chart = memo(function Chart({ options }: { options: Options }) {
  return <div>{options.theme}</div>;
});

function Page() {
  return <Chart options={{ theme: 'dark' }} />;
}

На каждом render Page создаёт новый объект { theme: 'dark' }. Для memo это новые props.

Варианты исправления:


const options = useMemo(() => ({ theme: 'dark' }), []);

return <Chart options={options} />;

или проще:


return <Chart theme="dark" />;

С функциями похожая ситуация:


<ShippingForm onSubmit={() => post(productId)} />

Каждый render создаёт новую функцию. Если ShippingForm обёрнут в memo, стабильная ссылка может потребовать useCallback.

29. `useMemo` и `useCallback` в контексте bailout

useMemo кэширует результат вычисления между render-ами:


const visibleTodos = useMemo(
  () => filterTodos(todos, tab),
  [todos, tab]
);

Он помогает, когда:

  • вычисление дорогое;
  • результат передаётся в компонент, обёрнутый в memo;
  • нужна стабильная ссылка для массива или объекта.

useCallback кэширует функцию:


const handleSubmit = useCallback((orderDetails: OrderDetails) => {
  post('/buy', { productId, orderDetails });
}, [productId]);

Он помогает, когда функция передаётся в компонент, обёрнутый в memo, или используется как зависимость другого Hook.

Важно:

Hook Что делает Чего не делает
useMemo Кэширует значение. Не запрещает re-render компонента.
useCallback Кэширует функцию. Не ускоряет код автоматически.

30. Context и bailout

Если компонент читает context через useContext, изменение значения ближайшего Provider вызывает re-render этого компонента.


const ThemeContext = createContext('light');

const Button = memo(function Button() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Save</button>;
});

Даже если Button обёрнут в memo, изменение ThemeContext важно для результата Button, поэтому React должен вызвать его снова.

Практические приёмы:

  • разделять contexts по независимым данным;
  • мемоизировать объект value у Provider;
  • читать context во внешнем компоненте и передавать в тяжёлый дочерний компонент минимальные props.

31. Hooks во время render

Hooks вроде useState, useReducer, useMemo, useCallback, useContext вызываются во время render. Поэтому их нужно вызывать в одном и том же порядке на каждом render.

Правильно:


function Component() {
  const [count, setCount] = useState(0);
  const value = useMemo(() => expensive(count), [count]);

  return <p>{value}</p>;
}

Неправильно:


function Component({ enabled }: { enabled: boolean }) {
  if (enabled) {
    const [count, setCount] = useState(0);
  }

  return null;
}

React связывает данные Hooks с компонентом между render-ами по порядку вызовов. Если порядок меняется, сопоставление ломается.

32. `useMemo` должен быть чистым

useMemo выполняет вычисление во время render. Поэтому его функция должна быть чистой.

Плохо:


const value = useMemo(() => {
  localStorage.setItem('x', '1');
  return compute();
}, []);

Это побочное действие во время render.

Правильно:


const value = useMemo(() => compute(input), [input]);

useMemo нужен для чистой калькуляции или стабильной ссылки. Синхронизация с внешними системами относится к Effects.

33. Strict Mode и повторные вызовы

В development <StrictMode> может вызвать компонентную функцию дополнительный раз, чтобы обнаружить нечистый render.

Плохой пример:


function StoryTray({ stories }: { stories: Story[] }) {
  stories.push({ id: 'create', label: 'Create Story' });

  return (
    <ul>
      {stories.map(story => (
        <li key={story.id}>{story.label}</li>
      ))}
    </ul>
  );
}

Компонент мутирует props. Повторный вызов render делает ошибку заметнее.

Правильный вариант:


function StoryTray({ stories }: { stories: Story[] }) {
  const items = [
    ...stories,
    { id: 'create', label: 'Create Story' },
  ];

  return (
    <ul>
      {items.map(story => (
        <li key={story.id}>{story.label}</li>
      ))}
    </ul>
  );
}

Strict Mode — проверка режима разработки. Не нужно оптимизировать приложение только потому, что в development компонент вызывается дополнительный раз.

34. Как фаза render подготавливает commit

К концу фазы render React получает завершённое рабочее дерево.

В нём есть:

  • новые props и состояние;
  • переиспользованные или созданные Fiber-узлы;
  • связь с текущим деревом через alternate;
  • флаги на узлах, где нужна commit-работа;
  • subtreeFlags, которые показывают, где внутри поддеревьев есть работа.

Если изменений нет, React может не выполнять DOM-мутации. Если изменения есть, рабочее дерево передаётся в commit phase.

Commit phase использует подготовленную информацию и применяет изменения к DOM. Именно там будут вставки, удаления, обновления DOM-свойств, прикрепление refs и запуск commit-related Effects.

35. Важное ограничение: render может быть прерван

В concurrent rendering React может начать работу над рабочим деревом, приостановить её, продолжить позже или выбросить незавершённый результат.

Поэтому render должен быть чистым.

Плохо:


function Product({ product }: { product: Product }) {
  analytics.track('render product');
  return <h1>{product.name}</h1>;
}

Если React начнёт render и потом выбросит эту работу, аналитика уже будет отправлена, хотя UI не был committed.

Правильно:


function Product({ product }: { product: Product }) {
  useEffect(() => {
    analytics.track('view product');
  }, [product.id]);

  return <h1>{product.name}</h1>;
}

Побочные действия относятся к event handlers, Effects или commit-related API, а не к render.

36. Bailout и прерванная работа — не одно и то же

Понятие Смысл
Bailout React понял, что ветку можно переиспользовать и не пересчитывать глубже.
Прерванная работа React начал render, но остановил или отложил его из-за планирования.

Bailout связан с отсутствием нужной работы или неизменившимися входами. Прерывание связано с приоритетами и возможностью React уступать более срочным обновлениям.

37. Почему вопрос “Есть changes?” — упрощение

В диаграмме может быть блок:


Render / Reconciliation
Есть changes?

Это полезное учебное упрощение, но в реальности React не задаёт один глобальный вопрос для всего приложения. Он принимает такие решения локально:

Где Что может произойти
На уровне state update Новое значение совпало со старым — дальнейшая работа может быть пропущена.
На уровне memo Props не изменились — компонент можно не вызывать.
На уровне детей Старые и новые children сопоставлены, часть переиспользована.
На уровне поддерева Нет работы для текущих lanes — можно не идти глубже.
На уровне флагов Если в ветке нет flags/subtreeFlags, commit может пропустить её.

Лучше понимать этот блок как набор локальных решений: где-то React продолжает работу, где-то переиспользует старое, где-то ставит флаги для commit.

38. Пример полного прохода

Компонент:


function App() {
  return (
    <main>
      <Header />
      <Counter />
    </main>
  );
}

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

Пользователь нажал кнопку.

Этап Что происходит
1 setCount ставит update в очередь.
2 Scheduler выбирает работу для выполнения.
3 React создаёт рабочее дерево из текущего дерева.
4 На спуске доходит до Counter.
5 React применяет update состояния и вызывает Counter().
6 Counter() возвращает button с новым текстом.
7 React сравнивает старый и новый результат.
8 Тип button тот же, но текст изменился.
9 Fiber-узел получает флаг будущего обновления.
10 На подъёме React поднимает сведения о флагах через subtreeFlags.
11 Рабочее дерево завершено и готово к commit.
12 На commit React обновит текст в DOM.

Главное: пункты 3–11 относятся к фазе render. DOM меняется только в пункте 12.

39. Ошибки мышления

“Fiber — это virtual DOM”

Нет. React elements описывают желаемый UI. Fiber — внутренняя рабочая структура React для обработки этого UI.

“React сравнивает DOM с DOM”

Нет. React сравнивает новый результат render с предыдущим деревом React/Fiber. DOM применяется позже на commit.

“Re-render всегда меняет DOM”

Нет. Компонент может быть вызван снова, но результат может совпасть с прошлым. Тогда DOM-мутация не нужна.

“Bailout означает, что update исчез”

Нет. React мог получить update, проверить входы и решить, что конкретную ветку можно переиспользовать.

“Флаги — это DOM-операции”

Нет. Флаги — это признаки будущей commit-работы. Сами DOM-операции выполняются позже.

“`subtreeFlags` нужны для render”

Нет. Их смысл — помочь следующим этапам быстро понимать, где в поддереве есть commit-работа.

“`memo` запрещает re-render навсегда”

Нет. memo — оптимизация. Собственное состояние, context и изменившиеся props всё равно могут привести к re-render.

“Render можно использовать для аналитики”

Нет. Render должен быть чистым. Иначе прерываемый render и Strict Mode будут выявлять проблемы.

40. Что важно не перепутать

Не путать Правильное различие
React render и браузерную отрисовку React вызывает компоненты; браузер позже рисует пиксели.
Фазу render и фазу commit Render вычисляет; commit применяет.
React element и Fiber Element — описание UI; Fiber — внутренняя рабочая структура.
Fiber и DOM node Fiber хранит работу React; DOM node живёт в браузере.
current tree и work-in-progress tree Current уже соответствует экрану; work-in-progress только вычисляется.
Reconciliation и DOM mutation Reconciliation вычисляет изменения; DOM mutation выполняется на commit.
flags и subtreeFlags flags относятся к узлу; subtreeFlags показывают работу ниже.
Bailout и отсутствие update Update мог быть обработан; React просто переиспользовал результат ветки.
Изменение key и обычный re-render Новый key может привести к remount и сбросу состояния.
useMemo и запрет render useMemo кэширует значение, но не запрещает вызов компонента.

Дальше начинается render phase. У React уже есть current tree: Fiber-дерево, которое соответствует текущему UI. Для нового обновления React начинает строить work-in-progress tree.

Рабочее дерево формируется во время reconciliation. React проходит по Fiber-узлам, вызывает компоненты там, где это требуется, получает новые React elements и сопоставляет их с текущими Fiber-узлами.

Если новый результат соответствует старой ветке по типу, позиции и key, React может переиспользовать узлы и сохранить состояние. Если соответствия нет, React создаёт новую ветку и помечает старую на удаление. Если изменились props, текст или состав детей, React ставит флаги будущей commit-работы. Если для ветки нет нужной работы, React может сделать bailout и не идти глубже без необходимости.

К концу render phase work-in-progress tree содержит следующую рабочую версию UI и информацию о том, что нужно применить на commit. DOM на этом этапе ещё не меняется.

41. Итоговая модель

Фаза render в React — это построение и проверка рабочей версии дерева.

React берёт текущее committed-дерево, создаёт рабочее дерево, проходит по нему вниз и вверх, вызывает компоненты, согласует детей, переиспользует подходящие старые Fiber-узлы, создаёт новые там, где нужно, помечает будущую commit-работу флагами и собирает сведения о флагах поддеревьев.

Если React понимает, что ветку можно переиспользовать, он делает bailout и не идёт глубже без необходимости. Если изменения есть, они не применяются сразу: рабочее дерево только получает признаки будущей работы. DOM будет изменён на следующем этапе — commit phase.
