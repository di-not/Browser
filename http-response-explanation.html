HTTP Response — ответ

Коротко о блоке

HTTP Response — это сообщение, которое сервер возвращает клиенту в ответ на HTTP Request.

В общей схеме:

HTTP Request — запрос
→ HTTP Response — ответ
→ Что вернул сервер?

Response определяет дальнейшую ветку браузера:

Redirect 3xx
→ Перейти по Location

Ошибка / сбой сети
→ Страница ошибки браузера или error document

JSON / API / ресурс
→ Передать ответ инициатору запроса

HTML document / app shell
→ Получен HTML document / app shell

HTTP Response — это не обязательно HTML-страница. Это может быть HTML, CSS, JavaScript, JSON, изображение, шрифт, redirect, ошибка, 304 для кэша или ответ без body.

1. Из чего состоит HTTP Response

Логически HTTP Response содержит:

status code
status text / reason phrase
headers
optional body

Пример HTTP/1.1 response:

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Cache-Control: max-age=0
Content-Encoding: br
Set-Cookie: session=abc; HttpOnly; Secure


<html>...</html>

Упрощённая структура:

status line
headers
empty line
optional body

В HTTP/2 и HTTP/3 ответ передаётся не как текстовая строка HTTP/1.1 200 OK, а через фреймы и pseudo-header: 

:status: 200
content-type: text/html; charset=utf-8
cache-control: max-age=0

Семантика остаётся той же:

status
headers
body

2. Status code

Status code — трёхзначный код, который сообщает результат обработки запроса.

Классы кодов:

1xx → informational — информационные ответы
2xx → successful — успешные ответы
3xx → redirection — перенаправления
4xx → client error — ошибка на стороне клиента / запроса
5xx → server error — ошибка на стороне сервера

Важно:

status code сам по себе не говорит, что body является HTML.
Для понимания типа данных важен Content-Type.

Например:

HTTP/1.1 200 OK
Content-Type: application/json

Это успешный ответ, но не HTML-документ.

3. Частые status codes

1xx — информационные ответы

100 Continue

Сервер сообщает клиенту, что начальная часть запроса получена и можно продолжать отправку body.

101 Switching Protocols

Сервер соглашается переключить протокол. Классический пример — переход с HTTP на WebSocket через механизм Upgrade.

103 Early Hints

Сервер заранее отправляет подсказки, обычно через Link headers, чтобы браузер мог раньше начать загрузку важных ресурсов.

2xx — успешные ответы

200 OK

Запрос успешно обработан. Для обычной навигации часто означает, что сервер вернул HTML-документ, но 200 OK может вернуть и JSON, CSS, JS, изображение или другой ресурс. Тип определяется Content-Type.

201 Created

Запрос успешно выполнен, и на сервере был создан новый ресурс.

202 Accepted

Запрос принят, но обработка ещё не завершена. Подходит для асинхронных операций.

204 No Content

Запрос успешно обработан, но body ответа нет.

206 Partial Content

Сервер вернул только часть ресурса. Используется с range requests.

3xx — перенаправления

301 Moved Permanently

Ресурс окончательно перенесён на новый URL. Сервер обычно отправляет Location, а браузер переходит по этому адресу.

302 Found

Временное перенаправление.

303 See Other

Сервер говорит клиенту получить результат по другому URL, обычно методом GET.

304 Not Modified

Ресурс не изменился с момента последней сохранённой версии. У ответа нет нового body ресурса: браузер использует body из HTTP Cache.

307 Temporary Redirect

Временный redirect, при котором клиент должен сохранить исходный HTTP method и body.

308 Permanent Redirect

Постоянный redirect, при котором клиент тоже должен сохранить method и body.

4xx — ошибки клиента / запроса

400 Bad Request

Сервер не может обработать запрос из-за ошибки в самом запросе.

401 Unauthorized

Запрос требует аутентификации.

403 Forbidden

Сервер понял запрос, но отказывается его выполнять.

404 Not Found

Сервер не нашёл ресурс по указанному URL. Важно: 404 — это HTTP Response, а не отсутствие ответа.

405 Method Not Allowed

Метод запроса не разрешён для этого ресурса.

408 Request Timeout

Сервер не дождался полного запроса от клиента за допустимое время.

409 Conflict

Запрос конфликтует с текущим состоянием ресурса.

410 Gone

Ресурс раньше существовал, но теперь удалён и больше не доступен.

415 Unsupported Media Type

Сервер не поддерживает формат body, который отправил клиент.

422 Unprocessable Content

Сервер понял формат запроса, но не может обработать данные по смыслу.

429 Too Many Requests

Клиент отправил слишком много запросов за короткое время.

5xx — ошибки сервера

500 Internal Server Error

Общая ошибка сервера.

501 Not Implemented

Сервер не поддерживает функциональность, нужную для обработки запроса.

502 Bad Gateway

Gateway или proxy получил некорректный ответ от upstream server.

503 Service Unavailable

Сервис временно недоступен.

504 Gateway Timeout

Gateway или proxy не дождался ответа от upstream server.

4. Response headers

Response headers — метаданные ответа. Они помогают браузеру понять, как обрабатывать response.

Примеры:

Content-Type: text/html; charset=utf-8
Content-Length: 10240
Content-Encoding: br
Cache-Control: max-age=3600
ETag: "abc123"
Location: /login
Set-Cookie: session=abc; HttpOnly; Secure

Content-Type

Content-Type указывает media type возвращаемых данных. 

text/html              → HTML document / app shell
application/json       → данные для fetch/API
text/css               → stylesheet
application/javascript → script
image/png              → image resource
font/woff2             → font resource

Content-Length

Content-Length указывает размер body в байтах, если он известен.

Content-Encoding

Content-Encoding указывает, как сжато body, например br или gzip.

Cache-Control

Cache-Control задаёт правила кэширования response.

ETag

ETag — идентификатор версии ресурса. Используется для revalidation.

Location

Location указывает URL для redirect.

Set-Cookie

Set-Cookie — header, через который сервер просит браузер сохранить cookie.

5. Body ответа

Body — полезная нагрузка HTTP Response.

Примеры body:

HTML document
CSS stylesheet
JavaScript file
JSON data
image bytes
font bytes
empty body

Body может отсутствовать, например:

204 No Content
304 Not Modified
response на HEAD request

6. Что вернул сервер? Четыре основные ветки

После получения HTTP Response браузер или инициатор запроса классифицирует результат. 

HTTP Response
→ Что вернул сервер?
   ├─ Redirect 3xx
   │  → Перейти по Location
   │
   ├─ Ошибка / сбой сети
   │  → Страница ошибки браузера или error document
   │
   ├─ JSON / API / ресурс
   │  → Передать ответ инициатору запроса
   │
   └─ HTML document / app shell
      → Получен HTML document / app shell

6.1. Перейти по Location

Эта ветка срабатывает, когда сервер вернул redirect status code: 301, 302, 303, 307, 308.

Обычно response содержит header:

Location: https://example.com/new-url

Логика:

сервер ответил redirect
→ браузер читает Location
→ строит новый URL
→ снова проходит URL parsing
→ снова может быть cache / DNS / connection / request / response

Важно:

redirect — это не рендер страницы,
а команда начать новый переход.

Также важно различать 302/303 и 307/308: 307 и 308 сохраняют исходный method и body при redirect, а 303 обычно ведёт к последующему GET.

6.2. Страница ошибки браузера или error document

Эта ветка покрывает два разных случая.

HTTP error document

Сервер вернул HTTP Response со статусом ошибки: обычно 4xx или 5xx.

Пример:

HTTP/1.1 404 Not Found
Content-Type: text/html

<html>Страница не найдена</html>

Это нормальный HTTP Response. Если body является HTML, браузер может отрендерить страницу ошибки, которую прислал сервер.

Примеры:

404 Not Found → серверная страница "не найдено"
500 Internal Server Error → серверная страница ошибки
503 Service Unavailable → страница временной недоступности

Browser error page / network error

Иногда нормального HTTP Response вообще нет.

Примеры:

DNS failure
connection refused
TLS certificate error
timeout
connection reset
network offline

В этом случае серверный HTML не получен. Браузер показывает собственную страницу ошибки.

Разница:

404 / 500 → HTTP Response есть
DNS/TLS/timeout error → нормального HTTP Response нет

6.3. Передать ответ инициатору запроса

Эта ветка используется, когда response не является основным HTML-документом навигации, а был запрошен конкретным механизмом.

Инициатором может быть:

fetch()
XMLHttpRequest
script loader
CSS loader
image loader
font loader
preload scanner
Service Worker

Примеры:

fetch("/api/users") → response передаётся JavaScript-коду
<script src="/app.js"> → response передаётся JS engine / script loader
<link rel="stylesheet" href="/style.css"> → response передаётся CSS loader
<img src="/logo.png"> → response передаётся image decoder

Пример JSON response:

HTTP/1.1 200 OK
Content-Type: application/json

{"items":[]}

Браузер не строит DOM из такого response. Он передаёт данные механизму, который сделал запрос.

Для fetch() JavaScript может обработать response так:

const response = await fetch("/api/users");
const data = await response.json();

6.4. Получен HTML document / app shell

Эта ветка срабатывает, когда response является HTML-документом для навигации.

Типичный response:

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8


<html>
  ...
</html>

После этого браузер переходит к document pipeline:

HTML_RECEIVED
→ HTML Parser
→ DOM
→ preload scanner
→ загрузка CSS/JS/images/fonts
→ CSSOM
→ выполнение scripts
→ rendering pipeline

HTML document

Обычный HTML document может содержать уже готовую разметку.

App shell

App shell — минимальная HTML-оболочка приложения.

Например, для React SPA:


<html>
  <body>
    <div id="root"></div>
    <script src="/app.js"></script>
  </body>
</html>

Сам по себе app shell может содержать мало контента. Основной интерфейс появится после загрузки и выполнения JavaScript bundle.

Next.js / SSR / streaming

Для Next.js response может быть:

server-rendered HTML
static generated HTML
HTML + __NEXT_DATA__
streaming HTML
app shell + RSC payload

Но для браузера на этом этапе главное:

получен response с HTML
→ можно начинать HTML parsing

7. Что важно не перепутать

HTTP error не равен network error

404 / 500 → HTTP Response есть
DNS failure / TLS error / timeout → нормального HTTP Response нет

200 OK не означает HTML

HTTP/1.1 200 OK
Content-Type: application/json

Это успешный ответ, но не HTML-документ.

304 не содержит новый body

304 Not Modified говорит браузеру использовать body из HTTP Cache.

Redirect запускает новый проход

Redirect не завершает загрузку страницы. Он создаёт новый переход по Location.

Content-Type определяет интерпретацию body

Расширение URL менее важно, чем headers и контекст запроса.

8. Мини-глоссарий

HTTP Response

Сообщение сервера клиенту в ответ на HTTP Request.

Status code

Трёхзначный код результата запроса.

1xx

Информационные ответы.

2xx

Успешные ответы.

3xx

Перенаправления.

4xx

Ошибки клиента или запроса.

5xx

Ошибки сервера.

Header

Метаданные HTTP-сообщения.

Body

Полезная нагрузка ответа.

Content-Type

Header, который указывает media type возвращаемого ресурса.

Content-Encoding

Header, который указывает способ сжатия body.

Cache-Control

Header, задающий правила кэширования.

ETag

Идентификатор версии ресурса.

Location

Header, указывающий URL для redirect.

Set-Cookie

Header, через который сервер устанавливает cookie.

304 Not Modified

Status, означающий, что ресурс не изменился и можно использовать cached body.

Error document

HTML или другой документ ошибки, который сервер вернул как HTTP Response.

Browser error page

Страница ошибки, которую показывает сам браузер, когда нормальный HTTP Response не получен.

App shell

Минимальная HTML-оболочка приложения, которая загружает клиентский JavaScript.
