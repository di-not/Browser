Ответ из Service Worker / Cache Storage / fallback в сеть

Коротко о блоке

Блок «Ответ из Service Worker / Cache Storage / fallback в сеть» описывает ситуацию, когда браузер не сразу идёт напрямую в сеть за ресурсом, а сначала передаёт запрос в Service Worker.

Service Worker может решить, откуда взять ответ:

Request
→ Service Worker
   ├─ вернуть Response из Cache Storage
   ├─ сходить в сеть через fetch()
   ├─ сначала попробовать кэш, потом сеть
   ├─ сначала попробовать сеть, потом кэш
   └─ вернуть fallback Response

Service Worker — это программируемый слой между страницей и сетью. Он может вернуть ответ из кэша, из сети или подготовить запасной ответ, если сеть недоступна.

1. Что такое Service Worker

Service Worker — это фоновый JavaScript worker, который браузер запускает отдельно от страницы.

Он не имеет доступа к DOM, но может перехватывать запросы страницы через fetch event. 

страница делает запрос
→ браузер видит, что страницу контролирует Service Worker
→ запрос попадает в Service Worker
→ Service Worker решает, какой Response вернуть

Service Worker часто используется для:

  • офлайн-режима;
  • кэширования ресурсов;
  • ускорения повторных загрузок;
  • fallback-страниц;
  • PWA-приложений.

Важно: Service Worker работает не всегда. Он должен быть зарегистрирован, установлен, активирован и должен контролировать страницу в пределах своего scope.

2. Что такое scope

Scope — это область URL, которую контролирует Service Worker.

Например:

navigator.serviceWorker.register("/sw.js");

Если Service Worker зарегистрирован из корня сайта, он обычно может контролировать весь сайт: 

/

Если он зарегистрирован из вложенного пути:

navigator.serviceWorker.register("/app/sw.js");

то область контроля обычно будет:

/app/

Это значит, что запросы вне этой области не обязаны проходить через этот Service Worker.

3. Что такое fetch event

fetch event — событие, которое возникает в Service Worker, когда контролируемая страница делает запрос.

Это может быть не только явный fetch() из JavaScript, но и обычные запросы браузера: 

HTML document
CSS
JavaScript
images
fonts
API requests

Пример:

self.addEventListener("fetch", (event) => {
  event.respondWith(fetch(event.request));
});

Здесь Service Worker перехватывает запрос, но просто передаёт его в сеть.

4. Что такое respondWith()

respondWith() — это метод, через который Service Worker говорит браузеру: 

Для этого Request я сам предоставлю Response.

Пример:

self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request)
  );
});

Но такой код опасен как полноценная стратегия: если в кэше ничего нет, ответ будет undefined, и запрос завершится ошибкой.

Обычно добавляют fallback:

self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cachedResponse) => {
      return cachedResponse || fetch(event.request);
    })
  );
});
проверить Cache Storage
→ если Response найден, вернуть его
→ если не найден, выполнить запрос в сеть

5. Что такое Cache Storage

Cache Storage — это браузерное хранилище, доступное через Cache API.

Оно хранит пары:

Request → Response

Пример:

const cache = await caches.open("app-cache-v1");
await cache.add("/styles.css");

const response = await cache.match("/styles.css");

Внутри это можно представить так:

CacheStorage
└─ app-cache-v1
   ├─ /index.html → Response
   ├─ /styles.css → Response
   └─ /app.js     → Response

Service Worker часто использует Cache Storage, чтобы сохранить важные ресурсы и вернуть их позже без сети.

6. Cache Storage — это не HTTP Cache

Важно не путать два разных кэша.

HTTP Cache

HTTP Cache управляется браузером автоматически по HTTP-заголовкам:

Cache-Control
ETag
Last-Modified
Expires
Vary

Приложение влияет на него через заголовки ответа сервера.

Cache Storage

Cache Storage управляется JavaScript-кодом:

cache.put(request, response)
cache.match(request)
cache.delete(request)

Разработчик сам пишет стратегию:

что положить в кэш
когда читать из кэша
когда идти в сеть
когда обновлять кэш
что показывать офлайн
HTTP Cache    → автоматический кэш браузера
Cache Storage → программируемый кэш для Service Worker / Cache API

7. Основные варианты ответа Service Worker

Вариант 1. Ответ из Cache Storage

Service Worker может вернуть уже сохранённый response.

Request
→ Service Worker
→ Cache Storage
→ cached Response
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request)
  );
});

Такой вариант быстрый и может работать офлайн. Но если ресурса нет в кэше, нужен запасной путь.

Вариант 2. Cache-first: сначала кэш, потом сеть

Cache-first — стратегия, при которой Service Worker сначала ищет ответ в Cache Storage, а в сеть идёт только при промахе. 

Request
→ Cache Storage
   ├─ cache hit  → вернуть cached Response
   └─ cache miss → fetch в сеть
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cachedResponse) => {
      return cachedResponse || fetch(event.request);
    })
  );
});

Подходит для:

CSS
JS bundles
fonts
icons
static images

Плюс: быстро. Минус: можно отдать устаревшую версию, если кэш не обновляется.

Вариант 3. Network-first: сначала сеть, потом кэш

Network-first — стратегия, при которой Service Worker сначала пытается получить свежий ответ из сети, а если сеть недоступна, возвращает кэш. 

Request
→ fetch в сеть
   ├─ успех  → вернуть свежий Response
   └─ ошибка → вернуть cached Response
self.addEventListener("fetch", (event) => {
  event.respondWith(
    fetch(event.request).catch(() => {
      return caches.match(event.request);
    })
  );
});

Подходит для:

HTML pages
API data
данные, где свежесть важнее скорости

Плюс: если сеть есть, пользователь получает свежий ответ. Минус: при медленной сети ответ может быть медленнее.

Вариант 4. Fallback response

Fallback response — запасной ответ, который используется, если основной путь не сработал. 

Request navigation page
→ fetch в сеть
→ ошибка
→ вернуть /offline.html из кэша
self.addEventListener("fetch", (event) => {
  if (event.request.mode === "navigate") {
    event.respondWith(
      fetch(event.request).catch(() => {
        return caches.match("/offline.html");
      })
    );
  }
});

Fallback нужен, чтобы пользователь не видел просто браузерную ошибку сети.

Вместо этого можно показать:

offline.html
заглушку изображения
кэшированную старую страницу
сообщение "нет соединения"

Вариант 5. Сеть как fallback

Фраза «fallback в сеть» может означать обратную стратегию:

сначала кэш
→ если в кэше нет ответа
→ сходить в сеть

То есть сеть становится запасным вариантом после Cache Storage.

caches.match(event.request).then((cachedResponse) => {
  return cachedResponse || fetch(event.request);
});
если есть локальный ответ — использовать его
если локального ответа нет — продолжить обычный сетевой путь

8. Что значит Response-like результат

Иногда Service Worker возвращает ответ, который не был получен от сервера прямо сейчас.

Например:

return new Response("<h1>Offline</h1>", {
  headers: { "Content-Type": "text/html" },
  status: 200
});

Такой ответ создан JavaScript-кодом, но для браузера он всё равно похож на обычный HTTP response: 

status
headers
body

Поэтому дальше браузер может обрабатывать его как результат запроса.

9. Как этот блок связан с общей схемой

До этого браузер уже разобрал URL:

scheme, host, port, path, query

После этого он проверяет:

Можно использовать локальный ответ?

Если запрос попадает в Service Worker, возможны варианты:

Service Worker
├─ вернул Response из Cache Storage
│  → сеть не нужна
│
├─ вернул fallback Response
│  → сеть не нужна или сеть уже упала
│
└─ вызвал fetch(event.request)
   → запрос возвращается в сетевой pipeline

Service Worker не обязательно отменяет сеть. Он может либо завершить запрос локально, либо передать его дальше в сеть.

10. Мини-глоссарий

Service Worker

Фоновый JavaScript worker, который может перехватывать запросы, работать с кэшем и возвращать собственные ответы.

Scope

Область URL, которую контролирует Service Worker.

Fetch event

Событие Service Worker, возникающее при запросе контролируемой страницы.

respondWith()

Метод, которым Service Worker задаёт ответ для перехваченного запроса.

Cache Storage

Программируемое хранилище Request → Response, доступное через Cache API.

Cache API

Набор браузерных API для открытия кэшей, сохранения response и поиска response по request.

Request

Объект, представляющий запрос.

Response

Объект, представляющий ответ: status, headers и body.

Cache-first

Стратегия: сначала кэш, если промах — сеть.

Network-first

Стратегия: сначала сеть, если ошибка — кэш.

Fallback

Запасной ответ или запасной путь, если основной способ не сработал.

Offline fallback

Fallback-ответ для ситуации, когда сеть недоступна.

PWA

Progressive Web App — веб-приложение, которое может использовать возможности вроде Service Worker, офлайн-режима и установки на устройство.
